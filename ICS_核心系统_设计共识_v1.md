### 4.3 变量管道（新方向）

**定位**：所有"读取数据组装到提示词"需求的统一入口。

**核心设计原则**：
- **只读不写、纯求值**——不允许有副作用的占位符（不学 ST 的 setvar 在提示词里偷偷改状态）
- **强制带命名空间**——不允许裸名 `{{user}}`，必须 `{{user:name}}`
- **统一三段式语法**：`{{namespace:identifier:modifier}}`，冒号分隔
- **子系统注册机制**——每个上层系统启动时向变量管道注册自己的命名空间和变量列表

**命名空间划分**：

| 命名空间 | 子系统 | 示例 |
|---------|--------|------|
| `char:` | 角色系统 | `{{char:name}}` `{{char:description}}` |
| `user:` | 用户/马甲系统 | `{{user:name}}` `{{user:persona_desc}}` |
| `v:` | 数值系统 | `{{v:affection}}` `{{v:affection:stage}}` `{{v:affection:desc}}` |
| `world:` | 世界状态 | `{{world:time}}` `{{world:weather}}` |
| `mode:` | 对话场景预设 | `{{mode:name}}` `{{mode:style_hint}}` |
| `wb:` | 世界书 | `{{wb:active_entries}}` |
| `time:` | 时间工具 | `{{time:now}}` `{{time:weekday}}` |
| `mem:` | 记忆系统（未来） | `{{mem:retrieved}}` |
| `evt:` | 事件系统（未来可选） | `{{evt:active_injections}}` |
| `preset:` | 预设参数（见 4.6） | `{{preset:yandere_intensity}}` |
| `util:` | 工具函数 | `{{util:random:a,b,c}}` `{{util:roll:1d6}}` |

**配套：变量目录页（开发期工具，但建议保留进生产）**

UI 内容：
- 按命名空间分组列出所有当前可用变量
- 每个变量显示：标识符、说明、**用当前角色+马甲实时求值后的当前值**、复制按钮
- 底部测试器：作者输入混合文本，实时看变量替换后的输出
- 这是反阳奉阴违的核心工具——CC 克声称某变量通了，打开目录页 1 秒验收

- 实现解析器和注册机制

### 4.4 触发与效果基础设施层（Trigger/Effect）

**定位**：所有"在某条件下做某事"需求的共享底层。事件系统、数值规则、后处理规则、时间触发、精神体渗透等多个上层系统的公共依赖。

**Trigger 类型库（初步清单）**：

| 类型 | 说明 |
|------|------|
| `chat_end` | 每轮对话结束 |
| `time_pass` | 时间流逝（每小时/每天） |
| `value_change` | 某数值变化 |
| `value_threshold` | 数值越过阈值 |
| `ai_output_match` | AI 输出匹配正则/关键词 |
| `user_input_match` | 用户输入匹配正则/关键词 |
| `date_match` | 日期匹配（生日、节日、纪念日） |
| `event_complete` | 某事件状态变为 completed |
| `mode_switch` | 对话场景预设切换 |
| `interaction_count` | 互动累积达到 N 次 |

**Effect 类型库（初步清单）**：

| 类型 | 说明 |
|------|------|
| 改数值 | 修改 character_values 中的值 |
| 设标记 | 设置布尔标记 |
| 触发事件 | 立即触发指定事件 |
| 解锁/锁定事件 | 改变事件 status |
| 注入提示词 | 进入 pending_injections 队列 |
| 添加世界书条目 | 创建新条目 |
| 激活/停用世界书条目 | 改变条目状态 |
| 写入记忆 | 进入记忆系统的写入管线 |
| 发送消息 | 角色主动给用户发消息 |

**关键设计**：
- 上层系统**保留各自特色字段和状态机**（事件系统继续有 locked/pending、冷却；数值规则继续有 trigger_on 列表；后处理规则只有简单开关）
- 但**共享同一套 Trigger/Effect 类型库**——新加一种 Effect，所有上层系统受益
- 可视化编辑器中"加条件/加效果"的下拉菜单内容统一来自此层

**与 UI 直接调用的边界（重要）**：

UI 按钮、表单提交、菜单点击等用户**显式动作**，**直接调用子系统接口**，不经过 Trigger/Effect 基础设施。

Trigger/Effect 仅服务于**系统自动化**——某条件成立时自动发生的动作。

```
用户点按钮       → 直接调用 子系统接口      → 改数据库
某条件触发       → Trigger → Effect → 子系统接口 → 改数据库
```

**两条路径共用同一套子系统接口**。例如：
- 世界书的 `addEntry()` 既被"新建条目"按钮调用，也被 Effect"添加世界书条目"调用
- 数值系统的 `updateValue()` 既被调试面板的"+10 好感度"按钮调用，也被 Effect"改数值"调用

不允许把 UI 操作伪装成"trigger=按钮点击"再走 Effect 系统——这会引入无谓的间接层、性能开销、调试复杂度。

**与变量管道的关系**：完全正交。变量管道只读，Trigger/Effect 只写。两者通过共享数据库间接联系，互不调用对方。

**情境 B 类需求强制走 Trigger/Effect**：AI 输出/用户输入匹配触发的自动动作（如自动建档世界书条目、关键词触发数值变化、对话内容触发事件解锁）一律通过后处理规则系统实现，**不允许**通过提示词副作用语法（如 ST 风格的 `{{setvar}}` `{{addLore}}`）实现。作者要让某动作自动发生，必须明确声明一条规则，不能藏在提示词里。

**AI/用户改数据的完整链路**：

AI 角色或用户的输出**不能直接修改数据库**，必须经过规则机制：

```
AI 输出某段内容
    ↓
[Trigger 检测] 这段输出是否匹配某条规则的触发条件？
    ↓ 匹配
[Effect 执行] 调用对应子系统接口
    ↓
数据库变更
```

**Trigger 的两个维度**：

维度一：**来源（source）**——内容从哪来

| 来源类型 | 说明 |
|---|---|
| `ai_output` | AI 角色的输出 |
| `user_input` | 用户的输入 |
| `narration` | 旁白/系统消息 |
| `inner_monologue` | 角色内心独白（若做此区分） |

维度二：**匹配条件（pattern）**——内容什么形态算匹配

| 匹配类型 | 说明 |
|---|---|
| `regex` | 正则匹配（最严格、最可控） |
| `keyword` | 关键词包含 |
| `keyword_any` | 任一关键词 |
| `keyword_all` | 全部关键词 |
| `semantic` | 语义匹配（需 AI 调用，开销大） |

**一条规则的完整形态示例**：

```
规则名：自动建档新角色
启用：✓
[Trigger]
  来源：ai_output
  匹配类型：regex
  匹配模式：\[新角色:(.+?)\]\s*(.+)
[Effect]
  类型：添加世界书条目
  参数：
    title：$1
    content：$2
    tags：[自动生成, NPC]
```

**作者-AI-规则的协作模式**：

```
作者层：定义规则（trigger 模式 + effect 动作）
   ↓
作者层：在提示词里告诉 AI "你可以使用 X 格式达成 Y 效果"
   ↓
AI 层：在恰当时机输出 X 格式
   ↓
规则层：检测到匹配，执行 Y
```

AI 看似"主动改了数据库"，实际是在作者搭好的协议框架里说话。这种**约定式的间接控制**比 function calling 简单、比放任 AI 自由发挥安全、比硬编码灵活。

**为什么强制经过规则机制（不让 AI 直接调）**：

1. **可见性**：所有 AI 能触发的修改都明确声明在规则列表里，是给 AI 的能力边界——没列出来的事它做不到
2. **可调试**：日志里能查到某条数据是哪条规则触发的，不会出现"AI 偷偷改了什么"的黑箱
3. **作者控制**：写规则就是设计"AI 在什么情况下能影响什么"，是叙事设计的一部分
4. **规避格式不稳定**：AI 自然语言输出格式天天变，作者定义的协议（如 `[新角色:X] Y`）把不稳定挡在外面——AI 不遵守协议规则不触发数据库不变，保底安全

- 先验收 CC 克现有数值/事件实现，看 Trigger/Effect 当前是怎么实现的
- 决策：抽出到基础设施层 vs 接受现状继续

### 4.5 对话场景预设（ConversationMode）

**定位**：替代当前硬编码的"线上/线下"双模式，作为作者可配置的对话场景类型。一等公民概念，不只是变量管道的消费者。

**模式涉及的耦合面（远不止提示词）**：
1. 提示词组装（模板 + 注入位置）
2. 世界书联动（默认激活哪些条目）
3. 前端展示（气泡 vs 长文叙事 vs 信件）
4. 事件系统（某些事件限定在某模式触发）
5. 数值变化（某些规则可能只在特定模式生效）
6. 截断/分段逻辑
7. AI 调用参数（max_tokens、temperature）
8. 记忆写入粒度

**ConversationMode 字段（初步设计）**：

```
id                          内部标识，如 'online' / 'offline' / 'dream'
display_name                显示名
description                 作者说明

[提示词层]
prompt_template             提示词模板，含 {{...}} 变量
prompt_position             注入位置（角色设定后/历史前/深度N）

[世界书联动]
default_active_wb_tags      默认激活的世界书标签

[前端展示]
ui_style                    'bubble' / 'narrative' / 'letter' / 'custom'
background_theme            背景配色 key
message_renderer            消息渲染器选择

[AI 调用]
max_tokens                  建议长度
temperature                 可选覆盖

[行为规则]
message_segmentation        单条/多条/不分段
default_length_hint         "20-150字" / "200-800字" / "自由"

[事件系统挂钩]
event_filter_tags           哪些事件标签在此模式生效

[记忆系统挂钩（未来）]
memory_draft_granularity    每消息一条 / 每段一条 / 手动标记

[元数据]
is_builtin                  是否系统内置（线上/线下不可删）
created_by
custom_data                 JSON 自由扩展
```

**内置预设**：
- `online`（线上）：聊天气泡、短消息、口语化
- `offline`（线下）：长文叙事、暖色调、200-800 字段落

均不可删。其他预设作者可从内置复制再改，或从零建（如梦境、信件、电话、日记）。

### 4.6 预设参数化（preset 命名空间）

**背景**：ST 用户的真实使用模式——预设作者在"配置中心条目"里用 `{{setvar}}` 集中定义参数，后续条目用 `{{getvar}}` 读取。用户改预设时只动配置中心，不用翻条目。

**我们的方案**：把这种模式从"提示词副作用"升级为"声明式参数化"。

**作者侧**：
- 预设/世界书包暴露"参数声明区"
- 每个参数有：参数名、显示名、类型（数字/文本/枚举/布尔）、默认值、说明、范围/选项
- 在预设条目中通过 `{{preset:参数名}}` 引用

**用户侧**：
- 启用预设后，看到"预设参数面板"，列出所有参数
- 带类型校验、说明提示、默认值
- 改完保存，整个会话生效

**与数值系统的区别**：

| 机制 | 作用域 | 谁定义 | 谁修改 | 用途 |
|------|--------|--------|--------|------|
| 数值系统 | 角色 | 创建角色时作者 | 系统通过事件/规则自动 | 模拟内在状态变化 |
| 预设参数 | 预设包 | 预设作者 | 用户在面板手动 | 预设的可调配置 |

数值"会变"，预设参数"基本不变"（设一次用整个会话）。

**遗留问题**：
- 命名空间名 `preset` 是否合适，待 Agnes 后续决策
- 生效范围（绑定到当前预设包还是当前会话）——本文档采"绑定到当前激活的预设包"，多预设各自有自己的命名空间

**待办**：
- 待变量管道设计文档完成后，作为子模块展开
- 优先级低于变量管道本身和 Trigger/Effect 基础设施

## 六点五、系统层级关系

本项目不强行套用某种标准分层架构（MVC、Clean Architecture 等），但需要明确各子系统之间的依赖关系，避免讨论时层级混乱。

```
┌─────────────────────────────────────────────────────────┐
│  用户界面层                                             │
│  聊天页 / 角色编辑器 / 数值编辑器 / 事件编辑器 /        │
│  变量目录页 / 预设参数面板 / 时间线 / 状态栏            │
└─────────────────────────────────────────────────────────┘
                          ↑ 暴露
┌─────────────────────────────────────────────────────────┐
│  应用功能层                                             │
│  对话循环 / 角色养成 / 剧情推进 / 角色主动生活 /        │
│  后处理规则                                             │
└─────────────────────────────────────────────────────────┘
                          ↑ 组合
┌─────────────────────────────────────────────────────────┐
│  子系统层                                               │
│  数值系统 / 事件系统 / 世界书 / 记忆系统 /              │
│  对话场景预设(ConversationMode) / 预设参数化 /          │
│  精神体系统                                             │
└─────────────────────────────────────────────────────────┘
                          ↑ 调用
┌─────────────────────────────────────────────────────────┐
│  基础设施层                                             │
│  变量管道(读) / Trigger 类型库 / Effect 类型库          │
│  AI 调用 / 数据库 / 日志                                │
└─────────────────────────────────────────────────────────┘
```

**层级使用规则**：

- **下层不知道上层存在**：基础设施层不知道有数值系统，子系统不知道有什么应用功能调用它。这是单向依赖。
- **同层之间的引用通过基础设施层中转**：数值系统不直接调用事件系统的内部函数，而是通过 Effect 类型库（"改数值"）和 Trigger 类型库（"value_change"）解耦。
- **跨层捷径要警惕**：UI 层直接读数据库表是反模式，必须经过子系统层暴露的接口。

**不是分层**：本项目里"角色"、"马甲"、"用户"这三个概念是横切的，每一层都要感知它们。它们不构成独立的层，是贯穿所有层的上下文标识。
