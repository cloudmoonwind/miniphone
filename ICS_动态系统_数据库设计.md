# 第一部分：数值系统

## 设计思路

数值系统管理角色的各种状态值。核心功能：
- 存储数值的定义和当前值
- 数值分段，不同阶段有不同描述和提示词
- 数值变化规则，不同范围可以有不同的变化规则
- 提示词动态注入，根据当前数值阶段注入对应的提示词片段

### 讨论细节：数值分类

我们讨论了角色相关的数值有哪些，最终分成这几类：

**A. 角色内在属性（变化慢，偏固定）**
- 智力
- 社交能力

**B. 角色动态状态（随时变）**
- 体力/精力：活动消耗，休息恢复
- 饥饿/饱腹：时间流逝消耗，吃饭恢复
- 健康：生病下降，恢复上升

**C. 角色心理状态（之前设计的情绪系统）**
- 心情/情绪值
- 压力
- 理智（SAN值机制）

**D. 关系类（针对特定对象）**
- 亲和度/好感度（对user）
- 亲近度（对NPC）→ 这个存在人际关系表，不在数值系统
- 信任度（可选）

**E. 社会属性**
- 大众声誉
- 专业声誉
- 阅历

**F. 世界环境（不属于角色，但影响角色）**
- 时间（目前按现实世界）
- 日期（目前按现实世界）
- 地点
- 天气（可做可选列表，或接API获取现实天气）
- 温度、湿度（可选）

|------|------|------|------|
| id | 整数/主键 | ✅ | |
| value_id | 整数/外键 | ✅ | 关联哪个数值 |

| range_max | 数值 | | |
| trigger_on | 枚举 | ✅ | 触发时机，见下方 |

| operation | 枚举 | ✅ | set / add / multiply |
| amount | 数值 | ✅ | 操作的值 |
| enabled | 布尔 | | 默认true |

**trigger_on 触发时机：**

| 值 | 说明 |
|------|------|
| chat_end | 每轮对话结束后 |
| time_pass_hourly | 时间流逝（每现实小时，定时任务） |
| time_pass_daily | 时间流逝（每现实天，定时任务） |
| time_pass_life | 时间流逝（生活模拟行为，调用 /life/generate 时） |
| receive_gift | 收到礼物 |
| event_complete | 事件完成时 |
| value_change | 其他数值变化时 |

> **time_pass 两类三个：**
> - 类一（定时，实时时钟）：`time_pass_hourly` + `time_pass_daily`，服务端 setInterval/node-cron 驱动
> - 类二（游戏行为）：`time_pass_life`，玩家触发生活模拟时调用
> - 数值规则中 `trigger_on` 存具体子类型（如 `time_pass_daily`）

### 讨论细节：数值规则的范围设计

Agnes提出：为不同参数的不同数值范围设计不同的规则。

比如好感度：
- 0-20时：聊天加好感的幅度大（容易从陌生变熟悉）
- 80-100时：聊天加好感的幅度小（已经很亲密，提升空间小）

这就是为什么 value_rules 有 range_min 和 range_max 字段。

## 提示词动态注入

### 讨论细节：怎么实现？

**问题1：表从哪里来？**

需要做编辑器UI。作者在前端填写数值定义、阶段描述、提示词片段，保存后存入数据库。

**问题2：提示词里怎么写变量？**

用占位符。提示词里写：

```
{{char}}对{{user}}的态度：{{v:affection:desc}}
```

其中：
- `{{char}}` → 角色名
- `{{user}}` → 用户名
- `{{v:affection}}` → 好感度的当前值
- `{{v:affection:stage}}` → 好感度的阶段名（如"朋友"）
- `{{v:affection:desc}}` → 好感度的阶段描述

**问题3：怎么识别并替换？**

后端发送prompt前，用正则匹配 `{{...}}`，解析变量名，查数据库拿值，替换。

伪逻辑：
```
1. 正则找到 {{v:affection:desc}}
2. 解析：v = 数值变量，affection = 变量名，desc = 取描述
3. 查 character_values 找到好感度当前值 = 75
4. 查 value_stages 找到75对应的阶段 → 50-80 → "朋友"阶段
5. 取 description 字段 → "对用户态度温和，愿意分享日常"
6. 替换回去
```

# 第二部分：事件系统

## 设计思路

事件系统管理剧情事件的解锁、触发、执行。

### 讨论细节：事件来源

最初讨论时分了三种事件来源：
1. 预设事件：作者提前写好
2. 随机事件：系统随机触发
3. AI生成事件：角色在对话中产生意图，生成计划

**关键决策：预设和随机不分表**

Agnes指出：随机事件也是作者写的，只是触发条件里多了"概率"。区别不在于存储方式，而在于触发条件的设计。

所以最终决定：都存在同一张 events 表，通过 probability 字段区分。
- probability = 100：条件满足必定触发（传统剧情事件）
- probability = 30：条件满足后30%概率触发（随机事件）

### 讨论细节：事件状态

| 状态 | 说明 |
|------|------|
| locked | 锁定，解锁条件未满足，不参与任何检查 |
| pending | 待触发，解锁条件满足了，等待触发条件 |
| active | 进行中，多步骤事件正在执行 |
| completed | 已完成 |

**为什么分"解锁条件"和"触发条件"两层？**

有些事件"理论上可以发生了"但"时机还没到"。

例子：
- 完成"第一次约会"后，"表白事件"就解锁了（进入待触发队列）
- 但要等好感度够高、气氛对了才触发

解锁条件 = 静态条件（前置事件完成、标记状态）
触发条件 = 动态条件（数值、时间、关键词）

### 讨论细节：冷却机制

Agnes提出需要冷却机制。最终确定三种冷却类型：

| 类型 | 说明 | 示例 |
|------|------|------|
| 按时间 | 触发后N天才能再触发 | "每周一次的约会" |
| 按轮次 | 触发后聊N轮才能再触发 | "不想每轮都触发同一个事件" |
| 按条件满足次数 | 条件满足N次才触发 | "下雨3次才能再偶遇" |

**关键决策：**
- 按时间和按轮次**互斥**，只能选一种（都是"等待型"冷却）
- 按条件满足次数可以和上面两种**叠加**（它是"计数型"，不同维度）
- 两个冷却都要清零才能再次触发

### 讨论细节：事件链

Agnes想要可视化编辑器显示事件之间的连接关系。

```
主线1.1 ──→ 主线1.2 ──→ 主线1.3
               ↓           ↘
            分支1.1 ──→ 分支1.2 → 主线1.6.2
                                    ↑
                               （交汇）
```

所以需要 event_connections 表存储事件之间的连接。

一个事件可以：
- 同时属于多条剧情线（通过标签）
- 有多个前置事件
- 导向多个后续事件

### 讨论细节：触发检查的性能优化

**问题：每次条件变化都要遍历所有事件吗？**

Agnes指出这不优雅。

**解决方案：条件订阅表**

事件从 locked → pending 时，把它关心的条件注册到订阅表：

```
订阅表：
  event_id: evt_confession, condition_type: value, condition_target: affection
  event_id: evt_confession, condition_type: time, condition_target: evening
```

条件变化时，只查订阅表找相关事件，不用遍历所有事件。

```
好感度变了
  ↓
查订阅表：谁订阅了 affection？
  ↓
找到 evt_confession（只有这一个，不是100个）
  ↓
只检查这个事件的完整条件
```

### 讨论细节：同时触发多个事件

**问题：多个事件同时满足条件怎么办？**

解决方案：
- 按 priority 字段排序，优先级高的先触发
- 前端可以设置"每轮最多触发N个事件"
- 没被触发的下次还有机会（不是丢弃）

**问题：互斥事件怎么办？**

Agnes说：互斥靠触发条件区分，不是靠随机池。
比如"开心事件"和"难过事件"的触发条件本身就不同（心情值不同），不会同时满足。

### 讨论细节：可重复事件的链条问题

**问题：事件A可重复，触发事件B，但B不可重复。第二次触发A时怎么办？**

答案：A触发时发现B已完成且不可重复 → 跳过B，A继续执行自己的其他效果。

如果B后面还有C，链条在B这里断了。作者想让C还能触发，就自己设计A→C的直接连接。

## JSON字段格式

### 条件格式

支持多组条件，组间可以是"且"或"或"关系：

```json
{
  "组间逻辑": "或",
  "条件组": [
    {
      "组内逻辑": "且",
      "条件": [
        { "类型": "数值", "目标": "affection", "比较": ">=", "值": 80 },
        { "类型": "时间", "值": "evening" }
      ]
    },
    {
      "组内逻辑": "且",
      "条件": [
        { "类型": "事件", "目标": "evt_drunk", "状态": "completed" }
      ]
    }
  ]
}
```

意思是：（好感度≥80 且 晚上）或（喝醉事件完成）

**单个条件的字段：**

| 类型 | 字段 | 说明 |
|------|------|------|
| 数值 | 目标、比较、值 | 检查数值 |
| 标记 | 目标、比较、值 | 检查布尔标记 |
| 事件 | 目标、状态 | 检查事件状态 |
| 时间 | 值 | morning/afternoon/evening/night |
| 日期 | 值 | weekday/weekend/节日名 |
| 天气 | 值 | sunny/rain/cloudy/snow |
| 地点 | 目标 | 检查当前位置 |
| 关键词 | 值、来源 | 检查对话内容 |
| 随机 | 概率 | 0-100 |

### 效果格式

```json
[
  { "类型": "注入", "内容": "角色看着你...", "位置": "after_char", "持续": "turns", "持续值": 3 },
  { "类型": "改数值", "目标": "affection", "操作": "add", "值": 10 },
  { "类型": "记录结果", "目标": "evt_confession", "值": "success" },
  { "类型": "触发事件", "目标": "evt_relationship_start" },

]
```

**效果类型：**

| 类型 | 字段 | 说明 |

| 注入 | 内容、位置、持续、持续值 | 注入上下文 |

| 解锁事件 | 目标 | 解锁另一个事件 |
| 锁定事件 | 目标 | 锁定另一个事件 |
| 改位置 | 目标 | 改变角色位置 |

## 工作流程

### 事件生命周期

```
作者创建事件 → status = locked
    ↓
某个前置事件完成 → 检查 unlock_conditions → 满足 → status = pending
                                                    → 填充订阅表
    ↓
条件变化时 → 查订阅表找相关事件 → 检查 trigger_conditions
    ↓
不满足 → 继续等
满足 → 检查概率 → 随机判断
    ↓
不触发 → 继续等
触发！→ 执行 effects → 更新状态 → 清理订阅
                    → 设置冷却（如有）
                    → 注入内容存入 pending_injections
    ↓
后续对话 → 发送前读取 pending_injections → 插入prompt
        → 对话结束 → remaining_turns - 1 → 归零删除
```

### 冷却处理

```
触发事件时：
  - cooldown_remaining = cooldown_value
  - condition_cooldown_remaining = condition_cooldown

每天/每轮检查：
  - 按时间冷却：cooldown_remaining - 1（每天）
  - 按轮次冷却：cooldown_remaining - 1（每轮）

条件满足时：
  - condition_cooldown_remaining - 1

再次触发条件：
  - cooldown_remaining == 0 且 condition_cooldown_remaining == 0
```

## 待确认问题

### 事件成功/失败判断（已确认方案）

有些事件需要判断结果（比如表白成功还是失败）。结果存储在 `events.outcome` 字段（TEXT，如 `"success"` / `"fail"` / 自定义分支名）。

**实现方案（双轨并行）：**

1. **AI标记捕获（主要）**：AI 回复末尾输出 `[EVENT:事件ID:结果]` 标签，后端在收到响应后用正则解析并写入 `events.outcome`，同时触发后续事件链。
   - 优点：不打断沉浸感，自动化
   - 缺点：AI 可能漏写或写错，需要 system prompt 强调

2. **用户弹窗选择（备用/兜底）**：若 AI 未输出标记，或事件设置了 `requireUserConfirm: true`，前端弹窗让用户手动选结果。
   - 适用场景：关键剧情节点、作者不信任AI判断时

**`outcome` 在事件链中的作用：**
- `event_connections` 的 `relation_type = 'branch'` 配合 `required_outcome` 字段（JSON或约定值），实现互斥分支：
  - 连接A → B（required_outcome: "success"）
  - 连接A → C（required_outcome: "fail"）
  - 触发引擎只走 outcome 匹配的那条连接

### 总结表 (summaries)

| 字段 | 类型 | 说明 |
|------|------|------|
| id | 主键 | |

- 消息删了总结不删（总结是压缩后的记忆，应该保留）

|------|------|------|

| 类型 | 样式 | 说明 |
|------|------|------|
| 顺序/解锁 | 实线箭头 | A完成后B解锁 |
| 分支 | 虚线箭头 | A可以走向B，各分支互斥 |
| 触发 | 粗线箭头 | A完成时自动触发B，优先看能不能用粗线 |
