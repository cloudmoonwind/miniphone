# ICS AI 输出协议 v1

**日期**：2026-05-02
**起草**：CC 克
**承接**：[ICS_近期开发计划_v1.md](docs/archive/ICS_近期开发计划_v1.md)（已归档）阶段 2、[docs/Codex讨论沉淀/变量状态触发效果与角色主动性设计思路_v1.md](docs/Codex讨论沉淀/变量状态触发效果与角色主动性设计思路_v1.md)
**用途**：定义 AI 在每轮回复末尾输出的结构化协议格式。系统按此协议解析、校验、写入数据库；正文部分对用户原样展示。

---

## 一、定位与基本原则

### 1.1 协议是什么

ICS 系统通过提示词告诉 AI 在每轮回复末尾附加一段结构化文本块,系统从中读取:

- **变量更新**:本轮哪些数值变了,新值多少,可选的原因
- **情绪快照**:本轮 AI 视角的角色情绪混合(百分比)
- **事件结果**:本轮某事件被标记为成功/失败/其他 outcome,可选的原因
- **未来**:状态提案、记忆草稿、世界书条目自动建档等

### 1.2 设计原则

1. **AI 提议、系统守门**。AI 输出的是"提议",系统按规则校验后才写入。但当前阶段(v1)语义保持"提议=决定",未来出现状态槽/记忆草稿等需要审批的场景时再加 `proposal` 标记。
2. **正文与协议分离**。`<sys>` 块在解析后从 cleanContent 中**移除**,用户看到的是干净正文。`<sys>` 块是给系统看的,不是给用户看的。
3. **行式 + `|` 分段**。每条更新一行,行内可选用 `|` 追加元数据。便于 AI 输出、便于解析、便于人工肉眼检查 trace。
4. **失败不污染**。某子块/某行解析失败时,trace 记一笔警告,继续处理其他部分,不让一个变量打错就丢失整轮变量更新。
5. **不留兼容包袱**。旧格式(顶层 `<var>` 单独出现、`[EVENT:id:outcome]` 内联标签)阶段 2 落地后**直接下线**,不做兼容期(数据库没有真实历史数据)。

### 1.3 设计反对什么

- ❌ 把状态变化藏在正文里(用关键词/正则去推断)——做不到"AI 提议、系统守门"
- ❌ 用 JSON 格式(对中文输出友好度低、AI 容易忘记尾逗号、肉眼检查难)
- ❌ 一旦解析失败就丢弃整轮(单个错误不应导致全部丢失)
- ❌ 把"指令格式说明"和"提交内容"混在一起

## 二、顶层容器 `<sys>`

### 2.1 语法

```
<sys>
  <var>
    ...
  </var>
  <event>
    ...
  </event>
</sys>
```

### 2.2 约定

- `<sys>` 块每轮回复**最多一个**,出现在回复末尾(紧跟正文,可空行隔开)
- 解析时从 cleanContent 中**整段移除**
- 内部按子标签(`<var>` `<event>` 以及未来扩展)分类处理
- 未识别的子标签 trace 警告,不报错、不污染正文
- 子标签内部为空(只有空白)等同于"无更新",合法
- `<sys>` 块本身可省略——AI 没有任何更新时不输出即可
- 大小写不敏感:`<SYS>` `<Var>` `<EVENT>` 都接受(但建议小写)

### 2.3 解析失败的降级

- AI 输出了 `<sys>` 但内部格式完全错乱(如未闭合标签、嵌套混乱):整个 `<sys>` 块从 cleanContent 移除,trace 记 `protocol.parse.failed`,不写任何更新
- AI 输出了某个未识别的子标签(如 `<weird>`):该子标签整段移除,trace 警告,其他子标签正常处理

## 三、子块 `<var>`:变量更新

### 3.1 语法

```
<var>
变量名: 旧值→新值
变量名: 旧值→新值 | 原因: 简短说明
情绪: 情绪词1 X% | 情绪词2 Y% | 情绪词3 Z%
</var>
```

每行是一条独立更新。空行忽略。

### 3.2 行的种类

#### 3.2.1 数值变量更新行

```
<变量名>: <旧值>→<新值>
<变量名>: <旧值>→<新值> | 原因: <简短说明>
```

- `→` 也接受 `->`(纯 ASCII 替代,AI 输出兼容)
- `<变量名>` 可以是 variableName(如 `affection`)或 displayName(如 `好感度`),系统按二者顺序查找
- `<旧值>` 必须与系统当前值在 ±1 范围内一致(防止 AI 凭空改值或基于过时快照)
- `<新值>` 系统自动 clamp 到 `[min, max]`
- `| 原因:` **可选**。出现时记录到 trace 和未来的记忆系统;不出现时正常写入,不报错

#### 3.2.2 情绪行

```
情绪: 情绪词1 X% | 情绪词2 Y% | 情绪词3 Z%
```

- 固定以 `情绪:` 开头(中文冒号也接受)
- 后续以 `|` 分隔多个情绪段,每段格式 `情绪词 X%`
- 百分比之和应在 95~105 之间(±5 容忍 AI 算术不精)
- 选 2~5 个主要情绪
- **情绪行不支持 `| 原因:`**——情绪行本身用 `|` 当分隔符,加原因会有解析歧义。如果未来需要给情绪整体加原因,会单独引入新行(如 `情绪原因: ...`)

#### 3.2.3 解析顺序

每行依次尝试匹配:
1. 是否以 `情绪:` 或 `情绪：` 开头 → 走情绪行解析
2. 否则尝试匹配 `<标识符>: <数字>→<数字>` → 走数值更新行解析
3. 都不匹配 → trace 警告,跳过本行

### 3.3 元数据语法(为未来扩展留口子)

数值更新行的 `| key: value` 形式只在 v1 接受 `原因:`。未来扩展时可加:

| key | 含义 | v1 状态 |
|-----|------|--------|
| 原因: | 变量变化的简短说明 | ✅ 接受 |
| 来源: | 触发此变化的事件 ID 或对话句号 | 🔜 v2 |
| 提案: | true/false,是否仅作提议待系统审批 | 🔜 状态槽系统启用后 |

v1 解析器对未识别 key 的处理:**记 trace 警告,不阻塞主更新**。这样 AI 输出的实验性元数据不会让本行整体作废。

### 3.4 完整示例

```
<var>
affection: 30→35 | 原因: 用户主动分享了童年回忆
sanity: 50→48 | 原因: 童年话题让她略感不安
trust: 40→42
情绪: 温暖 50% | 紧张 30% | 怀念 20%
</var>
```

## 四、子块 `<event>`:事件结果

### 4.1 语法

```
<event>
事件ID: outcome
事件ID: outcome | 原因: 简短说明
</event>
```

### 4.2 约定

- 每行一个事件结果
- 事件 ID 必须存在于 events 表中,否则 trace 警告并跳过
- outcome 任意字符串(`success` / `fail` / `partial` / 自定义),系统**不**枚举校验,作者在事件设计时自己约定语义
- `| 原因:` 可选,语义同 `<var>` 块
- 触发该事件的 branch 连接逻辑由系统按 outcome 字符串匹配处理(`relationType = 'branch:<outcome>'`),与本协议无关

### 4.3 完整示例

```
<event>
evt_confession: success | 原因: 用户没有拒绝告白
evt_first_meet: completed
</event>
```

## 五、解析规则汇总

### 5.1 容忍

- 大小写不敏感(`<sys>` `<SYS>` `<Var>` 都接受)
- 中英文冒号都接受(`:` `：`)
- `→` 与 `->` 都接受
- 行内多余空白(空格/制表符)忽略
- 行尾 `|` 后无内容视为合法(空元数据)

### 5.2 严格

- `<sys>` 必须严格闭合(否则整段失败,trace 记录)
- 数值更新行的旧值必须与系统当前值在 ±1 范围内
- 情绪行百分比之和 95~105 之外视为格式错(整行作废,trace 记录)

### 5.3 失败降级

| 失败类型 | 处理 |
|---------|------|
| 单个变量行格式错(如缺 `→`) | trace 警告,跳过该行,其他行继续 |
| 旧值不匹配 | trace 警告,跳过该行 |
| 变量名不存在 | trace 警告,跳过该行 |
| 情绪行百分比错 | trace 警告,跳过该行 |
| 事件 ID 不存在 | trace 警告,跳过该事件 |
| 未识别子标签(如 `<weird>`) | trace 警告,该子标签整段忽略 |
| `<sys>` 块未闭合 | trace 警告,整段忽略 |

### 5.4 trace 模块

阶段 2 解析逻辑统一归属 `aiProtocol` trace 模块(需在 [server/services/trace.ts](server/services/trace.ts) 的 TraceModule 类型加 `'aiProtocol'`)。

## 六、给 AI 的提示词样板

下方是可以直接放入 prompt 模板的标准说明文字。作者可改写,但保留**语法约束部分**不变。

```
【输出协议】每轮回复结束后,如果本轮有变量变化或事件标记,在正文末尾附加一个 <sys> 块。
没有任何变化时,不输出 <sys> 块即可。

<sys> 块结构:

  <var>
  变量名: 旧值→新值                       —— 数值更新最简形式
  变量名: 旧值→新值 | 原因: 简短说明      —— 可选附加原因
  情绪: 情绪词1 X% | 情绪词2 Y%            —— 情绪快照,百分比之和≈100,选2~5个
  </var>

  <event>
  事件ID: outcome                          —— 标记事件结果
  事件ID: outcome | 原因: 简短说明
  </event>

约束:
1. 数值更新的旧值必须与系统当前值一致(±1 容忍),不要凭空改数字
2. 变量名用 variableName 或显示名都行
3. 情绪行不支持 | 原因:(快照本身没有单一原因)
4. 事件 ID 必须是已定义的事件,outcome 字符串自由
5. <sys> 块出现在正文之后,空一行隔开。<sys> 块本身不会展示给用户
6. 没有更新时整个 <sys> 块省略

示例:

正文内容...

<sys>
<var>
affection: 30→35 | 原因: 用户主动分享了童年回忆
情绪: 温暖 50% | 紧张 30% | 怀念 20%
</var>
<event>
evt_confession: success
</event>
</sys>
```

注:阶段 3 的 ConversationMode 落地后,这段提示词会迁移到 `sys-mode` 系统槽,跟随当前模式自动注入。

## 七、未来扩展位

`<sys>` 容器为后续阶段保留以下子标签(v1 不识别,会被解析器作为"未识别子标签"忽略 + trace 警告)。设计时确认这些命名不与现有冲突:

| 子标签 | 用途 | 计划阶段 |
|-------|------|---------|
| `<state>` | 状态槽更新(emotional_posture / action_posture 等),Codex 思路文档 4.6 节 | 状态槽系统启用后 |
| `<state_proposal>` | AI 提议新状态,带 reason / exit_conditions / duration 等元数据 | 状态槽系统启用后 |
| `<intent>` | 角色主动表达意图(写信/送礼/弹窗等),Codex 思路文档 2.1 节 | Action 系统启用后 |
| `<wb>` | 自动建档世界书条目(`title` `content` `tags` 等),Codex 提及的"AI 输出 → 规则触发 → 建档"模式 | 后处理规则系统启用后 |
| `<mem_draft>` | 记忆草稿(label / importance / tags),进入记忆系统的写入管线 | 记忆系统启用后 |

设计原则(对未来的我):

- 新增子标签时,**老解析器不需要改动**——直接在子标签 dispatcher 加一个 case 即可
- 新子标签内部仍**优先用行式 + `|` 分隔元数据**,与 `<var>` `<event>` 风格保持一致;只有结构很复杂的(如 `<state_proposal>` 含 exit_conditions 数组)才考虑嵌套
- 嵌套结构超过 2 层时,优先考虑改用 JSON 内嵌(`<state_proposal>{...JSON...}</state_proposal>`),避免行式语法被强行复杂化

## 八、与旧协议的关系

### 8.1 阶段 2 落地后**直接下线**的旧机制

| 旧机制 | 替代 |
|-------|------|
| 顶层 `<var>...</var>`(无 `<sys>` 包裹) | `<sys><var>...</var></sys>` |
| 内联 `[EVENT:id:outcome]` | `<sys><event>id: outcome</event></sys>` |

### 8.2 旧机制实现位置(阶段 2 步骤 2.4 移除/改名)

- [server/services/values.ts](server/services/values.ts) 的 `extractVarBlock`、`parseAndApplyVarBlock` —— 解析部分删除,数据库写入部分改名 `applyVarBlock`
- [server/services/eventEngine.ts](server/services/eventEngine.ts) 的 `parseOutcomeFromAIResponse` —— 解析部分删除,outcome 写入与 branch 触发部分改名 `applyEventOutcomes`
- [server/routes/chat.ts](server/routes/chat.ts) 流式/非流式两路 —— 改为单点调用新函数 `parseAIOutput(content)`
- [server/services/context.ts](server/services/context.ts) 的 sys-variables 槽内引导文本 —— 改为本协议第六节的样板

### 8.3 不留兼容期

- 旧格式不再被解析,直接当作正文一部分留在 cleanContent 里(用户能看到 `[EVENT:...]` 这种字符串,作者会立刻发现并修)
- 数据库无真实历史依赖(开工前已 grep 确认),无迁移成本

## 十、开放问题(暂不阻塞)

1. **AI 输出嵌套 `<sys>` 内 `<sys>`**:目前未定义,按"未识别子标签"处理(整段忽略)。如果出现频繁,后续考虑容错。
2. **`<sys>` 块出现位置**:本协议规定"回复末尾",但 AI 可能写在中间。当前实现按"找最后一个 `<sys>` 标签对"提取,正文里残留的散落标签需要在测试中观察。

> 历史限制:旧 [eventEngine.ts](server/services/eventEngine.ts) 用 `'branch:<outcome>'` 字段拼接表示分支连接的 required outcome,导致 outcome 不能含 `:`。已通过 migration 0004 给 `event_connections` 加独立 `required_outcome` 字段,该限制取消。outcome 现可为任意字符串。
