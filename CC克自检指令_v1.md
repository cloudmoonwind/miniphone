## 一、任务定位

这份清单不是开发任务、不是重构任务、不是审查任务。**这是一份现状盘点——把你已经做了什么、用什么方式做的、效果如何、记录下来。**

为什么需要这件事：项目走到现在已经有数值系统、事件系统、双模式切换、世界书、API 通信等多个子系统在并行推进。设计文档在变化、实现在变化，文档和实现之间的同步状态不清楚。在推进任何新功能前，需要先把"地基现状"摸清楚。

**你不需要为这份清单的内容辩护。** 实现得不完整、不一致、有偏离设计文档的地方都正常。这份清单的价值不在"展示成果"，在于"准确反映现状"。如果你倾向于把不完整的部分说得完整一些，那就违背了这份清单存在的目的。

## 二、输出格式约束（强制）

### 2.1 总体格式

每一项问题必须包含以下结构，**任何一项都不能省略**：

```
### [问题编号] [问题简称]

**状态**：[已实现完整 / 部分实现 / 未实现 / 不适用]

**相关文件**（列出所有相关文件的绝对路径或项目根目录的相对路径）：
- src/xxx/yyy.ts
- src/xxx/zzz.ts

**关键代码**（贴出最能反映实际逻辑的代码片段，不超过 50 行；如片段不足以说明，可分多段，每段独立标注来源）：
```typescript
// from src/xxx/yyy.ts:行号
[code]
```

**实际行为**（运行时实际发生了什么；如果有日志/输出能贴的，贴出来）：
[描述]

**与设计文档的偏离**（如果有任何与现有设计文档不一致的地方，逐条列出；如果没有，明确写"无偏离"）：
- [偏离 1]
- [偏离 2]

**已知问题**（你自己知道但还没修的 bug、未完成的部分、有疑虑的地方）：
- [问题 1]
- [问题 2]

**该项备注**（任何上面没覆盖到、但 Agnes 应该知道的事）：
[内容]
```

### 2.2 禁用词

以下表达**不允许出现**在你的回答里：

- "基本完成"、"基本实现"、"基本可用"
- "差不多"、"大致"、"大概"
- "应该是"、"应该没问题"、"应该可以"
- "我记得"、"我印象中"
- "暂时这样"、"先这样"（如果是临时方案，明确标注"临时方案，原因：XXX，长期计划：XXX"）
- "可能"（除非用于描述明确不确定的情况，如"这个 bug 可能与 X 有关，我没复现过")

如果某件事你不确定，**直接说"不确定"或"未验证"**，比用模糊词好。

### 2.3 不许跳项

13 项问题里每一项都必须有回答。如果某项不适用（比如问到的功能你根本没做），明确写：

```
**状态**：未实现
**相关文件**：无
**关键代码**：无
**实际行为**：该功能尚未开始实现
**与设计文档的偏离**：N/A（未实现，无法谈偏离）
**已知问题**：无
**该项备注**：[说明为什么没做：是没轮到、是没思路、是觉得不需要、还是别的原因]
```

**沉默不算回答。** 漏掉一项等于这份清单作废。

### 2.4 必须包含的实证素材

清单结尾必须附上以下三段实证（独立成节，标题为"实证素材"）：

1. **启动命令与运行截图/日志**：从空状态启动应用到看到角色第一句回复，需要执行哪些命令、点哪些按钮、跑出什么样的日志
2. **数据库结构原样输出**：连接到当前数据库后执行 `\d` 或等价命令的完整输出（所有表、所有列）
3. **一次真实对话的完整原始 prompt**：从 Agnes 加的终端日志监控里，粘贴一次最近的真实对话产生的完整原始 prompt（包含所有注入内容、世界书条目、变量替换后结果）

这三段是反阳奉阴违的硬证据。前面 13 项可以靠话术包装，这三段没法。

## 三、详细问题清单

### 1. 数据库

实际使用的数据库是哪一种（PostgreSQL / SQLite / 其他）。如果与设计文档（动态系统设计文档说 SQLite，记忆系统 Schema 说 PostgreSQL，需求文档说 PostgreSQL）有冲突，明确说你实际用的是哪个、为什么选这个、选型决定是谁做的。

### 2. 数据库表结构

完整列出当前数据库里所有表，每张表的列名、类型、约束。**不要只列你认为重要的表，全部列出来**。包括 migrations 表、临时表、调试表、所有的。

### 3. 版本号与迁移机制

你之前实现的版本号机制——具体是怎么实现的？版本号字段在哪张表的哪一列？升级时怎么决定要跑哪些迁移脚本？迁移脚本目录在哪？最近一次迁移做了什么？

### 4. 对话循环完整调用链

用户在聊天框打字、按发送、到屏幕上看到角色回复——这条链路上经过的所有函数，按顺序列出，每个函数的文件路径和行号。包括：消息接收、上下文组装、API 调用、流式接收、消息渲染、消息持久化。

### 5. AI 调用

- 用了哪个 SDK / HTTP 客户端
- 当前默认用什么模型
- prompt 模板存在哪里（文件路径或数据库表）
- 失败处理：API 报错时会怎么样、用户能看到什么、有没有重试

### 6. 上下文组装详情

组装一次对话的上下文具体包含哪些部分，按在 prompt 里的位置顺序列出（系统提示词、角色设定、世界书激活条目、历史消息、注入内容、用户当前输入等）。每一部分的来源（哪个表、哪个变量）。

### 7. 数值系统现状

- `character_values` / `value_stages` / `value_rules` 三张表是否都建了
- 增删改查接口在哪
- 设计文档里的占位符语法 `{{v:affection:desc}}` 是否实现了，在哪个文件解析，**实际组装的 prompt 里能不能看到 prompt_snippet 的内容**——这个是关键，请贴一段能证明它真的注入了的实际 prompt
- value_rules 的 trigger_on（chat_end / time_pass / receive_gift / event_complete / value_change）哪些实现了哪些没实现
- 数值变化的范围规则（不同 range_min/range_max 用不同规则）实现了吗

### 8. 事件系统现状

- events 主表的所有字段是否实现了（status / priority / probability / repeatable / max_triggers / unlock_conditions / trigger_conditions / effects / 三种 cooldown / steps 等）
- event_tags / event_connections / condition_subscriptions / pending_injections 这四张辅助表分别建了没、用了没
- 状态机（locked → pending → active → completed）的转换逻辑在哪个文件
- 解锁条件 vs 触发条件是否真分成两层，还是合并成一层处理
- 三种冷却（按时间、按轮次互斥；按条件次数叠加）实现了吗
- condition_subscriptions 是真在用（即条件变化时只检查订阅了的事件）还是退化成全表扫描
- 事件的"成功/失败判断"问题（设计文档第 594 行待确认问题）你目前是用哪种方案——AI 自判 / 副 API / 用户选 / 后续条件推进 / 还没做

### 9. 双模式切换现状

- UI 上的"线上/线下"切换按钮做了什么
- **切换是否真的影响发给 AI 的 prompt**——请贴出"切换前的 prompt"和"切换后的 prompt"，让 Agnes 直观对比
- 如果只切换前端显示没切 prompt，明确承认
- 输出截断 bug 的现状：什么场景出、出的是什么样、有没有定位到原因

### 10. 世界书现状

- 数据结构（条目的字段）
- 触发条件（关键词、常驻、向量匹配等）实现了哪些
- 注入位置（角色描述前/后、深度 N 等）实现了哪些
- 一个条目从被触发到被注入到 prompt 里的完整链路

### 11. 端到端测试

- 当前有没有自动化测试代码（单元测试、集成测试、端到端测试任意类型）
- 测试在哪个目录、用什么框架、怎么跑
- 哪些路径有覆盖、哪些没有
- **如果完全没有测试，明确写"无任何自动化测试"**——不要回避

### 12. 日志与可观测性

Agnes 之前让你做的"终端功能监控每次发给 AI 的原始内容和 AI 回复的原始内容"——

- 这个监控的实现位置
- 除了原始 prompt 和原始回复，是否还记录了**触发链路**（哪些世界书条目被激活、哪些 pending_injections 被消费、哪些数值规则触发了、检索到什么内容/被汇总层挤掉什么）
- 如果只有原始内容没有触发链路，明确说

### 13. 大富翁/梦境/日历/前端各页面状态

每个页面单独说：
- 哪些 UI 元素是真的接了后端逻辑、哪些是占位
- 哪些数据是从数据库读的、哪些是 mock
- 还差什么才能跑通完整功能

## 四、Agnes 使用本清单的方式

为了让你理解为什么要这么严格，告知一下 Agnes 拿到这份清单后会怎么做：

1. 逐项核对设计文档（动态系统设计、记忆系统、需求文档第二版、本轮共识文档 v1）与你的实现描述

3. "与设计文档的偏离"那一节会被重点关注——这是后续讨论是否接受偏离/要求修改的依据
4. 实证素材里的"完整原始 prompt"会被用来核对：你声称已实现的注入机制、变量替换、世界书激活，在真实 prompt 里是不是真的发生了

如果发现你的清单与代码事实有差距（比如你写"已实现"但实际代码里没有），后续讨论的协作信任会受到影响。**所以请把"如实写"放在"看起来好"前面。**

## 五、不需要做的事

为避免误解，明确几件**不需要**做的事：

- **不需要修复 bug**。这是盘点不是修复。已知问题列出来即可。
- **不需要补全设计文档没要求的功能**。如果设计文档某项你没做，照实写"未实现"，不要临时补。
- **不需要重构代码使其"看起来更好"**。代码现状是什么样就是什么样。
- **不需要写新文档**。你的输出就是这份清单本身，不需要再产生其他文档。
- **不需要给出未来计划**。这份清单只反映"现在"，不反映"将来"。
