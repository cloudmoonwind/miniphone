# 事件系统实施计划 v2

> 基于 [事件系统设计讨论记录](Codex讨论沉淀/事件系统设计讨论记录.md) §九 修订决策版本。
> 取代旧版 EVENT_SYSTEM_PLAN.md。

## 一、目标

把事件系统重构为「定义层 + 运行实例层」架构：
- 事件定义（`events`）只描述「这个事件是什么」，不再保存任何角色相关运行态。
- 事件图（`event_graphs`）描述「这一组事件如何连接」，属于定义层。
- 事件图运行实例（`event_graph_runs`）描述「某角色 / 功能 / 预设 当前在跑哪张图」。
- 事件状态（`event_states`）按 `graph_run_id + event_id` 存放运行态，同一事件可被多角色多功能独立运行。

## 二、不在范围内（第一版不做）

- 事件模板与图节点实例拆分
- 强互斥组、事件组、同时激活组
- 复杂事务回滚
- AI intent planner
- 单独事件日志表（沿用现有 trace 系统）
- `effect.visibility` 字段
- `result_marking_mode='ai_suggest_user_confirm'`

### 阶段 1：新 schema 落地

**任务：** 在 [server/db/schema.ts](../server/db/schema.ts) 中：

1. `event_books`：保留现有字段（含 `scope / character_id`），用于编辑器作者范围。
2. `events`（瘦身）：保留 `id / book_id / name / description / enabled / priority / probability / max_triggers / unlock_conditions / trigger_conditions / effects / cooldown_type / cooldown_value / result_marking_mode / created_at / updated_at`。删除 `repeatable / character_id / status / outcome / trigger_count / cooldown_remaining / condition_cooldown / condition_cooldown_remaining / steps / current_step / last_triggered_at / weight`。
3. `event_tags`：保留。
4. **新增** `event_graphs`：`id / name / description / book_id?(可选) / enabled / created_at / updated_at`。
5. **新增** `event_graph_runs`：`id / graph_id / character_id / function_key / preset_id / is_active / turn_counter(default 0) / started_at / ended_at? / created_at / updated_at`。唯一约束：`(character_id, function_key)` 同时只允许一条 `is_active=1`。
6. **新增** `event_states`：`id / graph_run_id / event_id / status / result_status / result_summary / trigger_count / last_triggered_at / next_available_at / next_available_turn / condition_hit_count / created_at / updated_at`。唯一约束：`(graph_run_id, event_id)`。
7. `event_connections`（重写）：`id / graph_id / from_event_id / to_event_id / relation_type('unlock' | 'trigger') / created_at / updated_at`。删除 `required_outcome`。
8. `condition_subscriptions`（重写）：增加 `graph_run_id` 字段；`condition_type` 枚举追加 `event_result`。
9. `pending_injections`（重写）：把 `character_id` 替换为 `graph_run_id`。

`drizzle-kit generate` 生成迁移 `0006_event_system_v2.sql`。

**验收：** 迁移跑通；`tsc --noEmit` 服务层导入新表类型无报错。

### 阶段 3：调度器重写

**目标：** 重写 [server/services/eventEngine.ts](../server/services/eventEngine.ts) 核心调度。

**任务：**
1. `buildSnapshot(graphRunId)`：含 `values / worldState / event_states / turn_counter / 时间`。
2. `checkUnlockConditions(graphRunId, snapshot)`：把符合解锁条件的 locked → pending；进入 pending 后**立即**检查 trigger_conditions，满足则进入候选池。
3. `findCandidates(graphRunId, ctx, snapshot)`：通过 `condition_subscriptions` 索引（按 `graph_run_id` 过滤），或全量扫该 graph_run 的 pending 状态。
4. `canFire(state, evtDef, snapshot)`：依次检查
   - `next_available_at` vs `now`（`cooldown_type='time'`）
   - `next_available_turn` vs `turn_counter`（`cooldown_type='turn'`）
   - `condition_hit_count` vs `cooldown_value`（`cooldown_type='condition_hit'`）
   - `max_triggers` vs `trigger_count`
   - `trigger_conditions` 评估
   - `probability` 概率
5. `fireEvent(graphRunId, evt, ctx, snapshot)`：
   - 执行 effects
   - 更新 event_state：`trigger_count += 1`、`last_triggered_at`、`next_available_*` 按 cooldown_type 设
   - 触发后回流：若已达 `max_triggers` → `status='completed'`；否则**重新检查 unlock 条件**，回 `pending` 或 `locked`
   - 若事件 `result_marking_mode != 'none'` → 进入 `active` 等结果，`processConnections` 推迟到结果落地后
6. `processConnections(graphRunId, fromEventId, snapshot)`：
   - `unlock`：调用 `tryUnlock(toEventId)`（走完整 unlock 条件检查）
   - `trigger`：把 toEventId 加入候选池，**走完整 canFire**，不绕过任何检查
7. `tickTurnCounter(graphRunId)`：chat_end 时 `turn_counter += 1`。**取代旧的 tickCooldowns**。
8. 候选池规则：同一事件同一轮只执行一次；按 `priority desc, event_id asc` 稳定排序；本轮触发上限默认 1（可配置）；`processConnections` 触发的连锁最多走 1 层（防递归）。

**验收：** smoke `server/scripts/smoke-event-engine-v2.ts` 覆盖 locked → pending → trigger → completed/回流 全流程；canFire 各分支单测覆盖。

### 阶段 4：effects 重新对齐

**目标：** 沿用现有 effect 注册表设计，改写内置 effect 适配新 schema。

**任务：**
1. `inject_prompt` → 写 `pending_injections.graph_run_id`。
2. `modify_value` → 不变（走 values 服务，由 `onValueChange` 唤醒）。
3. `set_outcome` → 写 `event_states.result_status / result_summary`，调用 `onEventResultChange` 唤醒订阅。
4. `trigger_event` → 默认 `graph_run_id` 为发出者所在 run；走完整 `canFire`，不绕过条件。
5. `unlock_event / lock_event` → 操作 `event_states.status`，不操作 events 定义。
6. `change_location / record_history` → 不变。

**验收：** 每个 effect smoke 通过。

---

### 阶段 5：订阅唤醒接入

**目标：** 各类状态变化能精确唤醒订阅事件。

**任务：**
1. `onValueChange(charId, varName)` — 在 [server/services/values.ts](../server/services/values.ts) `adjustValue` 后调用，遍历该 char 所有活跃 graph_run，按 `(graph_run_id, 'value', varName)` 找订阅事件并唤醒。
2. `onWorldStateChange(key, value)` — location / weather / time。
3. `onEventResultChange(graphRunId, eventId)` — set_outcome 或 `applyEventOutcomes` 调用后触发。
4. `onChatEnd(charId, functionKey)` — 取活跃 graph_run → `tickTurnCounter` → `consumeInjectionTurns` → `checkAndFireEvents(trigger='chat_end')`。
5. **所有唤醒入口均按 `graph_run_id` 过滤订阅**，避免跨 run 污染。

**验收：** 数值变化能精确唤醒对应订阅事件；事件 outcome 变化能唤醒下游。

---

### 阶段 6：注入层接入 context

**任务：**
1. [server/services/context.ts](../server/services/context.ts) 找当前 `(charId, functionKey)` 的活跃 graph_run。
2. 查 `pending_injections.graph_run_id` 拼入上下文（按 position 顺序）。
3. 上下文组装完成后调 `consumeInjectionTurns(graphRunId)`。

**验收：** 触发 inject_prompt effect 后，下一轮 AI 上下文确实包含注入；持续 3 轮的注入恰好出现 3 次。

### 阶段 7：AI 协议层

**目标：** [server/services/aiProtocol.ts](../server/services/aiProtocol.ts) 解析后的 outcome 写入 event_states。

**任务：**
1. `applyEventOutcomes(graphRunId, parsed)` — 写 `event_states.result_status / result_summary`，触发 `onEventResultChange`。
2. `result_marking_mode='user'` 的事件不应被 AI 标记（跳过并 trace）。
3. AI 标记成功后，事件从 `active` 回流到 `locked / pending / completed`（按 max_triggers 与 unlock_conditions 决定）。

**验收：** AI 标记 evt_x → success，event_states 正确更新，下游订阅被唤醒。
