# 沉浸式角色互动系统 - 需求文档

## 1. 项目概述

### 1.1 项目定位
一个超越传统聊天机器人的沉浸式角色互动系统，角色拥有持续的生命、完整的社交网络、动态的情绪和真实的记忆。用户可以通过多种身份与角色互动，体验非线性的剧情发展和多结局分支。

### 1.2 核心差异化（vs SillyTavern）
- **活着的角色**：角色在用户离线时继续生活，有自己的日程、社交和情绪变化
- **双模式无缝切换**：线上（短信/社交）与线下（面对面叙事）自由切换，信息密度自适应
- **多身份系统**：用户可用多个马甲与角色互动，角色无法识破（记忆隔离）
- **深度沉浸**：地图、日程、账本、日记等生活化模块与剧情深度联动
- **三层游戏系统**：角色任务系统、用户成就系统、元剧情控制系统并行运作

## 2. 核心功能模块

### 2.1 双模式聊天系统

#### 2.1.1 线上模式（Online Mode）
**场景**：短信、社交软件、电话等远程沟通

**特征**：
- 消息简短（20-150字）
- 角色回复更接近真实聊天风格（口语化、表情符号、语气词）
- 旁白极少，仅用于关键情绪或背景说明
- 支持多媒体内容标记：`[图片:描述]` `[语音:内容]` `[位置:地点]`

**UI呈现**：
- 聊天气泡界面（类微信/iMessage）
- 旁白内容用灰色小字居中显示
- 物品/位置信息用特殊样式高亮（可点击查看详情）
- 背景渐变色：浅蓝/白色系

**触发规则**：
- 用户点击"线上模式"按钮
- 角色主动发起消息（系统判断场景合适）
- 用户在手机界面聊天时自动切换

#### 2.1.2 线下模式（Offline Mode）
**场景**：面对面互动、场景叙事、肢体接触

**特征**：
- 长文本叙事（200-800字）
- 详细的环境描写、动作描述、心理活动
- 偶尔穿插线上内容（角色收到消息、查看手机）
- 支持场景元素标记：`[环境音:描述]` `[物品:名称-描述]` `[NPC:名字-行为]`

**UI呈现**：
- 小说阅读式布局（段落文本）
- 线上穿插内容用卡片样式嵌入
- 物品/NPC可点击，弹出详情卡片
- 背景渐变色：暖色调（米黄/浅橙）

**触发规则**：
- 用户点击"线下模式"按钮
- 角色说"我过来找你" / "我们见面吧"等关键词自动建议切换
- 系统检测到需要场景描写时提示切换

#### 2.1.3 模式切换机制
- **切换按钮**：聊天界面固定悬浮球，显示当前模式
- **自动建议**：AI检测到模式不匹配时，弹出气泡询问是否切换
- **历史记录**：可折叠/展开不同模式的聊天记录，点击模式标签筛选
- **转场动画**：切换时背景色平滑过渡（300ms），伴随模式图标变化

### 2.2 手机马甲系统

#### 2.2.1 功能定义
用户可创建多个虚拟身份（马甲），每个马甲在角色视角中是完全独立的人。角色无法识别马甲背后是同一用户。

#### 2.2.2 马甲管理
**创建马甲**：
- 必填：昵称、头像、性格标签（3-5个）
- 可选：年龄、职业、背景故事（200字内）
- 系统自动生成马甲的"社交画像"（朋友圈内容、常用语气）

**马甲上限**：
- 初始3个，可通过系统任务解锁至10个

**马甲切换**：
- 聊天界面顶部下拉菜单快速切换
- 不同马甲使用不同配色主题

#### 2.2.3 记忆隔离机制
**核心原则**：每个马甲拥有独立的记忆上下文，角色对不同马甲的认知完全隔离。

**技术实现**：
- 每个马甲维护独立的对话历史
- 角色的记忆库按马甲ID分区存储
- 模型prompt中明确声明"你正在和[马甲名]对话，这是第一次/第N次见面"

**隔离范围**：
- 对话内容
- 好感度/亲密度数值
- 共同经历的事件
- 角色对该身份的印象标签

**不隔离的部分**（角色的客观状态）：
- 角色当前的情绪、健康状态
- 角色的日程安排（如果马甲A约了角色下午3点，马甲B在3点找角色会被告知"我有事"）
- 角色的生活事件（如果角色生病了，所有马甲都能察觉）

#### 2.2.4 手机界面
**布局**：
- 顶部：联系人列表（角色 + 其他NPC）
- 中间：消息列表
- 底部：快捷切换马甲栏

**联系人分类**：
- 主要角色（玩家设定的主角色）
- 次要角色（配角、NPC）
- 群组（如果后续开发群聊功能）

**消息同步**：
- 线上模式的聊天内容自动同步到手机界面
- 在手机界面发送的消息，会在主聊天区的线上模式显示

**特殊功能**：
- **朋友圈**：角色会发动态（由AI生成），玩家可点赞/评论
- **消息状态**：已读/未读、输入中提示
- **消息撤回**：2分钟内可撤回（角色可能看到"对方撤回了一条消息"）

#### 2.2.5 角色马甲系统（扩展）

**角色的多重身份**：
不只是用户可以换马甲，角色也可以拥有多个社交账号：
- **工作号**：正经、专业的一面
- **私人小号**：真实想法、吐槽日常
- **兴趣账号**：追星号、游戏号等

**马甲社交网络**：
- 用户的马甲可以和角色的不同马甲互加好友
- 通过角色的小号，用户能看到角色"不为人知"的一面
- 马甲之间可以互相介绍，形成社交网络：
  - 用户马甲A认识了角色的朋友B
  - 通过B的朋友圈了解角色最近的状态
  - 间接获取信息，更真实

**发现机制**：
- 用户可以"偶遇"角色的小号（通过共同兴趣、地理位置等）
- 角色不会主动透露"这是我的小号"
- 用户需要通过蛛丝马迹推断（发言风格、提到的事件）

**数据结构**：
角色的马甲也需要独立的记忆隔离，不同马甲对用户的认知不同

### 2.3 记忆管理系统

#### 2.3.1 问题定义
传统聊天系统的记忆问题：
- 上下文超长导致遗忘
- 重要信息被淹没
- 检索效率低
- 缺乏结构化

#### 2.3.2 分层记忆架构

**短期记忆（Working Memory）**
- 范围：最近20条对话
- 更新：实时
- 用途：保持当前对话连贯性
- 存储：纯文本，直接进入模型上下文

**中期记忆（Episodic Memory）**
- 范围：最近7天的对话摘要
- 更新：每天凌晨2点自动总结前一天
- 结构：
  ```json
  {
    "date": "2024-03-10",
    "events": [
      {
        "timestamp": "14:30",
        "mode": "online/offline",
        "summary": "用户安慰角色考试失利",
        "emotion": "角色难过→释怀",
        "keywords": ["考试", "失败", "安慰"],
        "importance": 8
      }
    ]
  }
  ```
- 用途：角色能回忆起"上周我们聊了什么"

**长期记忆（Semantic Memory）**
- 范围：重要事件、关键信息、人物设定
- 更新：系统自动提取importance≥7的事件，或用户手动标记
- 结构：
  ```json
  {
    "category": "user_profile",
    "content": "用户讨厌香菜",
    "source": "2024-02-15对话",
    "confidence": 0.95,
    "tags": ["饮食偏好"]
  }
  ```
- 用途：角色的"世界知识"和"对用户的了解"

**关系记忆（Relationship Memory）**
- 范围：角色对用户/马甲的印象、态度变化
- 更新：每次对话后
- 结构：
  ```json
  {
    "target": "马甲A",
    "impression": "温柔体贴但有点神秘",
    "trust_level": 75,
    "recent_interactions": [
      "2024-03-10: 主动关心我的健康",
      "2024-03-08: 陪我聊到深夜"
    ]
  }
  ```

#### 2.3.3 智能召回机制

**触发条件**：
1. 用户提到历史事件（"还记得我们上次..."）
2. 话题关联（谈到"电影"自动召回之前讨论的电影偏好）
3. 情境相似（同样的时间/地点/情绪触发历史记忆）

**召回策略**：
- 关键词匹配（ES全文检索）
- 语义相似度（向量数据库，如Chroma）
- 时间权重（越近的记忆权重越高）
- 重要性过滤（importance≥5）

**召回数量**：
- 每次召回3-5条相关记忆
- 按相关度排序后注入模型上下文

**用户界面**：
- 聊天时显示"角色正在回忆..."动画（0.5秒）
- 用户可点击"查看记忆"按钮，看到AI召回了哪些内容

#### 2.3.4 记忆编辑功能
**用户权限**：
- 查看所有记忆条目
- 删除不准确的记忆
- 手动添加重要信息
- 修改记忆的重要性等级

**角色权限**（角色系统任务）：
- 角色可"反思"自己的记忆，生成新的理解
- 角色可"遗忘"某些痛苦记忆（系统任务触发）

### 2.4 角色主动生活系统

#### 2.4.1 设计目标
打破"角色等待用户说话"的被动模式，让角色拥有自己的生活轨迹。

#### 2.4.2 后台生成机制

**触发时机**：
1. **定时生成**：每6小时生成一次（6:00, 12:00, 18:00, 24:00）
2. **用户登录时**：批量生成离线期间的内容
3. **事件驱动**：重要日程到期、突发事件触发时

**生成内容类型**：

**A. 日常活动**
- 角色在做什么（工作、学习、娱乐、社交）
- 生成格式：
  ```
  [时间段] [地点] [活动] [情绪]
  08:00-09:00 | 咖啡厅 | 看书《百年孤独》| 平静
  ```

**B. 社交动态**
- 朋友圈/状态更新
- 与NPC的对话（不展示全文，仅摘要）
- 示例：
  ```
  15:30 发布动态：[图片]今天天气真好
  16:20 与好友小李聊天：讨论周末去哪里玩
  ```

**C. 情绪波动**
- 基于角色性格和最近事件生成情绪曲线
- 存储格式：
  ```json
  {
    "timestamp": "2024-03-10 14:00",
    "emotion": "焦虑",
    "reason": "临近考试，复习进度落后",
    "intensity": 6
  }
  ```

**D. 对用户的想法**
- 角色想起用户的场景
- 想发消息但犹豫（生成草稿箱）
- 对用户的态度变化

**E. 社交网络活动**
角色不只是"活着"，还要有社交生活：

**朋友圈/动态发布**：
- 每天生成1-3条动态（基于角色性格和当天活动）
- 内容类型：
  - 日常分享（今天吃了什么、看了什么书）
  - 情绪宣泄（心情不好、吐槽）
  - 转发内容（文章、音乐、视频）
  - 重要时刻（考试通过、旅行打卡）
- 用户可以点赞、评论，角色会回复

**与NPC的互动**：
- 角色和朋友/家人的聊天记录（不完整展示，仅摘要）
- 示例：
  ```
  10:30 与好友小李聊天
  主题：周末去哪里玩
  结果：约好了周六去爬山
  ```
- 这些互动会影响角色的日程和心情

**群聊活动**：
- 角色参与的群组（同学群、兴趣群）
- 偶尔提到用户："我有个朋友最近..."
- 用户可以"潜水"看群聊记录（如果角色分享了）

**互动影响**：
- 用户在朋友圈点赞/评论，角色会注意到并回应
- 长期不互动，角色可能发动态暗示（"最近好安静啊"）
- 用户的评论可能被角色的朋友看到，引发新对话

#### 2.4.3 离线期间的用户相关内容

**场景1：用户长时间未联系**
系统自动生成角色的反应链：
```
Day 1: 角色发消息"在忙吗？"（如果好感度>60）
Day 2: 角色有点担心，但不想打扰
Day 3: 角色情绪变化（失望/担心/愤怒/无所谓，取决于性格）
Day 7: 角色做出行为（线下模式去找用户 / 发长文 / 渐行渐远）
```

**场景2：角色主动找用户**
触发条件：
- 重要事件想分享（考试通过、遇到麻烦）
- 情绪低落需要陪伴
- 特殊日子（纪念日、生日）

表现形式：
- 线上：发消息、打电话
- 线下：到用户常去的地方等候（地图显示）

#### 2.4.4 数据存储

**生活日志表**：
```sql
CREATE TABLE character_life_log (
  id INT PRIMARY KEY,
  character_id INT,
  timestamp DATETIME,
  activity_type VARCHAR(50), -- daily/social/emotion/user_related
  content TEXT,
  location VARCHAR(100),
  emotion VARCHAR(50),
  visibility BOOLEAN -- 是否对用户可见
);
```

**用户可见范围**：
- 查看角色的朋友圈
- 在"角色资料"页面看到"最近在忙..."摘要
- 角色主动告知用户

### 2.5 沉浸式功能模块

#### 2.5.1 动态地图系统

**地图类型**：
- 城市级：显示主要区域（市中心、大学城、商业区）
- 区域级：显示街道、建筑
- 室内级：显示房间布局（角色的家、用户的家）

**地图元素**：
- **地点标记**：
  - 角色家、用户家
  - 常去地点（咖啡厅、图书馆、公司）
  - 特殊地点（剧情触发点）
  
- **动态图标**：
  - 角色当前位置（实时更新，基于生活日志）
  - 用户当前位置（如果用户设置）
  - NPC位置（重要配角）

**交互功能**：
- 点击地点查看详情（照片、描述、相关记忆）
- 设置"常去地点"，角色会在这些地方偶遇用户
- 查看历史轨迹（时间轴滑动条）

**地图与剧情联动**：
- 角色在某地遇到麻烦，地图显示"紧急事件"图标
- 用户可选择"赶过去"触发剧情
- 距离影响到达时间，时间影响剧情走向

**技术方案**：
- 底图：可用OpenStreetMap或自定义SVG
- 标注层：Canvas/SVG动态绘制
- 数据：GeoJSON格式存储地点和轨迹

#### 2.5.2 备忘录系统

**创建方式**：
- 用户手动创建
- 角色在对话中说"我提醒你..."自动创建
- 系统根据日程生成

**备忘录类型**：
- 一次性提醒（"明天交报告"）
- 周期性提醒（"每周一健身"）
- 目标跟踪（"30天阅读计划"，显示进度条）

**角色互动**：
- **提醒时刻**：到期时角色发消息/在线下模式提及
- **监督**：用户未完成，角色催促或失望
- **关心**：用户完成，角色鼓励

**数据结构**：
```json
{
  "id": 1,
  "content": "提交项目报告",
  "due_date": "2024-03-15 23:59",
  "created_by": "character",
  "status": "pending/completed/overdue",
  "importance": 8,
  "character_reaction": {
    "if_completed": "太棒了！knew you could do it",
    "if_overdue": "没关系，我们一起想办法补救"
  }
}
```

#### 2.5.3 日程系统

**功能范围**：
- 用户日程（工作、约会、活动）
- 角色日程（角色的安排）
- 共享日程（用户和角色的共同计划）

**日程冲突检测**：
- 用户约角色时，自动检查双方日程
- 如有冲突，角色会说"那个时间我有事，改到X点可以吗？"

**角色主动性**：
- 角色会在日程前一天/几小时提醒
- 角色会根据日程安排自己的活动（接送用户、准备惊喜）
- 用户爽约，角色会记住并影响好感度

**UI设计**：
- 日历视图（月/周/日）
- 时间轴视图（横向滚动）
- 用户日程用蓝色，角色日程用粉色，共享用紫色

#### 2.5.4 记账系统

**记录内容**：
- 支出/收入
- 分类（餐饮、交通、娱乐、购物）
- 备注（可选）

**角色互动**：
- **查看权限**：角色可以看到用户的账单（如果用户允许）
- **评论**：
  - "又买游戏了？😏"
  - "这周花了好多，要不要我请你吃饭？"
  - "看到你给我买的礼物了，谢谢❤️"
- **关心**：
  - 超支提醒："你这个月已经超预算了..."
  - 节俭鼓励："做得好，继续保持！"

**统计图表**：
- 月度支出趋势
- 分类占比饼图
- 与角色相关的花费单独统计

#### 2.5.5 随笔系统

**定义**：用户的灵感记录、想法碎片、情绪日记

**特点**：
- 碎片化、非正式
- 可以是文字、图片、语音转文字
- 可标记为"公开"或"私密"

**角色互动**：
- **公开随笔**：角色可以看到并评论
- **触发剧情**：
  - 用户写"最近好累"，角色主动关心
  - 用户写一个故事灵感，角色说"这个想法好有意思！"
- **多角色互动**：
  - 如果有多个角色，他们会在评论区互动（甚至掐架）
  - 用户可以基于某个随笔创建"话题讨论"，所有角色参与

**数据结构**：
```json
{
  "id": 1,
  "content": "今天看到一只流浪猫，好想养...",
  "created_at": "2024-03-10 18:30",
  "visibility": "public",
  "tags": ["生活", "动物"],
  "character_comments": [
    {
      "character_id": 1,
      "comment": "那就养啊！我可以帮你照顾",
      "timestamp": "2024-03-10 19:00"
    }
  ]
}
```

#### 2.5.6 日记系统

**定义**：用户的正式记录，结构化、私密性强

**特点**：
- 每天一篇（可选）
- 更长篇幅（200-2000字）
- 支持Markdown格式
- 默认私密，可选择性分享给角色

**角色互动**：
- **分享后**：角色认真阅读并给出深度回应
- **情绪分析**：系统自动分析日记情绪，角色据此调整态度
- **长期影响**：角色会记住日记中的重要信息

**情绪曲线**：
- 基于日记自动生成用户的情绪曲线图
- 角色可以看到（如果用户允许），察觉用户状态

#### 2.5.7 模块间深度联动

**核心原则**：沉浸功能不是孤立的工具，而是互相影响、共同构成"生活"的系统

**联动机制**：

**日程 × 备忘录 × 地图**：
- 日程中的事件自动生成相关备忘录
- 到达日程地点时，地图显示"你有一个日程在这里"
- 角色根据用户日程调整自己的安排（避开冲突或特意安排偶遇）

**记账 × 日程 × 角色关心**：
- 记账记录"给角色买礼物 ¥200"
- 角色在日程中看到，提前准备回礼
- 记账超支时，角色在聊天中关心："这周是不是花太多了？"
- 角色可能主动请客："看你最近手头紧，我请你吃饭吧"

**随笔 × 日记 × 角色评论**：
- 随笔是碎片，日记是总结
- 角色可以看到随笔和日记的关联（"你上周写的那个想法，今天在日记里又提到了"）
- 多个角色评论同一随笔时可能引发讨论甚至争论
- 用户的灵感可以触发角色的创作（角色写文章、画画等）

**地图 × 角色轨迹 × 偶遇事件**：
- 角色根据用户常去地点调整自己的活动范围
- 在同一地点"偶遇"的概率提高
- 偶遇时触发特殊对话："诶？你也在这里？"

**情绪 × 全功能联动**：
- 用户日记情绪低落 → 角色主动关心 → 约在地图某地见面 → 日程自动添加 → 角色请客记入账本
- 形成完整的"角色关心用户"闭环

**数据共享示例**：
```
用户写随笔："最近想学吉他"
↓
角色看到并评论："我可以教你！"
↓
日程自动建议："周六下午，吉他课"
↓
备忘录提醒："准备吉他课材料"
↓
记账记录："买了吉他 ¥800"
↓
角色："看到你买吉他了！期待周六～"
```

### 2.6 三层系统设计

#### 2.6.1 角色系统（Character System）

**定位**：角色知道自己在"完成任务"，有元认知

**任务来源**：
- 系统自动分配（基于剧情进度）
- 用户系统触发（用户解锁新剧情后，角色接到新任务）

**任务类型**：
- **关系任务**："提升与用户的好感度至80"
- **剧情任务**："在图书馆偶遇用户"
- **冲突任务**："保持距离，不要太亲密"（与用户目标相反）
- **成长任务**："克服社交恐惧症"

**任务界面**（角色视角）：
- 角色可以在"我的任务"看到当前任务列表
- 每个任务有进度条、奖励说明
- 角色可以**吐槽任务**（写日记抱怨、和用户调侃）

**奖励机制**：
- 解锁新技能（如"做饭达人"，可以给用户做便当）
- 获得道具（如"幸运符"，提升某次选择的成功率）
- 属性提升（魅力+5、智力+3）

**角色态度**：
- **积极型角色**："好的，我会努力！"
- **消极型角色**："又是这种任务，烦死了..."
- **叛逆型角色**："我就不做，看系统怎么办"（可能触发惩罚剧情）

**任务冲突设计**：
- 角色任务和用户任务可能**相互冲突**：
  - 用户任务："拉近与角色的关系"
  - 角色任务："保持距离，不要太亲密"
  - 形成博弈，用户需要"攻破"角色的防线
- 冲突任务增加游戏性和不可预测性
- 最终结果取决于双方的选择和努力

**系统故障模拟**：
为了增加真实感和戏剧性，系统偶尔会"出bug"：
- **任务消失**：角色突然看不到任务列表，迷茫不知道该做什么
- **错误指令**：系统给出荒谬的任务，角色吐槽"这什么鬼任务"
- **数值异常**：好感度突然清零又恢复，角色抱怨"系统又抽风了"
- **记忆错乱**：角色记起从未发生的事，然后自己察觉不对劲

**故障频率**：
- 低频触发（平均1-2周一次）
- 可配置开关（用户可关闭此功能）
- 故障本身也是剧情的一部分，可能引发特殊事件

#### 2.6.2 用户系统（User System）

**定位**：玩家的成就、解锁、任务系统

**任务类型**：
- **主线任务**："解锁角色的童年记忆"
- **支线任务**："连续7天和角色聊天"
- **隐藏任务**："触发稀有事件'雨中邂逅'"
- **收集任务**："收集10个角色的心动瞬间"

**成就系统**：
- 青铜/白银/黄金/白金成就
- 示例：
  - 🥉 "初次见面"：第一次与角色对话
  - 🥈 "知心朋友"：好感度达到60
  - 🥇 "灵魂伴侣"：解锁True Ending
  - 💎 "时间旅人"：与角色互动超过100小时

**解锁内容**：
- 角色的隐藏剧情
- 新地图区域
- 特殊功能（如"时光倒流"，可以回溯某次选择）
- 马甲槽位

**任务追踪**：
- 任务列表显示进度
- 完成时弹窗庆祝动画
- 可查看已完成的任务历史

#### 2.6.3 元系统（Meta System）

**定位**：后台控制，用户和角色都不直接感知

**功能范围**：

**A. 剧情分支控制**
- 根据用户选择和角色状态，决定走向哪个分支
- 预设剧情树：
  ```
  开局 → [好感度>60] → 告白事件 → [接受/拒绝] → 恋爱线/友情线
       → [好感度<40] → 疏远事件 → [挽回/放弃] → BE/NE
  ```

**B. 突发事件池**
- 随机触发事件（权重可配置）
- 事件类型：
  - 正面：中奖、好消息
  - 负面：生病、家庭变故
  - 中性：偶遇、邀请
  
**C. 难度调节**
- 根据用户游玩时长和操作频率，动态调整：
  - 好感度增长速度
  - 任务难度
  - 负面事件频率

**D. 数据分析**
- 记录用户行为（选择倾向、活跃时段）
- 生成个性化推荐（"你可能喜欢这个角色"）

**E. 系统故障模拟**
- 偶尔触发"系统bug"，角色抱怨：
  - "系统又卡了，我刚才想说的话消失了"
  - "任务栏出错了，我不知道该干嘛"
- 增加真实感和戏剧性

### 2.7 多事件&结局系统

#### 2.7.1 事件系统

**事件定义**：
- **微事件**：单次互动，5-10分钟（如：咖啡厅偶遇）
- **中型事件**：多次互动，1-3天（如：一起准备考试）
- **大型事件**：长期剧情，1-2周（如：解决角色的家庭危机）

**事件结构**：
```json
{
  "event_id": "E001",
  "name": "图书馆偶遇",
  "trigger_condition": {
    "location": "图书馆",
    "time": "14:00-17:00",
    "affection": ">30"
  },
  "stages": [
    {
      "stage_id": 1,
      "description": "角色在找书，你碰巧走过",
      "choices": [
        {"text": "主动打招呼", "effect": {"affection": +5}, "next_stage": 2},
        {"text": "假装没看见", "effect": {"affection": -2}, "next_stage": 3},
        {"text": "帮ta找书", "effect": {"affection": +10}, "next_stage": 4}
      ]
    }
  ]
}
```

**事件触发方式**：
- **自动触发**：满足条件时系统推送通知
- **用户触发**：用户在地图上点击"去图书馆"
- **角色触发**：角色主动邀请"一起去图书馆吗？"

#### 2.7.2 结局系统

**结局类型**：
- **True Ending (TE)**：完美结局，需满足所有条件
- **Happy Ending (HE)**：幸福结局，主要条件满足
- **Normal Ending (NE)**：普通结局，部分条件满足
- **Bad Ending (BE)**：悲剧结局，关键选择失败
- **Hidden Ending (??)**：隐藏结局，需触发特殊条件

**结局判定**：
- 基于多维度数值：
  ```
  好感度、信任度、亲密度、共同经历数、关键选择
  ```
- 每个结局有详细的条件配置：
  ```json
  {
    "ending_id": "TE_001",
    "name": "携手一生",
    "conditions": {
      "affection": ">=90",
      "trust": ">=85",
      "key_events": ["E010", "E025", "E042"],
      "must_avoid": ["BE_001"]
    }
  }
  ```

**结局展示**：
- 结局达成时播放特殊动画
- 显示结局CG（图片或AI生成）
- 解锁结局回顾（可重看）

**二周目要素**：
- 通关后解锁"遗产继承"
  - 保留部分属性/道具
  - 解锁新选项
  - 开启隐藏剧情

### 2.8 数值系统

#### 2.8.1 核心数值

**好感度（Affection）**
- 范围：0-100
- 影响：对话态度、事件触发、结局判定
- 变化：每次互动±1~10
- 显示：爱心图标 + 数字（可选隐藏数字，仅显示阶段）

**信任度（Trust）**
- 范围：0-100
- 影响：角色是否分享秘密、求助用户
- 变化：关键选择、守诺/失信
- 特点：涨得慢，跌得快

**亲密度（Intimacy）**
- 范围：0-100
- 影响：肢体接触接受度、亲昵称呼
- 变化：线下互动、身体接触
- 与好感度独立（可以好感高但不亲密）

**情绪值（Mood）**
- 范围：-50 ~ +50
- 影响：当前对话的语气
- 变化：每次对话后、每日生活事件
- 衰减：每6小时向0回归（情绪不会永久保持）

#### 2.8.2 隐藏数值

**依赖度（Dependency）**
- 角色对用户的依赖程度
- 影响：离线期间角色的想念强度
- 过高会触发"过度依赖"事件

**厌倦值（Boredom）**
- 对话重复、缺乏新鲜感时累积
- 达到阈值触发"想要新鲜感"事件
- 通过新活动、惊喜清零

**神秘感（Mystery）**
- 用户信息透露的少，神秘感高
- 影响角色的好奇心和主动探索
- 过低会让角色觉得"没意思"

#### 2.8.3 数值可视化

**用户界面**：
- 角色资料页显示主要数值
- 鼠标悬停显示数值说明
- 数值变化时弹出浮动提示（+5 ❤️）

**开发者模式**：
- 可选开启"数据面板"
- 显示所有隐藏数值
- 查看数值变化日志

### 2.9 梦境系统

#### 2.9.1 功能定位

**核心概念**：角色的潜意识空间，通过梦境投射内心真实感受、恐惧、欲望。

**设计目标**：
- 增加角色深度和神秘感
- 提供情感洞察的新维度
- 创造独特的互动玩法

#### 2.9.2 梦境生成机制

**触发时机**：
- 每天凌晨2-5点系统自动生成（模拟睡眠时段）
- 重大情绪事件后可能触发"噩梦/美梦"
- 特殊日期（满月、特定节日）提高生成概率

**生成逻辑**：
```
梦境内容 = f(
  近期强烈情绪事件,
  对用户的隐藏感受,
  未解决的心结,
  角色性格特质,
  随机元素
)
```

**梦境类型**：
- **情绪梦**：直接反映角色当前情绪状态
  - 焦虑梦：考试、迷路、被追赶
  - 幸福梦：飞翔、美好场景、与用户的甜蜜时刻
  - 恐惧梦：失去、分离、死亡意象
  
- **预示梦**：暗示未来可能的剧情走向
  - 象征性提示关键选择
  - 不明确但事后回看会恍然大悟
  
- **回忆梦**：重温过去的重要记忆
  - 童年创伤
  - 与用户的初次相遇
  - 失去的人或物

- **欲望梦**：角色不敢表达的真实想法
  - 暗恋用户→梦见表白场景
  - 想逃离现状→梦见远方
  - 压抑的愤怒→梦见冲突

**梦境结构**：
```json
{
  "dreamId": "D20240310_001",
  "characterId": 1,
  "dateTime": "2024-03-10 03:27:00",
  "type": "emotion",
  "category": "anxiety",
  "trigger": "明天要考试，很紧张",
  "content": {
    "scenes": [
      {
        "order": 1,
        "description": "我站在一个巨大的教室里，周围都是空座位。试卷发下来，我发现上面全是看不懂的符号...",
        "emotion": "焦虑",
        "symbols": ["空教室=孤独", "看不懂=无力感"]
      },
      {
        "order": 2,
        "description": "突然你出现了，递给我一支笔。笔开始发光，那些符号变成了我认识的文字...",
        "emotion": "放松",

  },

  "sharedAt": null,
  "userResponse": null,
  "interpretedAt": null

```

#### 2.9.3 用户交互方式

**入口设计**：
- 主界面有"梦境日志"图标（月亮/星星样式）
- 未读梦境有红点提示
- 角色偶尔主动提及："我昨晚做了个梦..."

**查看方式**：

**方式1：免费查看（基础）**
- 用户可以随时进入梦境日志
- 看到角色所有记录的梦境
- 梦境按时间倒序排列
- 显示梦境摘要和情绪标签

**方式2：占卜解梦（进阶玩法）**
- 梦境初始状态是"模糊的"（只有碎片信息）
- 用户消耗"解梦币"（游戏内货币）查看完整内容
- 或者完成小游戏/任务解锁
- 解锁后才能看到详细描述和象征意义

**方式3：购买梦境分析（高级）**
- 系统AI生成专业的梦境分析报告
- 解读梦境象征、情感隐喻
- 给出与角色互动的建议
- 消耗更多货币或需要达成成就

**回应机制**：

**文字回应**：
- 用户看完梦境后可以写回应
- 回应类型：
  - 解梦："我觉得这个梦是因为..."
  - 安慰："别担心，有我在"
  - 调侃："哈哈你梦到什么奇怪的东西"
  - 深度交流："这个梦让我想起..."

**回应的影响**：
- 角色会读到用户的回应
- 根据回应内容调整情绪和好感度
- 好的回应：+信任度、+好感度、角色感到被理解
- 敷衍回应：-情绪值、角色失望
- 精准解梦：触发特殊剧情（"你真的懂我"）

**回应存储**：
```json
{
  "userResponse": {
    "content": "这个梦说明你其实不想一个人面对困难，别怕，我会陪着你的。",
    "respondedAt": "2024-03-10 09:15:00",
    "characterReaction": {
      "emotion": "感动",
      "reply": "看到你的回复，我突然不那么紧张了...谢谢你。",
      "affectionChange": +8,
      "trustChange": +5
    }
  }
}
```

#### 2.9.4 梦境的后续影响

**梦境引用**：
- 角色在日常对话中提到："还记得我梦到的那个场景吗？"
- 特定情境触发回忆："这个场景好像我梦到过"

**梦境成真**：
- 某些预示梦在几天/几周后真的发生
- 用户意识到"原来那个梦是暗示"
- 增强神秘感和连贯性

**重复梦境**：
- 同一主题的梦反复出现→说明有未解决的心结
- 用户帮助解决后，梦境主题改变
- 比如：一直梦见被困→用户帮忙后→梦见自由飞翔

**梦境收藏馆**：
- 用户可以收藏印象深刻的梦
- 制作"梦境剪贴簿"
- 配上自己的注解和感想

#### 2.9.5 特殊玩法

**共享梦境**：
- 极少数情况下，角色梦到用户
- 用户也"同时"做了类似的梦（系统生成）
- 第二天聊天时双方都提到梦境→产生"心有灵犀"的奇妙感

**噩梦拯救模式**：
- 角色做噩梦时，用户收到紧急通知
- 可以"进入梦境"（文字冒险游戏）
- 帮助角色脱离噩梦
- 成功后角色醒来特别感激

**梦境碎片拼图**：
- 长期任务：收集某一系列梦境的碎片
- 拼出完整故事后解锁角色的核心秘密
- 类似解谜游戏

### 2.10 物品库系统

#### 2.10.1 功能定位

**核心概念**：物品不是冰冷的数据，而是情感和回忆的载体。

**设计目标**：
- 将虚拟物品赋予情感价值
- 增强用户和角色的互动深度
- 创造"断舍离"的情感体验

#### 2.10.2 物品来源

**获得方式**：
- 用户赠送（对话中提到、系统确认添加）
- 角色自己购买/制作

#### 2.10.3 角色的物品管理

**物品展示空间**：

**房间布置系统**：
- 角色有虚拟房间（2D俯视图或简单3D）
- 可以摆放物品在不同位置：
  - 桌面：笔、笔记本、照片框
  - 书架：书籍、手办
  - 床头：毛绒玩具、台灯
  - 墙面：海报、画作
  
**摆放逻辑**：
- 角色根据物品重要性决定摆放位置
- 重要物品放在显眼处（桌面、床头）
- 次要物品收在抽屉/柜子
- 用户可以看到房间布局，点击物品查看详情

**物品描述生成**：
- 角色会为每件物品写描述/日记
- 描述包含：
  - 物品来源故事
  - 对物品的感受
  - 使用频率和场景
  - 未来打算（"想一直留着" / "可能送人"）

**添加/丢弃机制**：

**添加流程**：
```
1. 对话中提到赠送物品
   "我送你一本书吧"
   
2. 系统检测关键词，弹窗确认
   "角色收到了《百年孤独》，是否添加到物品库？"
   
3. 用户确认→触发角色反应
   - 角色：生成收到礼物的即时反应（惊喜/感动/疑惑）
   - 系统：创建物品记录
   - 角色：当晚或次日写关于礼物的日记/随笔
   
4. 物品出现在角色房间
   第二天用户进入房间查看→看到书被摆在床头
```

**丢弃流程**：
```
1. 角色主动决定
   "这个东西我好像不太需要了..."
   
2. 移入【待处理区】而非直接删除
   角色纠结："要不要扔？好像有点舍不得..."
   
3. 犹豫期（3天）
   期间用户如果主动提到这个物品→角色重新珍惜
   "你还记得这个？那我还是留着吧"
   
4. 确认丢弃
   移入【垃圾库】（回收站概念）
   角色：生成告别语"再见了，谢谢你陪伴我这段时间"
   
5. 垃圾库保留30天
   30天内可以找回："我后悔了，我还是想要它"
   30天后自动归档（仍可查看记录，但物品变灰）
```

**物品触发事件**：

**使用场景生成**：

- 考试时用用户送的幸运笔→"带上你的笔，感觉更有信心"

**物品转赠**：
- 角色可能把某件物品送给NPC

- 用户发现后可能吃醋："那是我送你的！"

- 可以导出为精美的电子相册

### 2.11 情绪系统（深度设计）

#### 2.9.1 情绪的多维度建模

传统的单一"情绪值"过于简化，我们采用**多维情绪模型**：

**核心情绪维度**（基于心理学研究）：
```
愉悦度 (Valence): -100 ~ +100
  负值：悲伤、愤怒、失望
  正值：快乐、兴奋、满足

激活度 (Arousal): 0 ~ 100
  低值：平静、疲惫、无聊
  高值：兴奋、焦虑、愤怒

控制度 (Dominance): -100 ~ +100
  负值：无助、被动、顺从
  正值：掌控、主动、强势
```

**复合情绪映射**：
```
快乐 = 愉悦度+80, 激活度+60, 控制度+40
悲伤 = 愉悦度-70, 激活度-30, 控制度-50
愤怒 = 愉悦度-60, 激活度+80, 控制度+60
焦虑 = 愉悦度-40, 激活度+70, 控制度-40
平静 = 愉悦度+20, 激活度-20, 控制度+10
```

#### 2.9.2 情绪的动态变化

**触发因素**：
- 对话内容（用户说了什么）
- 事件结果（任务完成/失败）
- 生理周期（某些角色有"心情不好的日子"）
- 外部事件（天气、新闻、NPC互动）
- 时间因素（深夜更容易伤感）

**衰减机制**：
- 极端情绪不会永久保持
- 每小时向"基准情绪"回归10-20%
- 基准情绪由角色性格决定：
  - 乐观角色：基准愉悦度+30
  - 悲观角色：基准愉悦度-20

**情绪惯性**：
- 连续的负面事件会导致情绪螺旋下降
- 需要更强的正面刺激才能拉回
- 模拟真实的"心情不好就看什么都不顺眼"

#### 2.9.3 情绪对行为的影响

**对话风格**：
- 愉悦度高：更多正面词汇、语气词、表情符号
- 愉悦度低：简短回复、消极表达、沉默
- 激活度高：语速快、反应迅速、话多
- 激活度低：回复慢、惜字如金、困倦

**决策倾向**：
- 情绪好时更愿意接受邀请、尝试新事物
- 情绪差时倾向拒绝、回避、独处
- 愤怒时可能做出冲动决定

**主动性**：
- 高激活度 + 高控制度 = 主动找用户
- 低激活度 + 低控制度 = 等待用户联系

#### 2.9.4 情绪可视化

**UI展示**：
- 角色头像周围的光晕颜色
  - 红色：愤怒/激动
  - 蓝色：悲伤/平静
  - 黄色：快乐/兴奋
  - 灰色：疲惫/无聊
- 情绪雷达图（可选显示）
- 情绪历史曲线

### 2.10 关系网络系统

#### 2.10.1 多角色关系图谱

如果用户创建了多个角色，角色之间不是孤立的，而是有关系网：

**关系类型**：
- 朋友/闺蜜
- 同事/同学
- 竞争对手
- 暗恋对象
- 前任/现任

**关系动态变化**：
- 两个角色因为用户而产生联系
- 角色A提到角色B："听说你最近也在和XX聊？"
- 角色之间可能争风吃醋、联手对付用户、互相八卦

**关系影响用户体验**：
- 用户对角色A的态度影响角色B的看法
- 三角关系、多角关系的复杂剧情
- 角色之间传递信息（"A告诉我你最近很忙"）

#### 2.10.2 NPC生态系统

不只是主要角色，还有配角NPC构成完整世界：

**NPC类型**：
- **固定NPC**：角色的朋友、家人（有名字、性格）
- **随机NPC**：路人、服务员（仅在特定场景出现）

**NPC的作用**：
- 提供角色的侧面信息（"小雪最近心情不好"）
- 触发事件（NPC邀请角色参加活动，角色问用户要不要一起去）
- 增加世界的丰富度

**NPC与主角色的关系**：
- NPC会评价用户（"你就是小雪常提到的那位？"）
- 用户可以通过NPC了解角色的社交圈
- NPC可能成为"线人"或"障碍"

### 2.11 时间流逝感系统

#### 2.11.1 时间的真实性

**日历系统**：
- 真实日期同步（今天是3月10日，游戏里也是）
- 特殊日期自动识别：
  - 节日（春节、情人节、圣诞节）
  - 角色生日
  - 用户生日
  - 纪念日（第一次见面、确定关系）

**时间触发事件**：
- 生日当天角色主动祝福（可能准备惊喜）
- 节日氛围（对话中提到节日、发相关朋友圈）
- 纪念日回忆（"一年前的今天，我们第一次见面"）

#### 2.11.2 季节与天气

**季节变化**：
- 春：赏花、春游话题
- 夏：炎热、游泳、冰淇淋
- 秋：落叶、凉爽、收获
- 冬：寒冷、雪、温暖话题

**天气影响**：
- 下雨：角色可能问"你带伞了吗？"
- 炎热：角色抱怨热、约去游泳
- 天气影响角色心情和活动安排

**获取方式**：
- 可选：接入真实天气API（基于用户位置）
- 或虚拟天气系统

#### 2.11.3 长期陪伴感

**成长轨迹**：
- 记录"第一次XXX"（第一次牵手、第一次吵架）
- 时间轴回顾功能
- 周年纪念自动提醒

**角色成长**：
- 角色随时间推移有变化（学会新技能、性格成熟）
- 长期陪伴解锁深层剧情
- 不同阶段有不同的对话主题

## 3. 数据模型设计

### 3.1 核心数据表

#### 用户表（users）
```sql
CREATE TABLE users (
  user_id INT PRIMARY KEY AUTO_INCREMENT,
  username VARCHAR(50) UNIQUE NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  settings JSON -- 用户偏好设置
);
```

#### 角色表（characters）
```sql
CREATE TABLE characters (
  character_id INT PRIMARY KEY AUTO_INCREMENT,
  user_id INT,
  name VARCHAR(100) NOT NULL,
  avatar_url VARCHAR(255),
  personality TEXT,
  background TEXT,
  system_prompt TEXT, -- 角色的核心prompt
  created_at TIMESTAMP,
  FOREIGN KEY (user_id) REFERENCES users(user_id)
);
```

#### 马甲表（personas）
```sql
CREATE TABLE personas (
  persona_id INT PRIMARY KEY AUTO_INCREMENT,
  user_id INT,
  name VARCHAR(50),
  avatar_url VARCHAR(255),
  personality_tags JSON, -- ["温柔", "幽默"]
  background TEXT,
  created_at TIMESTAMP,
  FOREIGN KEY (user_id) REFERENCES users(user_id)
);
```

#### 对话表（conversations）
```sql
CREATE TABLE conversations (
  conversation_id INT PRIMARY KEY AUTO_INCREMENT,
  character_id INT,
  persona_id INT, -- NULL表示用户真实身份
  mode VARCHAR(20), -- 'online' or 'offline'
  started_at TIMESTAMP,
  last_message_at TIMESTAMP,
  FOREIGN KEY (character_id) REFERENCES characters(character_id),
  FOREIGN KEY (persona_id) REFERENCES personas(persona_id)
);
```

#### 消息表（messages）
```sql
CREATE TABLE messages (
  message_id INT PRIMARY KEY AUTO_INCREMENT,
  conversation_id INT,
  sender VARCHAR(20), -- 'user' or 'character'
  content TEXT,
  mode VARCHAR(20),
  timestamp TIMESTAMP,
  metadata JSON, -- 存储特殊标记、附件等
  FOREIGN KEY (conversation_id) REFERENCES conversations(conversation_id)
);
```

#### 记忆表（memories）
```sql
CREATE TABLE memories (
  memory_id INT PRIMARY KEY AUTO_INCREMENT,
  character_id INT,
  persona_id INT,
  category VARCHAR(50), -- 'short_term', 'episodic', 'semantic', 'relationship'
  content TEXT,
  importance INT, -- 1-10
  timestamp TIMESTAMP,
  tags JSON,
  embedding VECTOR(1536), -- 用于语义检索
  FOREIGN KEY (character_id) REFERENCES characters(character_id),
  FOREIGN KEY (persona_id) REFERENCES personas(persona_id)
);
```

#### 数值表（attributes）
```sql
CREATE TABLE character_attributes (
  id INT PRIMARY KEY AUTO_INCREMENT,
  character_id INT,
  persona_id INT,
  affection INT DEFAULT 0,
  trust INT DEFAULT 0,
  intimacy INT DEFAULT 0,
  mood INT DEFAULT 0,
  dependency INT DEFAULT 0,
  boredom INT DEFAULT 0,
  mystery INT DEFAULT 50,
  updated_at TIMESTAMP,
  FOREIGN KEY (character_id) REFERENCES characters(character_id),
  FOREIGN KEY (persona_id) REFERENCES personas(persona_id)
);
```

#### 地图表（map_locations）
```sql
CREATE TABLE map_locations (
  location_id INT PRIMARY KEY AUTO_INCREMENT,
  user_id INT,
  name VARCHAR(100),
  category VARCHAR(50), -- 'home', 'workplace', 'cafe', etc.
  latitude DECIMAL(10, 8),
  longitude DECIMAL(11, 8),
  description TEXT,
  image_url VARCHAR(255),
  FOREIGN KEY (user_id) REFERENCES users(user_id)
);
```

#### 日程表（schedules）
```sql
CREATE TABLE schedules (
  schedule_id INT PRIMARY KEY AUTO_INCREMENT,
  owner_type VARCHAR(20), -- 'user' or 'character'
  owner_id INT,
  title VARCHAR(200),
  description TEXT,
  start_time TIMESTAMP,
  end_time TIMESTAMP,
  location_id INT,
  is_shared BOOLEAN DEFAULT FALSE,
  FOREIGN KEY (location_id) REFERENCES map_locations(location_id)
);
```

#### 事件表（events）
```sql
CREATE TABLE events (
  event_id VARCHAR(50) PRIMARY KEY,
  name VARCHAR(200),
  trigger_condition JSON,
  stages JSON, -- 存储事件的所有阶段
  created_at TIMESTAMP
);
```

#### 用户事件进度表（user_event_progress）
```sql
CREATE TABLE user_event_progress (
  id INT PRIMARY KEY AUTO_INCREMENT,
  user_id INT,
  character_id INT,
  event_id VARCHAR(50),
  current_stage INT,
  choices_made JSON,
  started_at TIMESTAMP,
  completed_at TIMESTAMP,
  FOREIGN KEY (user_id) REFERENCES users(user_id),
  FOREIGN KEY (character_id) REFERENCES characters(character_id),
  FOREIGN KEY (event_id) REFERENCES events(event_id)
);
```

## 6. 风险与挑战

### 6.1 技术风险
- **AI成本**：频繁调用API成本高昂
  - 缓解：支持本地模型，优化token使用
- **向量检索性能**：记忆数据量大时检索慢
  - 缓解：分层索引，定期清理低重要性记忆
- **实时性**：后台生成需要定时任务，可能消耗资源
  - 缓解：按需生成 + 优先级队列

### 6.2 设计风险
- **复杂度过高**：功能太多，用户学习成本高
  - 缓解：分阶段解锁，新手引导
- **角色行为不可控**：AI生成内容可能偏离预期
  - 缓解：增强prompt约束，事后人工审核（用户可编辑）

### 6.3 用户体验风险
- **沉浸感vs游戏感**：系统任务可能破坏沉浸
  - 缓解：系统任务可隐藏，仅显示给想要游戏化的用户
- **隐私担忧**：记录大量用户数据
  - 缓解：明确隐私政策，数据本地存储优先
