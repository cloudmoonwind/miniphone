## [变量系统] 快照合并策略：时间戳排序 + 同世界假设

**决策日期**: 2026-04-26  
**涉及功能**: 变量系统快照、聊天记录变量状态、角色时间线变量状态

**做法**:  
角色时间线事件和聊天消息各自在 `variable_snapshot` 字段存储完整变量快照。  
"当前变量状态"由以下规则决定：
- 取当前活跃聊天记录的最后一条消息的快照
- 时间线事件的快照与聊天快照通过 **消息/事件的生成时间戳** 确定先后顺序
- 假设角色生活时间线与用户聊天时间线**处于同一个世界、时间同步**

**为什么这样做**:  
- 避免设计复杂的跨来源合并逻辑
- 同世界假设在当前功能范围内成立（不支持角色独立推进时间的玩法）
- 多聊天记录切换时，变量状态跟着聊天记录走，不共享，符合直觉

**已知局限 / 未来待改**:  
- 如果日后支持"角色时间线独立推进"（角色时间和用户时间不同步），时间戳排序会失效
- 如果支持聊天记录分支/回档，快照的继承关系需要重新设计（目前直接读最后一条）
- 不同来源的快照冲突时（同时间戳）目前无处理逻辑，取最新写入的

**改动时注意**:  
`server/services/variables.ts` 中读取当前快照的函数，以及 `messages` 表和 `timeline_events` 表的 `variable_snapshot` 字段。

## [变量系统] AI 只输出变化量，情绪整体替换

**决策日期**: 2026-04-26  
**涉及功能**: AI 回复解析、变量更新机制

**做法**:  
- AI 在每轮回复末尾用 `<var>...</var>` 标签包裹变量更新内容
- **只写发生变化的变量**，未变化的变量直接继承上一快照的值
- 数值变量格式：`变量名：原值→新值`（系统校验原值匹配，不匹配则忽略）
- 情绪状态格式：`情绪：愤怒 40% | 委屈 35% | 不甘 25%`（整体替换，百分比必须和为100）
- 格式标签（`<var>` 等）可在元系统页面配置

**为什么这样做**:  
- 原值校验防止重新生成消息时旧 delta 累积错误
- 情绪是主观整体描述，不适合 delta 格式
- 只输出变化量减少 AI 输出长度和格式出错概率

**已知局限 / 未来待改**:  
- AI 格式遵守率依赖提示词质量，格式错误时静默忽略不报错（用户无感知）
- 原值校验在快速连续消息时可能误判（并发写入场景）

## [事件系统] 任何修改 `character_values` 的路径都必须触发 `value_change`

**决策日期**: 2026-05-01
**涉及功能**: 数值更新、事件订阅触发

**做法**:
凡是写入 `character_values.current_value` 的代码路径，紧随其后必须为每个真正变化的变量调用：

```ts
checkAndFireEvents(charId, {
  trigger: 'value_change',
  changedVariable: <name>,
  newValue: <value>,
});
```

当前已接入的入口：
- `POST /api/values/item/:id/adjust` —— 手动调整接口（[server/routes/values.ts](../server/routes/values.ts) 内）
- `POST /api/chat/respond` 流式与非流式分支 —— AI `<var>` 块自然驱动（[server/routes/chat.ts](../server/routes/chat.ts) 内）

`parseAndApplyVarBlock()`（[server/services/values.ts](../server/services/values.ts)）返回 `changedVariables` 数组供调用方迭代触发。

**为什么这样做**:
- 事件系统的 `condition_subscriptions` 索引按变量名订阅；订阅条件能不能命中**完全取决于 fire 是否被调用**
- 历史上 AI `<var>` 这条最常用的路径漏接了，导致"好感度首次≥80时触发表白"这类基础事件在自然对话中根本不触发，只能靠手动拖数值——属于事件系统核心入口被绕过
- 文档化为契约后，新增入口（批量导入、定时器、迁移脚本等）时能被搜到这条要求

**已知局限 / 未来待改**:
- 同一轮可能触发多次（多个变量同时变化），目前每次独立 fire；若要做"批触发"需要给 TriggerContext 加 `changedVariables[]` 字段
- 没有自动化测试卡住这一点；一旦回归只能靠手动复盘事件链路

**改动时注意**:
新增任何写 `character_values` 的代码路径，必须按上面的模板补 fire；否则事件系统会沉默失效。

## [事件系统] `record_history` 效果写入 `timeline`，记忆系统接管时再决定是否搬迁

**决策日期**: 2026-05-01
**涉及功能**: 事件 effects、时间线、（未来）记忆系统

**做法**:
事件 effect 的 `record_history` / `记录历史` 类型，写入 `timelineStore`，字段约定：
- `source: 'event_engine'` —— 标记来源，便于后续过滤/迁移
- `linkedEventId: <事件id>` —— 反向关联事件
- `type: 'event'`、`title` 取 content 前 30 字、`timestamp` 取当前时间

写入是 fire-and-forget（`executeEffect` 是同步签名，事件触发链路不能因记录历史失败而阻塞）。

**为什么这样做**:
- 之前是 `console.log` 占位，作者写的 `record_history` 在用户侧完全不可见
- 选 `timeline` 而不是 `memories`：timeline 已有 `linkedEventId` 字段、是用户层"历史记录"语义的天然落点；memory 当时还是 blob、且记忆系统正要重做
- `source='event_engine'` 是个干净的过滤入口；记忆系统上线后如果决定改写到记忆库，可一句 SQL 反查/迁移

**已知局限 / 未来待改**:
- fire-and-forget 失败只能在服务器日志看到，前端无感
- 记忆系统正式接入后需要决策：保留这条 timeline 入口、还是改写到记忆库（届时再视设计决定）

**改动时注意**:
若日后扩展 effect 类型涉及"用户可见的历史/记录"，先确认是否应统一走 timeline，避免出现多路写历史。
当 `executeEffect` 切换为 async（如果未来必须 await 某些 effect），记得把这里的 fire-and-forget 改为 await。

## [变量系统] 变量定义不存快照，快照只存当前值

**决策日期**: 2026-04-26

**做法**:  
`variable_snapshot` JSON 只存 `{ variableName: currentValue }` 的键值对，  
变量的元信息（名称、范围、分段、规则等）仍在 `character_variables` / `value_stages` / `value_rules` 表里。

**为什么这样做**:  
- 快照 JSON 保持轻量，避免每条消息冗余存储全量定义
- 如果变量定义被修改，历史快照的值依然有效（只是分段描述会用新定义重新解释）

**已知局限**:  
- 删除一个变量后，历史快照里还有它的值，读取时需要容错处理（忽略未知键）

## [变量管道] 占位符通用解析器：注册式命名空间 + 强制三段式

**决策日期**: 2026-05-02
**涉及功能**: `server/services/placeholders.ts`、`builtinNamespaces.ts`、`context.ts` 槽位组装

**做法**:
- 引入通用解析器 `resolvePlaceholders(template, ctx)` 与 `resolveAndExplain(template, ctx)`，所有"作者可写文本"槽位经此入口
- 子系统通过 `registerNamespace(name, resolver, list)` 在启动钩子注册自己（[server/index.ts](../server/index.ts)）
- 占位符语法强制 `{{namespace:identifier}}` 或 `{{namespace:identifier:modifier}}` 三段式；裸名 `{{user}}` `{{char}}` 不解析，原样保留
- 命名空间表已冻结，见 [ICS_近期开发计划_v1.md](archive/ICS_近期开发计划_v1.md)（已归档）第二节

**为什么这样做**:
- 旧 `resolveValuePlaceholders` 只识别 `v:` 一种命名空间、且只在 4 处槽位被调用，作者在 life/dreams/memories 等槽位写的占位符全部静默失效
- 子系统注册式使新增命名空间不动核心代码，与共识文档"子系统层向基础设施层注册自己"的方向一致
- 三段式语法消除歧义（`{{user}}` 究竟是用户名还是裸名占位？强制 `{{user:name}}` 后无悬念）

**已知局限 / 未来待改**:
- 阶段 1 尚未实现占位符转义语法。若作者文本里需要写出字面量 `{{val:xxx}}`（如做教程示例），无法绕过解析。等真有需求再加（建议 `\{\{...\}\}`）
- 解析器是同步纯函数，依赖各 resolver 也同步——这一约束限制了部分实现方式（见下条）

**改动时注意**:
- 加新命名空间时，必须在启动钩子调用 `registerNamespace`
- 修改命名空间命名规则前，先检查所有数据库的作者可写文本字段（worldbook content / persona / character core 等）

## [变量管道] resolver 必须同步：底层用 better-sqlite3 而非 store async 接口

**决策日期**: 2026-05-02
**涉及功能**: `server/services/builtinNamespaces.ts` 中 char/user/world resolver 的实现

**做法**:
char/user/world 三个 resolver 内部直接 `db.prepare(...).get(...)` 同步查表，**不**走 `characterStore.getById()` / `personaStore.getById()` 这种 async 接口（虽然存在）。
返回约定：
- 识别 identifier → 返回字符串
- 不识别 identifier → 返回 `null` / `undefined`，由解析器统一标记 `[未知字段:...]`
- 抛异常 → 解析器捕获并标记 `[解析错误:...]`，不让单个变量爆掉整段文本

**为什么这样做**:
- `resolvePlaceholders` 是 `String.prototype.replace` 的回调驱动，必须同步——一旦改 async 会导致所有调用方（context.ts 11 个槽位）跟着改 async，跨整个组装链路传染
- better-sqlite3 本身是同步 API，绕开 async store wrapper 不丢功能、性能更好
- null/异常的双轨错误信号让 resolver 实现简单：识别就返回字符串、不识别返回 null、运行时挂掉就让它挂（解析器兜底）

**已知局限 / 未来待改**:
- 如果未来某子系统的"取值"必须 async（如调外部 API），需要重新设计——可能要引入"预求值阶段"在 assembleMessages 顶部异步预取所有需要的 ctx 数据，再同步走 replace
- world: 命名空间未来接外部天气/时间 API 时会撞这个限制，到时候要么改架构、要么本地缓存 + 后台异步刷新

**改动时注意**:
- 别把 resolver 改成 async 来"修复"看起来的 bug，会破坏整个解析链路
- 加新 resolver 时优先选同步查询；如果只有 async 数据源，本地缓存而不是改 resolver 签名

## [变量管道] sys-variables 槽与新用户消息不过解析器

**决策日期**: 2026-05-02
**涉及功能**: [server/services/context.ts](../server/services/context.ts) 统一出口

**做法**:
统一出口对所有作者可写槽位调 `resolvePlaceholders`，但**两类内容跳过**：
1. `sys-variables` 槽：`if (blockType === 'variables')` 直接用 raw content
2. 新用户消息（`newUserContent`）：直接 push，不解析

**为什么这样做**:
- sys-variables 槽是**系统生成的指令文本**（"【变量更新】每轮回复末尾必须附加..."），里面如果未来有人写了 `{{val:affection}}` 作示例，解析器会把"示例"替换成实际值，AI 看到的就不是格式说明而是结果，破坏指令本意
- 用户输入是**事实记录**，不是模板。用户在聊天里输入 `{{val:affection}}` 应该原样保留（用户可能在讨论占位符语法本身）

**已知局限 / 未来待改**:
- "系统生成 vs 作者可写"这个二分目前是硬编码在 `blockType === 'variables'` 这一条 if，未来如果再加新的"系统生成槽"（如未来的 `sys-mode` 模式提示），需要扩展跳过名单。建议改成显式标记 `slotDef.skipPlaceholders = true`

**改动时注意**:
- 新加槽位时考虑是"作者可写"还是"系统生成"。前者必须过解析器，后者必须跳过

## [变量管道] 故意未实现的位置：world:time / wb / util

**决策日期**: 2026-05-02
**涉及功能**: [server/services/builtinNamespaces.ts](../server/services/builtinNamespaces.ts)

**做法**:
- `world:time` 故意不在 `world:` 命名空间下暴露
- `wb:` 与 `util:` 命名空间已注册但 resolver 永远返回 null（任何 identifier 都报"未知字段"）

**为什么这样做**:
- **`world:time`**: 当前没接外部 API，它和 `time:now` 求值结果完全相同——同时暴露会让作者困惑"这两个有什么区别"。等接入外部城市天气/时间 API 后，`time:now` 仍是设备时间、`world:time` 才有独立含义（角色所在城市当地时间），那时再暴露
- **`wb:`**: 世界书在 prompt 里通过 wb-pre / wb-post 槽位直接注入，不需要占位符。但保留命名空间是为了未来出现 `{{wb:active_count}}` 之类的元数据需求
- **`util:`**: `{{util:random:a,b,c}}` `{{util:roll:1d6}}` 等工具函数阶段 1 没有用例，只占位

**已知局限 / 未来待改**:
- 看到 `wb:` `util:` 注册但永远报错容易被误以为是 bug，注释里已说明"占位"。未来谁补全 identifier 时直接加 resolver case 即可
- `world:time` 被作者意外用到时只会得到 `[未知字段:world:time]`，需要在文档/能力清单页里说明"用 time:now 代替"

**改动时注意**:
- 实现 `world:time` 前先确认外部 API 已接入或者已确定数据源，否则恢复"作者困惑"问题
- 给 `wb:` `util:` 加 identifier 时，记得同时在 `list()` 函数里返回它们，否则能力清单页看不到

## [变量管道] v:→val: 一刀切迁移、self-test 替代测试框架

**决策日期**: 2026-05-02
**涉及功能**: 全项目占位符语法、`server/scripts/test-*.ts` / `smoke-*.ts`

**做法**:
- 阶段 1 把唯一存在的旧命名空间 `v:` 一刀切改名为 `val:`，**不留兼容期、不写迁移脚本**

- 测试覆盖率没有自动化报告，全靠人肉确认 case 完备
- 跨文件集成测试如果开始变多，可能需要引入测试框架。届时 self-test 脚本可以保留作为快速诊断工具

**改动时注意**:
- 未来再加新命名空间名变更时，先 grep 数据库实际数据再决定是否需要迁移脚本
- 添加 self-test case 时遵循现有格式（[server/scripts/test-placeholders.ts](../server/scripts/test-placeholders.ts) 是模板）

## [事件系统] branch 连接的 required outcome 改用独立字段（前端 UI 待补）

**决策日期**: 2026-05-02
**涉及功能**: `event_connections` 表、branch 类型分支连接的判定、AI 协议 outcome 字符集

**做法**:
旧版 [eventEngine.ts](../server/services/eventEngine.ts) 用 `relation_type` 字段后缀拼接表示分支需要的 outcome：

```
relation_type = 'branch:success'   →  上一个事件 outcome=success 时走该分支
relation_type = 'branch:fail'      →  上一个事件 outcome=fail 时走该分支
relation_type = 'branch'           →  任意 outcome 都触发
```

阶段 2 重构 AI 输出协议时改为：

```
relation_type      = 'branch'
required_outcome   = 'success' | 'fail' | ... | null
```

`required_outcome=null` 表示"任意 outcome 都触发"，与旧版纯 `'branch'` 等价。

**改动清单**:
- [server/db/schema.ts](../server/db/schema.ts) `eventConnections` 加 `requiredOutcome: text('required_outcome')` 字段
- [server/db/migrations/0004_event_required_outcome.sql](../server/db/migrations/0004_event_required_outcome.sql) ALTER TABLE + 防御性补丁（把可能存在的 `'branch:xxx'` 拆开）
- [server/services/events.ts](../server/services/events.ts) `createConnection` / `updateConnection` 接 `requiredOutcome` 参数
- [server/services/eventEngine.ts](../server/services/eventEngine.ts) 删除 `relationType.split(':')` hack，改用 `conn.requiredOutcome` 直接判断
- [server/routes/events.ts](../server/routes/events.ts) `POST /connections` 透传 `requiredOutcome`

**为什么这样做**:
- 旧的拼接 hack 让 outcome 字符集被迫受限（不能含 `:`），AI 协议层一直要带"outcome 不能含冒号"的怪异约束
- 修干净后 outcome 可为任意字符串（中文、含符号都行），协议设计无怪味
- 改动小、回归风险低（开工前 grep 数据库实际无 `'branch:xxx'` 数据，前端编辑器也只能选纯 `'branch'`，无破坏面）

**已知局限 / 未来待改**:
- ⚠️ **前端 [EventEditor.tsx](../client/src/apps/meta/EventEditor.tsx) 编辑器还没暴露 `requiredOutcome` 字段的编辑 UI**。目前作者选了 `branch` 类型连接后，无法在 UI 上指定"该分支需要哪个 outcome"，新创建的 branch 连接 `requiredOutcome` 都是 null（等于"任意 outcome 都触发"）。**梳理事件系统时务必补这块 UI**——可能在连接编辑面板里加一个文本输入框（仅当 `relationType='branch'` 时显示）
- 服务端 GET 路由返回的连接对象现在带了 `requiredOutcome` 字段（drizzle inferSelect 自动），前端可读取但目前没用到
- 如果未来 outcome 字符集再扩展（如允许换行/特殊字符），需要确认前端展示和 AI 协议解析两端都正确处理

**改动时注意**:
- 在前端补 UI 时记得 `POST /api/events/connections` 和 `PUT /api/events/connections/:id` 都已支持透传 `requiredOutcome`，不需要后端再加路由
- 测试时构造 `relation_type='branch'` + `required_outcome='success'` 的连接，触发上一个事件 outcome=success，确认下一个事件被解锁
- 旧 `'branch:xxx'` 形式的代码搜索关键词：`branch:`、`relationType.split(':')`、`'branch:`——已全部清除，再出现就是回退

## [移动端路线] Web 优先，APK 优先走 Capacitor/WebView，不默认原生重写

**决策日期**: 2026-05-02
**涉及功能**: 移动端部署、APK 打包、前后端分离、未来本地数据库迁移

**做法**:
当前阶段继续以 Web 项目为主：

```
React/Vite 前端
-> Express API
-> SQLite / AI provider
```

手机体验按阶段推进：
1. 近期开发/自测：电脑运行前后端，手机浏览器访问局域网地址
2. 低成本体验：PWA 或手机浏览器添加到桌面
3. 进入 APK 阶段：优先用 Capacitor/WebView 复用现有 React 前端
4. 有后端条件时：APK 连接局域网/云端 HTTP API
5. 明确不想依赖后端且核心玩法稳定后：再考虑 APK 本地数据库 + 云 AI

不把 Kotlin/Flutter/React Native 原生重写作为默认路线。只有当 WebView 体验明显卡住核心需求（例如复杂后台任务、深度系统权限、原生性能瓶颈）时再评估。

**为什么这样做**:
- 当前项目的价值核心是角色系统、变量状态、事件触发、AI 协议和交互体验，不是 APK 外壳本身
- React/Vite 已经能快速迭代，WebView/Capacitor 能最大化复用现有前端
- 直接原生开发会引入 Activity 生命周期、Gradle、权限、键盘适配、签名发布等额外成本，容易拖慢核心玩法验证
- 云服务器有持续成本；项目早期不默认上云，先用本机/局域网/浏览器路线降低成本
- 完全本地 APK 难点不在 UI，而在后端逻辑、数据库、迁移、备份、AI key 安全和多端同步

**APK 更新策略取舍**:
- **内置前端**：APK 包内带 `dist`。打开快、更像 App；前端改动需要重新打包发 APK
- **远程网页**：APK WebView 加载线上地址。前端更新不用重发 APK；依赖网络和服务器
- **混合方案**：内置基础壳，部分页面/资源远程更新。后期可评估，早期不增加复杂度

**维护性约束**:
为了以后 APK 阶段更新功能不痛，必须避免“两套逻辑”：
- UI 继续优先复用 React
- 前端请求集中在 `client/src/services`
- 业务规则集中在 `server/services`，未来确需双端复用时再抽到 `shared/core`
- 数据库访问集中在 repository/store
- AI 调用集中在 provider/facade
- 不在 UI 组件里写死 `/api`、`localhost`、供应商 API 或本地数据库细节

**不推荐路线**:
- 早期直接 Kotlin/Flutter/React Native 重写：成本高，且会形成 Web/App 两套 UI 和逻辑
- Android 内强行跑 Node/Express/better-sqlite3 当长期方案：维护和稳定性风险高
- Termux 作为正式用户路线：适合开发者折腾，不适合普通使用体验
- 项目尚未稳定前购买云服务器：成本先行，收益不明确

**未来触发重新评估的条件**:
- 核心玩法已经稳定，手机端成为主使用场景
- WebView 无法满足关键体验
- 确定需要离线可用、本地隐私或无服务器运行
- 确定需要多设备同步/账号体系/云端备份
- 有明确预算或部署条件支撑云服务器

## [数据库路线] 当前 SQLite，本地阶段保持；记忆系统成熟后可评估切 PostgreSQL

**决策日期**: 2026-05-02
**涉及功能**: 数据库选型、记忆系统、语义检索、云端部署、未来多用户

**做法**:
当前阶段继续使用：

```
SQLite + better-sqlite3 + Drizzle schema/migrations
```

不因为“以后可能需要 PostgreSQL”而提前迁移。只有当记忆系统进入需要更强检索/索引/并发/云端能力的阶段，再正式评估 PostgreSQL。

**为什么这样做**:
- 当前项目仍是本地开发/单用户 demo，SQLite 零部署、单文件、备份简单，最省心
- 记忆系统未来可能需要语义检索、向量索引、复杂筛选、时间/重要性排序、多用户隔离，这些会让 PostgreSQL 更合适
- 过早切 PG 会引入服务器、连接池、迁移、备份、账号权限、部署成本，拖慢当前核心玩法验证
- 现在已经有 `server/db/`、`server/storage/`、Drizzle schema 和 Store 边界；只要继续保持数据库访问集中，未来切 PG 的痛感可控

**触发重新评估 PostgreSQL 的条件**:
- 记忆系统需要向量检索、全文检索或复杂多条件排序
- 需要多用户、多设备同步、云端备份
- SQLite 文件库在并发写入、数据量、查询速度上开始成为瓶颈
- 需要更明确的权限隔离、审计、远程访问
- 项目进入云端/API 服务阶段，且有预算长期运行数据库

**迁移前必须做到**:
- 前端不直接依赖数据库形态，只通过 service
- 后端业务逻辑不散落 SQL，数据库访问集中在 repository/store
- migration 能完整描述当前 schema
- 先写导出/导入脚本：SQLite -> 中间 JSON/SQL -> PostgreSQL
- 先定义记忆系统的查询模型：按角色、时间、重要性、关键词、语义相似度如何组合

**改动时注意**:
- 不要为了未来 PG 把当前 SQLite 体验复杂化
- 不要在 route 或 UI 中写死 SQLite 特性

## [AI 协议] 解析层与应用层分开：parseAIOutput 不连库，apply* 才写库

**决策日期**: 2026-05-02
**涉及功能**: `server/services/aiProtocol.ts`、`server/services/values.ts` 的 `applyVarBlock`、`server/services/eventEngine.ts` 的 `applyEventOutcomes`、`server/routes/chat.ts` 流式与非流式两路

**做法**:
AI 输出协议（[ICS_AI输出协议_v1.md](../ICS_AI输出协议_v1.md)）的处理拆成严格分层：

```
fullContent (AI 原始输出)
  → parseAIOutput  纯解析    →  { cleanContent, varUpdates, emotion, events, diagnostics }
  → applyVarBlock 校验 + 写库 →  { snapshot, emotionState, changedVariables }
  → applyEventOutcomes 写库   →  applied[]
```

**parseAIOutput 不读、不写数据库**：只把字符串拆成结构化对象。失败降级（未识别子标签、单行格式错）只通过 `diagnostics` 字段返回，不抛错。

**应用层（apply*）才连库**：旧值校验 ±1、clamp、updateValue、updateEvent、unlockEvent。校验失败只跳过该项，不阻塞其他项。

旧版 `parseAndApplyVarBlock` / `parseOutcomeFromAIResponse` 把"字符串解析"和"写数据库"混在一起，已删除。

**为什么这样做**:
- **单元测试可离线跑**：parseAIOutput 是纯函数，[scripts/test-ai-protocol.ts](../server/scripts/test-ai-protocol.ts) 25 个 case 不需要 db、不需要 mock store，覆盖各种边界
- **AI 协议演进时只动一处**：将来加 `<state>` `<intent>` `<wb>` 等子标签，只需改 `parseAIOutput` 的 dispatcher + 加对应 `apply*` 函数；解析逻辑和应用逻辑互不污染
- **防止"AI 输出错乱导致数据库不一致"**：解析阶段就检测格式错（如 emotion 百分比之和超出容忍），数据库写入阶段假设输入已是合法结构；两层错误信号清晰分离
- **trace 链路清晰**：解析阶段记 `aiProtocol.*` 事件，应用阶段记 `variables.var.apply.*` / `eventEngine.event.outcome.*` 事件，调试时分别可查

**已知局限 / 未来待改**:
- 解析层与应用层之间的数据契约现在是 TypeScript 接口（`ParsedVarUpdate` / `ParsedEventOutcome`），靠类型系统约束。如果某天加 IPC 或跨进程调用，可能要改成 JSON schema
- `applyEventOutcomes` 当前接受 `reason` 字段但只写 trace 不写库（见下条决策）

**改动时注意**:
- 加新协议子标签（如 `<state>`）时：`aiProtocol.ts` 加 case + 加 `Parsed<X>` 接口；`server/services/<相关子系统>.ts` 加 `apply<X>` 函数；`chat.ts` 单点调用链路里加一行 dispatch
- 不要在 `parseAIOutput` 内部调任何 store / db 函数——这会破坏"解析层离线可测"的核心价值
- 应用层失败降级要保持"跳过该项、不抛错、其他项继续"的语义，否则 AI 一行格式错会导致整轮变量更新全部丢失

## [AI 协议] applyEventOutcomes 的 reason 字段当前只写 trace 不写库

**决策日期**: 2026-05-02
**涉及功能**: `server/services/eventEngine.ts` 的 `applyEventOutcomes`、`events` 表

**做法**:
AI 协议允许 `<event>` 子块写：

```
evt_confession: success | 原因: 用户没拒绝告白
```

`parseAIOutput` 把 `原因` 解析成 `ParsedEventOutcome.reason` 字段。`applyEventOutcomes` 接收到该字段后**只通过 traceDetail 记录**（事件类型 `event.outcome.applied`，data 含 `providedReason`），**不写入 events 表**。

events 表当前没有"outcome reason"列，只有 `outcome` 字段（字符串）。

**为什么这样做**:
- 阶段 2 的目标是"统一协议、收编两条 AI→DB 通路"，不是改 events schema
- `reason` 是 AI 的解释性元数据，对核心因果链路（branch 触发、状态机推进）不直接有用——branch 连接只看 outcome 字符串匹配
- trace 已经能查到"AI 说为什么标了这个 outcome"，调试和将来记忆系统消费足够
- 避免半成品 schema：要写库就该有完整的"outcome history"表（含时间戳、源消息 id、reason、是否被人工修正等），单加一个 reason 列是不彻底的解决方案

**已知局限 / 未来待改**:
- 如果未来需要在 UI 上展示"事件历史 + AI 当时给出的理由"，需要新增 `event_outcome_history` 表（events_id, outcome, reason, applied_at, source_message_id），并把 `applyEventOutcomes` 写入新表
- 当前 events 表的 `outcome` 字段是单值快照（最近一次结果覆盖前一次），没有历史记录。如果同一事件被多次重新标记 outcome，旧 outcome 丢失（仅 trace 能查回）
- 变量更新行的 `| 原因:` 也是同样状态：`applyVarBlock` 接收 reason 但只 trace、不写库

**改动时注意**:
- 添加 events 表的 reason 列前，先想清楚是否要做完整的 outcome 历史表——单加一列会把数据模型搞乱
- AI 协议层 `ParsedEventOutcome.reason` 接口保持不变；将来加表时改 `applyEventOutcomes` 内部即可，不影响协议层和 chat.ts
- 同样原则适用于 `applyVarBlock`：`ParsedVarUpdate.reason` 已可用，未来记忆系统需要时再决定写库形式
- 如果为了性能直接用 `better-sqlite3` 原生 SQL，必须放在 store/repository 层并说明原因
- 记忆系统如果引入向量能力，优先评估 PostgreSQL + pgvector；是否采用取决于实际部署条件和模型嵌入成本

## [ConversationMode] mode 不建表，保持 message 字段是字符串

**决策日期**: 2026-05-02
**涉及功能**: 对话模式切换、`message.mode` 字段、`sys-mode` 系统槽

**做法**:
不引入 `conversation_modes` 表。`message.mode` 字段保持现状——就是个字符串（`'online'` / `'offline'` / 未来作者自定义的字符串）。

模式提示词模板**存在 contextItem 的 content 里**，按 JSON 映射结构：

```json
{
  "online": "（聊天气泡场景：短消息为主，口语化、自然停顿）",
  "offline": "（长文叙事场景：每段 200-800 字，描写动作环境、内心活动）"
}
```

[server/services/context.ts](../server/services/context.ts) 的 `case 'mode'` 按 `currentMode` 字符串从映射里取对应模板注入。

**为什么这样做**:
- 当前唯一的需求是"切换 mode 时切换 prompt 模板"，建独立的 `conversation_modes` 表后还得做迁移逻辑（把已有 message 的 mode 字段改成外键）、删除 mode 时如何处理悬空引用、UI 加 CRUD 入口等等。每多一层都增加一份未来不一定用得上的负担
- contextItem 已经天然是"按预设包配置"的存储，作者切换不同 prompt 预设包就能换不同的 mode 模板风格——比建独立的 conversation_modes 表更灵活
- 字符串保持简单：未来如果想加自定义 mode，只是 JSON 多一个 key 的事，不需要改 schema、不需要 migration

**已知局限 / 未来待改**:
- 没有"模式列表"的中心化管理。前端 mode 切换器需要从 sys-mode 条目的 content keys 读列表（当前还是硬编码 `online/offline`，没补全）
- 删掉 sys-mode 条目后，某些消息的 `message.mode='online'` 会成为"悬空字符串"——但这并非真悬空（字符串本身合法），只是 prompt 不再注入对应模板。降级行为干净
- 没有 ui_style / event_filter_tags / max_tokens 等 ConversationMode 可能的属性。当前实现只能改"模板提示词"。如果未来这些耦合面真的有需求，再考虑独立表

**改动时注意**:
- 不要因为"看起来 mode 应该是个表"就建表——架构上要先确认是否有"按 mode 的非提示词属性需求"
- 加自定义 mode 时只需让 sys-mode 条目的 JSON 多一个 key；前端 mode 切换器读 keys 而不是硬编码
- 前端 [ChatApp.tsx](../client/src/apps/ChatApp.tsx) 当前 mode 切换按钮还是硬编码 `online/offline`，未来可能需要从 sys-mode 条目读 keys 来支持自定义 mode（**当前未做**）

## [ConversationMode] sys-mode 槽：双重解析处理 + 在 sys-history 之后

**决策日期**: 2026-05-02
**涉及功能**: [server/services/context.ts](../server/services/context.ts) 的 `sys-mode` 槽设计、SLOT_DEFS、DEFAULT_CONTEXT_ITEMS

**做法**:

**1. 槽位置**：`sys-mode` 在默认 contextItems 列表中放在 `sys-history` **之后**、`sys-syspost` **之前**：

```
... → sys-summaries → sys-history → sys-mode → sys-syspost
```

**2. 解析处理是"双重"的**：
- SLOT_DEFS 给 sys-mode 标记 `skipPlaceholders: true`，**统一出口跳过它**——不让 `resolvePlaceholders` 直接处理 raw content（content 是 JSON 字符串，整体过解析器毫无意义）
- `case 'mode'` 内部：先 `JSON.parse(content)`、按 `currentMode` 取对应模板字符串、**再显式调** `resolvePlaceholders(template, placeholderCtx)` 让作者写的 `{{char:name}}` 等占位符能被替换

**为什么这样做**:

**关于位置**:
- mode 切换会让其后所有 prompt 内容缓存失效。`sys-history` 通常是 prompt 中最长的段（数 KB），放在 mode 之前能保护这部分缓存——只有 sys-mode 之后的少量内容（sys-syspost）每次切换 mode 时缓存失效
- 仍放在 `sys-syspost` 之前，是因为 sys-syspost 通常是作者写的"最末尾必要规则"（输出格式、绝对禁止等），它语境上应该最后说话、不该被 mode 模板抢走 closure 位置

**关于双重解析**:
- 槽 raw content 是 JSON 字符串——如果统一出口对它过 `resolvePlaceholders`，会扫整个 JSON 字符串里的 `{{...}}`，但 JSON 里没有占位符，徒增开销
- 真正需要解析占位符的是**取出当前 mode 对应模板字符串之后**——作者在线上模板里可能写 "你是 {{char:name}}，处于聊天模式"，这时才该过解析器
- 用 `skipPlaceholders` 标记跳过统一出口、case 内部手动调 `resolvePlaceholders`，**双层处理避免无效解析、保证作者占位符能用**

**已知局限 / 未来待改**:
- "双重解析"对未来开发者可能不直观——看到 `skipPlaceholders=true` 容易误以为整个槽完全不过解析器。代码注释已写明"case 内部已自行解析"，但仍可能被改回去
- `case 'mode'` 内的 `JSON.parse(content)` 失败时（作者手动改坏了 JSON）只是 trace 警告 + 跳过，不报错给前端。如果未来需要在 UI 上提示"sys-mode JSON 格式错"，需要把诊断信息透传到 assemble 返回值

**改动时注意**:
- 调整槽位置时考虑 prompt 缓存影响——把变动频繁的槽放在长内容（如 sys-history、sys-memories）之后
- 给新加的"系统生成 / 系统二次处理"槽位标 `skipPlaceholders=true` 时，必须在 case 内部自己处理占位符解析，否则作者写的 `{{...}}` 完全失效
- sys-mode 的 content 是 JSON 字符串而非对象——这是因为 ColumnarStore 字段保存为 string；前端 [FilesApp.tsx](../client/src/apps/FilesApp.tsx) 编辑器在保存时 `JSON.stringify`、读取时 `JSON.parse` 还原。改动这种"JSON-as-string"约定时前后端要同步

## [Trigger/Effect] effect 注册表保留中英别名 + eventEngine 不主动注册

**决策日期**: 2026-05-03
**涉及功能**: [server/services/eventEngine.ts](../server/services/eventEngine.ts) 的 `registerEffect` / `effectRegistry` / `registerBuiltinEffects`、[server/index.ts](../server/index.ts) 启动钩子

**做法**:

**1. 别名作为 Map key**：每个 effect 注册时同时把英文 type 和中文 alias 都写入 `effectRegistry`，但 `listRegisteredEffects()` 只返回 meta 数组（去重后的逻辑 effect）。

```ts
registerEffect(
  { type: 'inject', aliases: ['注入'], description: '...', paramsHint: '...' },
  injectEffectHandler,
);
// → effectRegistry.set('inject', { meta, handler })
//   effectRegistry.set('注入', { meta, handler })  ← 别名也是 key
//   effectMetas.push(meta)                          ← 但 list 只一份
```

数据库已存的事件 `effects` JSON 字段里写的是 `'注入'` 还是 `'inject'`，executeEffect 都能正确派发。

**2. eventEngine 不主动注册自己为 trigger listener**：[triggerBus.ts](../server/services/triggerBus.ts) 不被 eventEngine.ts import；由 [server/index.ts](../server/index.ts) 在启动钩子里 `registerTriggerListener(checkAndFireEvents)` 完成连接。

**为什么这样做**:

**关于别名**:
- 数据库里历史数据用的是中文（如 `'注入'`、`'改数值'`），全切英文要做迁移
- 中英双键避免迁移；同时给作者两种风格的选择（写中文事件描述更顺手时用中文，写英文 schema 时用英文）
- 单一 meta 列表保证能力清单页不出现 16 条（8 个 type × 中英两份）

**关于不主动注册**:
- triggerBus 已经 import `TriggerContext` from eventEngine（单向依赖），如果 eventEngine 反过来 import triggerBus 注册自己，就成了循环引用（TypeScript ESM 在某些情况下会出现 undefined 问题）
- 由 index.ts 做"接线"是一种依赖倒置：eventEngine 和 triggerBus 都不知道对方存在，index.ts 把它们粘起来
- 未来如果要替换 listener 实现（比如换成异步 worker 模式），不动 eventEngine、不动 triggerBus，只动 index.ts 一行
- 代价：每加一个新 listener（如未来的后处理规则系统）都要在 index.ts 里加一行 `registerTriggerListener(...)`——这是显式注册的成本，但换来了模块边界清晰

**已知局限 / 未来待改**:
- 别名机制目前只在 effect 上实现；trigger 字符串本身没有别名（都是英文如 `'chat_end'`）。如果未来作者需要写中文 trigger，要扩展 trigger 总线
- index.ts 启动钩子越来越长（命名空间注册 + effect 注册 + listener 注册）。可以考虑抽到 `bootstrap.ts` 单文件，但当前还能容忍
- 重复 `registerBuiltinEffects()` 是幂等的（覆盖语义），但 `registerTriggerListener` 是去重添加（同一函数不重复加）——两者语义不一致。如果未来加新的注册函数，注意保持语义一致

**改动时注意**:
- 加新 effect 时，别名跟主名一起填入 `aliases` 字段；不要分两次 `registerEffect` 调用（会让 effectMetas 重复）
- 数据库里 `events.effects` JSON 改 effect type 的拼写时，记得检查别名表 — 改了主名而忘了删别名会导致两个 type 指向同一 handler、其中一个隐式失效
- 加新 listener 时**必须**在 index.ts 启动钩子里 `registerTriggerListener(...)`；不要在自己的 service 文件 import triggerBus 自己注册（破坏单向依赖）

## [Trigger/Effect] trigger 总线选 listener 模式，KNOWN_TRIGGERS 只是元数据

**决策日期**: 2026-05-03
**涉及功能**: [server/services/triggerBus.ts](../server/services/triggerBus.ts) 的 `registerTriggerListener` / `dispatchTrigger` / `listKnownTriggerTypes`

**做法**:

trigger 总线用的是 **listener 模式**：

```ts
type TriggerListener = (charId: string, ctx: TriggerContext) => void;

registerTriggerListener(listener);     // 加一个"我对所有 trigger 都感兴趣"的回调
dispatchTrigger(charId, ctx);          // 派发任意 trigger 字符串，所有 listener 都收到
```

**不是**按 trigger type 注册：

```ts
// 没有这种 API：
// registerTriggerHandler('chat_end', handler);
```

`listKnownTriggerTypes()` 返回的 `KNOWN_TRIGGERS` 只是**元数据列表**，给能力清单页 UI 展示用。它不限制 `dispatchTrigger` 接受的 trigger 字符串——任何字符串都能 dispatch、所有 listener 都会收到。

**为什么这样做**:
- 当前唯一的 listener（`checkAndFireEvents`）内部已经按 trigger 分支处理，不需要总线层做按 type 路由——会变成"总线分一次，listener 又分一次"的双重 dispatch
- listener 模式让总线**对 trigger 字符串完全开放**：未来作者写自定义后处理规则、AI 在某些 ctx 里产生新的 trigger 字符串、调试期临时插一个 trigger 字符串，都不需要先去注册 type
- 元数据展示和路由是两件事；混在一起会让"加一个 trigger 字符串"必须先 register 它的 description——而很多临时/实验性的 trigger 不需要
- 未来出现"我只关心 value_change trigger"的 listener 时，让 listener 自己内部 `if (ctx.trigger !== 'value_change') return;` 即可，比按 type 注册更灵活

**已知局限 / 未来待改**:
- 性能上每次 dispatch 都遍历所有 listener。当前 listener 数量极少（1 个），未来如果有几十个 listener 都需要按 type 过滤的话，可能引入"按 type 索引的 listener Map"作为优化。但这是"优化时再做"的事，不是当前设计的问题
- 如果某个 listener 是"重 CPU 操作"（比如 AI 调用），应该自己 fire-and-forget 不阻塞 dispatch；总线本身是同步遍历的
- `KNOWN_TRIGGERS` 可能跟实际派发的 trigger 字符串集合**不同步**——比如某个新功能 dispatch 了 `'mood_drift'` 但忘了把它加进 KNOWN_TRIGGERS。能力清单页就看不到。当前没有自动检查，需要靠 review

**改动时注意**:
- 加新 listener 时，**不**在 listener 入口就过滤 trigger 类型，而是在 listener 内部用 `if (ctx.trigger !== 'foo') return;` 早返回——保持总线 API 一致
- 加新 trigger 字符串时，记得同步加到 `KNOWN_TRIGGERS` 列表里（否则能力清单页看不到）
- 不要为 trigger 引入"必须先注册才能 dispatch"的限制；这会破坏对开放字符串的容忍设计
- listener 抛错只 console.error 不阻塞其他 listener（已实现），但**不要**让 listener 长时间同步阻塞——它会拖慢整个 dispatch 链
