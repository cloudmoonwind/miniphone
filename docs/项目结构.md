## 1. 数据库

### 1.1 一句话现状

SQLite + better-sqlite3（WAL 模式），单一文件 [server/data/ics.db](../server/data/ics.db)。Schema 由 Drizzle ORM 单一来源声明，所有 DDL 走 migration，运行时不再隐式建表。表分两层：核心/查询型表 = 列式表，配置/功能本地表 = JSON-in-TEXT（blob）。

### 1.3 表清单（35 张业务表 + 1 张迁移表）

#### 列式表（18 张）

核心实体（之前是 blob，0003 升级为列式）：
- `characters` — 角色主数据（FK 被 messages/summaries/memories 引用，CASCADE）
- `messages` — 对话消息（含 `mode`/`timestamp`/`variable_snapshot`，索引 `idx_messages_char_timestamp`）
- `summaries` — 摘要（按 `char_id + type + date` 索引）
- `memories` — 记忆（按 `char_id + importance` 索引）

数值系统（3 张）：
- `character_values` — 变量定义 + 当前值（唯一约束 `(character_id, variable_name)`）
- `value_stages` — 阶段名 + 描述 + promptSnippet
- `value_rules` — 自然语言规则文本（0002 重构后只剩 `rule_text` 字段）

事件系统（6 张）：
- `event_books` / `events` / `event_tags` / `event_connections` / `condition_subscriptions` / `pending_injections`

世界书 + 状态（4 张）：
- `worldbooks` / `worldbook_entries` / `worldbook_event_entries` / `world_state`

会话（1 张）：
- `sessions`

#### Blob 表（18 张）

格式统一为 `{id TEXT PK, char_id TEXT, data TEXT(JSON), created_at TEXT}`，每张表带 `idx_<table>_char_id` 索引：

`active`（singleton）/ `presets` / `prompt_presets` / `char_stats` / `stat_defs` / `life` / `items` / `timeline` / `skills` / `relations` / `dreams` / `personas` / `maps` / `calendar_events` / `dafu_game` / `diary` / `suixiang_cards` / `suixiang_entries`

#### 系统表

- `__drizzle_migrations` — Drizzle 自动维护的迁移记录（hash + created_at）

### 1.5 对外工作流

每个面向上层的功能都应能用「输入 → 经过 → 输出」一句话描述。

#### 1.5.1 启动时初始化

1. `server/index.ts` 第一次 `import` 触发 [storage/index.ts](../server/storage/index.ts)。
2. `getDb()` 打开 `ics.db`，设定 `journal_mode=WAL`、`foreign_keys=ON`、`synchronous=NORMAL`。
3. 同步运行 `migrate()`，把 [migrations/](../server/db/migrations/) 下未应用的脚本按 `_journal.json` 顺序执行。
4. 每个 Store 在构造时调用 `assertTable()` / `initTable()`，发现缺表或缺列直接抛错（提示跑 migration）。
5. 全部就绪后 `seedAllCharacters()` 兜底注入示例数据。

**结果**：路由层拿到的 Store 一定对应已就绪的、列结构正确的表。

#### 1.5.2 列式表读写（ColumnarStore）

- **写入**：路由层传入业务对象 → `itemToRecord()` 把已声明的列直接编码到对应 column，未声明的字段塞进 `metadata` JSON。
- **读取**：`rowToItem()` 先解 `metadata` 还原未列化字段，再用列值覆盖（列优先）。
- **意义**：业务代码不需要关心"列在 schema 里没"，未来加列只需在 schema + Store 列规格里声明，旧数据自动从 metadata 升级。

#### 1.5.3 Blob 表读写（SqliteStore）

- 整个对象 `JSON.stringify` 进 `data` 列；`char_id` 抽到独立列方便按角色查找。
- 不支持 SQL 内部过滤，只能 `getAll() + filter()`。
- 单对象模式（`active`）：`setObject()` 用 `INSERT ... ON CONFLICT DO UPDATE` 写固定 id `singleton`。

#### 1.5.4 备份

```bash
cd server && npm run db:backup
```

产出 `server/data/backups/<时间戳>/` 目录，包含：
- `ics.db` + `-wal` + `-shm` 文件级拷贝
- `ics.sqlite-backup.db`：通过 SQLite Online Backup API 生成的一致性副本
- `schema.jsonl` / `schema.sql`：完整 schema 原样
- `tables.txt` / `row-counts.json` / `migrations.json`：盘点元数据
- `manifest.json`：源路径 + 字节数 + 时间戳

#### 1.5.5 重置开发库

```bash
cd server && npm run db:reset:dev
```

流程：先备份 → 删除当前 db / wal / shm → 跑全部 migration 建空库 → ATTACH 备份 → 把测试数据迁回（核心表用 `json_extract` 从旧 blob 解出列；其余表按列名取交集复制） → seed → 生成 health 报告。

**用途**：schema 改了想重新跑一遍干净流程，但保留之前积累的测试数据。

#### 1.5.6 健康检查

```bash
cd server && npm run db:health
```

或 `db:reset:dev` 末尾自动调用。报告写到 `server/data/health-latest.md`，包含：
- 迁移版本列表
- 全表列/索引/外键
- 行数
- 孤儿数据扫描（messages/summaries/memories/character_values/events 是否引用了不存在的 character/event）
- 必备索引检查
- `PRAGMA foreign_key_check`

退出码：`0 = 无 issue`、`1 = 有 issue`，可接 CI。

### 1.6 当前选择与理由

| 选择 | 理由 |
|---|---|
| **SQLite 而不是 PostgreSQL** | 当前是单机本地 demo，零部署成本，单文件即数据库。设计文档里的 PG 是云端/多用户阶段的预案，未启用。 |
| **better-sqlite3 同步 API** | 单进程脚本场景，避免 async/await 噪音；WAL 模式让读写不互锁。 |
| **Drizzle ORM 仅用 schema/migrate，不用 query builder** | 业务层一直用原生 prepared statement + Store 抽象，Drizzle 只承担"声明 schema → 生成迁移 → 应用迁移"。引入查询构造器会和现有 SqliteStore/ColumnarStore 冲突。 |
| **核心表列式 + metadata 兜底**（而不是纯列式或纯 blob） | 核心表参与 prompt 组装、跨表外键、检索/排序，必须列式才能用索引和 SQL 过滤。但旧字段陆续迁移期间用 metadata 兜底，避免一次性破坏数据。 |
| **配置/功能本地表保留 blob** | active/presets/calendar 这类数据 ① 整对象读写、② shape 变动频繁、③ 不参与跨表查询。强行列式只增加迁移成本。 |
| **DDL 全部走 migration**（Store 不再 `CREATE TABLE IF NOT EXISTS`） | 此前 SqliteStore 隐式建表导致表结构和 schema 文件可能漂移。现在统一从 migration 出，启动时 Store 校验失败 = 提示跑迁移。 |
| **没有 ORM 级别的查询缓存 / 连接池** | 单进程 + 文件库，better-sqlite3 自带的 prepared statement 缓存够用。 |
| **备份策略 = 每次重置前自动 + 手动 db:backup** | 没有云端，且数据量小（KB~MB 级），文件级拷贝 + Online Backup 已足够。 |
