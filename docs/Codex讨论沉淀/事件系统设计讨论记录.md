# 事件系统设计讨论记录
## 一、讨论议题
本轮讨论围绕事件系统的后端设计展开，目标不是推翻原有系统，而是理清原设计中“事件定义、事件图、事件运行状态、提示词注入”之间的边界。
核心议题包括：
1. 事件是否就是事件图上的节点。
2. `event_connections` 应该表达哪些关系。
3. 事件定义和事件运行态是否需要拆分。
4. 事件图是否属于定义层，运行时是否需要单独的图运行实例。
5. 事件状态表应该记录哪些内容。
6. 事件触发、解锁、冷却、结果、重复触发之间的工作逻辑。
7. 事件触发后生成的提示词注入应该如何持续、过期和清除。
最终判断是：原事件系统主干成立，不需要推翻。原设计中的 `events + event_connections` 已经具备事件图雏形，其中 `events` 是事件节点，`event_connections` 是节点之间的有向边。真正需要调整的是：把事件定义和角色/上下文中的运行状态拆开，并收窄 `event_connections` 的关系语义。
## 二、讨论结论及理由
### 1. event 就是事件节点
当前版本中，一个 `event` 就是事件图上的一个点。事件不是抽象模板，也不需要再拆成“事件条目”和“图节点”。如果两个事件的提示词、触发条件或效果不同，就应该是两个不同事件。
理由：
当前系统里的事件是具体事件，而不是通用模板。强行拆成“模板 + 图节点实例”会引入覆盖参数、局部提示词、实例化等复杂度，当前没有必要。
### 2. 事件图关系用 `event_connections` 表达
事件之间的关系不应该存在单个事件字段里，而应该通过独立连接表表达。事件作为节点，连接作为边。每条边记录从哪个事件指向哪个事件。
理由：
事件系统本质上是稀疏有向图。一个事件可以有多个前置事件，也可以有多个后续事件。只记录父节点或只记录子节点都不合适。数据库只需要记录一条有向边，查询时既可以按起点找子节点，也可以按终点找父节点。
### 3. `event_connections.relation_type` 第一版只保留运行关系
原设计中的 `relation_type` 包含 `next`、`branch`、`trigger`、`unlock`。讨论后决定，第一版应收窄为：
* `unlock`：上游事件完成后，通知系统检查下游事件的解锁条件。
* `trigger`：上游事件完成后，通知系统检查下游事件的触发条件。
`next` 和 `branch` 暂时不作为核心运行关系。顺序感可以由连线方向和前端布局表达；分支感可以由一个事件连接多个后续事件、并由后续事件自己的条件表达。
理由：
`next`、`branch` 更像展示结构，`unlock`、`trigger` 才是调度器真正使用的运行逻辑。混在一个字段里会让一条边同时承担“画布展示”和“运行行为”两套语义，容易造成歧义。第一版保持字段干净更重要。
### 4. 事件定义和事件运行态需要拆分
事件定义不指定角色，因此不能在事件定义表里保存某个角色的触发状态、触发次数、冷却剩余、上次触发时间等运行记录。
理由：
同一个事件定义可以被不同角色、不同功能、不同预设、不同事件图运行实例使用。每个使用场景里的状态都可能不同。如果状态直接写在事件定义里，会造成不同角色或不同上下文之间互相污染。
### 5. 事件图也是定义层，事件图运行实例是运行层
事件图本身记录“这一组事件如何连接”，属于定义层。某个角色、某个功能、某个预设正在使用哪张事件图，则应由事件图运行实例记录。
理由：
同一张事件图未来可能被多个角色或多个上下文复用。图定义本身不应保存角色运行状态。运行实例负责绑定角色、功能、预设和当前启用的事件图。
### 6. 事件状态绑定图运行实例和事件定义
事件状态表记录“在某个图运行实例里，某个事件当前是什么状态”。一个图运行实例下，一个事件只应有一条状态记录。
理由：
这样可以保证同一张图在不同角色或不同功能下拥有独立状态，同时避免同一运行实例中同一事件重复触发或重复记录。
### 7. 不需要单独的终止结果字段
表白成功后不能再次触发、表白失败后可以再次触发，这类逻辑可以直接写在事件的触发条件里。例如：触发条件包含“当前事件结果状态不等于成功”。
理由：
能通过条件系统表达的逻辑，不需要额外增加字段。这样字段更少，判断逻辑更统一。事件能否再次触发，由最大触发次数、冷却规则、触发条件和当前结果状态共同决定。
### 8. `repeatable` 和 `max_triggers` 可以合并
不再同时保留“是否可重复”和“最大触发次数”两个字段。建议只保留“最大触发次数”。
推荐定义：
* 空值 / `null`：不限制触发次数。
* `1`：最多触发一次。
* 其他正整数：最多触发对应次数。
理由：
`repeatable` 和 `max_triggers` 容易表达重复语义，保留一个字段即可。`null` 表示不限，比用 `0` 表示无限更清楚，因为 `0` 容易被理解为不能触发。
### 9. 注入持续时间属于提示词注入任务，不属于事件状态
事件触发后可以生成提示词注入。注入可以持续一轮、三轮、五轮，或者持续到某个条件满足。这个生命周期应该由 `pending_injections` 记录。
理由：
注入是事件 effects 产生的上下文任务，不是事件本身的流程状态。注入过期后，只需要删除或失效该注入任务，不代表事件需要回到 locked 或改变事件流程状态。
## 三、事件系统工作逻辑
事件系统整体遵循：先解锁，再触发；连接只负责通知检查，复杂条件放在事件自身。
### 1. 状态变化唤醒订阅
当系统中发生状态变化时，例如数值变化、时间变化、地点变化、用户输入、事件完成，调度器通过 `condition_subscriptions` 找到关心该变化的事件。
### 2. locked 事件先检查解锁条件
如果被唤醒的事件当前是 locked，则先检查它的解锁条件。解锁条件用于判断该事件是否获得“可以等待触发”的资格。
例如事件 a 需要事件 A 和事件 B 都完成，则应写在 a 的解锁条件里：A 已完成，并且 B 已完成。
A 完成时，系统检查 a，发现 B 未完成，a 仍保持 locked。B 完成时，系统再次检查 a，发现 A 和 B 都已完成，a 进入 pending。
### 3. 解锁成功后立即检查触发条件
事件从 locked 变为 pending 后，应立即检查一次触发条件。如果触发条件已经满足，则进入本轮候选事件池；如果不满足，则保持 pending，等待后续状态变化再次唤醒。
### 4. pending 事件直接检查触发条件
如果事件已经是 pending，当相关变量变化时，调度器直接检查它的触发条件。满足则进入候选池，不满足则继续等待。
### 5. `unlock` 和 `trigger` 的区别
`unlock` 表示：上游事件完成后，通知系统检查下游事件的解锁条件。它不强制解锁，也不绕过下游事件自己的条件。
`trigger` 表示：上游事件完成后，通知系统检查下游事件的触发条件。它不强制执行，也不绕过冷却、次数、状态和触发条件。
大多数事件链使用 `unlock` 即可。`trigger` 适合即时反应类事件，例如某个事件发生后，需要立刻检查另一个已经解锁的反应事件。
### 6. 候选池允许多个事件
系统允许同一轮有多个满足条件的事件进入候选池。调度器根据优先级、触发次数、冷却、状态、每轮数量限制等规则决定实际执行哪些事件。
同一事件在同一轮中只应执行一次。重复的 unlock 或 trigger 通知应做幂等处理。
### 7. 事件触发后的状态
普通瞬时事件触发后，如果仍可能再次触发，不应标记为 completed，而应在本次触发结束后根据冷却和条件回到 locked 或 pending。completed 用于表示该事件在当前图运行实例中已经结束，不再进入常规触发检查。
需要等待结果的事件触发后可以进入 active。之后由角色回复检测、用户标记或系统判断更新结果。结果状态和结果摘要并不绑定 completed；它们记录的是本次或最近一次触发的结果。若结果意味着事件仍可再次触发，则写入结果状态和结果摘要后回到 locked 或 pending；若结果意味着事件永久结束，则写入结果状态和结果摘要后标记为 completed。
### 8. 事件结果影响后续触发
事件状态中记录结果状态和结果摘要。例如表白事件可以有成功、失败、被打断、未定等结果。结果状态和结果摘要不只属于 completed 事件，可重复事件每次触发结束后也可以更新最近一次结果。
如果希望“失败后可重试，成功后不可重试”，不需要修改事件定义，也不需要单独终止字段。直接在触发条件中加入“当前事件结果状态不等于成功”。失败时写入 `failed` 后回到 locked 或 pending；成功时写入 `success` 后标记 completed，或由触发条件阻止后续触发。
### 9. 提示词注入的生命周期独立
事件触发后产生的提示词注入进入 `pending_injections`。注入可以持续固定轮次、持续到某个时间、或持续到某个清除条件满足。注入结束后，只清除注入任务，不改变事件本身的流程状态。
## 四、修改后的事件系统后端数据表
当前建议的数据表包括：
1. `event_books`：事件书表。
2. `events`：事件定义表。
3. `event_tags`：事件标签表。
4. `event_connections`：事件连接表。
5. `event_graphs`：事件图定义表。
6. `event_graph_runs`：事件图运行实例表。
7. `event_states`：事件状态表。
8. `condition_subscriptions`：条件订阅表。
9. `pending_injections`：待注入提示词表。
其中，`events` 和 `event_graphs` 属于定义层；`event_graph_runs`、`event_states`、`pending_injections` 属于运行层。
## 五、数据表具体字段及定义
### 1. `event_books`：事件书表
用于存储事件书。事件书是事件定义的集合。
建议字段：
* `id`：事件书编号。
* `name`：事件书名称。
* `description`：事件书描述。
* `enabled`：是否启用。
* `created_at`：创建时间。
* `updated_at`：更新时间。
### 2. `events`：事件定义表
用于存储具体事件定义。事件定义不指定角色，不存运行状态。
建议字段：
* `id`：事件编号。
* `book_id`：所属事件书编号。
* `name`：事件名称。
* `description`：事件描述。
* `enabled`：是否启用。
* `priority`：优先级。
* `probability`：触发概率。
* `max_triggers`：最大触发次数。
* `unlock_conditions`：解锁条件。
* `trigger_conditions`：触发条件。
* `effects`：触发效果。当前主要用于生成提示词注入，未来也可以扩展为改数值、设标记、写历史、改地点、触发系统动作等。
* `cooldown_type`：冷却类型。
* `cooldown_value`：冷却值。
* `condition_cooldown`：条件冷却规则。
* `created_at`：创建时间。
* `updated_at`：更新时间。
不建议放在事件定义表中的字段：
* `character_id`：角色编号。事件定义不指定角色。
* `status`：当前流程状态。应放入事件状态表。
* `trigger_count`：已触发次数。应放入事件状态表。
* `cooldown_remaining`：剩余冷却。应放入事件状态表，或用下次可触发时间表达。
* `condition_cooldown_remaining`：剩余条件冷却。应放入事件状态表。
* `last_triggered_at`：上次触发时间。应放入事件状态表。
* `current_step`：当前步骤。当前多步骤事件用事件链表达，不放事件定义。
* `steps`：步骤定义。当前暂不作为主设计使用。
### 3. `event_tags` / `events.tags`：事件标签
用于给事件打标签，方便筛选、分类和编辑器展示。
第一版可以直接在 `events` 表中使用 `tags` 字段，存字符串数组或 JSON 数组。这样更轻，不必单独建标签表。
只有在未来需要大量按标签查询、标签统计、标签管理或跨表复用时，再单独拆出 `event_tags` 表。
建议第一版字段：
* `events.tags`：事件标签列表。
### 4. `event_connections`：事件连接表
用于存储事件图中的有向边。每条连接表示从一个事件指向另一个事件。
建议字段：
* `id`：连接编号。
* `graph_id`：所属事件图编号。
* `from_event_id`：起点事件编号。
* `to_event_id`：终点事件编号。
* `relation_type`：连接关系类型。
* `created_at`：创建时间。
* `updated_at`：更新时间。
第一版 `relation_type` 建议只保留：
* `unlock`：检查下游事件的解锁条件。
* `trigger`：检查下游事件的触发条件。
### 5. `event_graphs`：事件图定义表
用于存储事件图定义。事件图是事件和连接组成的网络，属于定义层。
建议字段：
* `id`：事件图编号。
* `name`：事件图名称。
* `description`：事件图描述。
* `book_id`：关联事件书编号，可选。如果一张图只使用某本事件书中的事件，可以保留该字段。
* `enabled`：是否启用。
* `created_at`：创建时间。
* `updated_at`：更新时间。
说明：事件图本身不一定绑定角色和预设。角色、功能、预设的绑定建议放在事件图运行实例表中。
### 6. `event_graph_runs`：事件图运行实例表
用于记录某个角色、某个功能、某个预设正在运行哪张事件图。
建议字段：
* `id`：图运行实例编号。
* `graph_id`：对应事件图编号。
* `character_id`：角色编号。
* `function_key`：功能标识，例如聊天、日程、地图、离线生活。
* `preset_id`：预设编号。
* `is_active`：是否为当前活跃运行实例。
* `started_at`：开始运行时间。
* `ended_at`：结束运行时间，可为空。
* `created_at`：创建时间。
* `updated_at`：更新时间。
建议约束：同一角色、同一功能、同一时间只允许一个活跃事件图运行实例。
### 7. `event_states`：事件状态表
用于记录某个图运行实例里，某个事件当前的运行状态。
建议字段：
* `id`：事件状态编号。
* `graph_run_id`：所属事件图运行实例编号。
* `event_id`：对应事件编号。
* `status`：流程状态。
* `result_status`：结果状态。
* `result_summary`：结果摘要。
* `trigger_count`：已触发次数。
* `last_triggered_at`：上次触发时间。
* `next_available_at`：下次可触发时间。
* `next_available_turn`：下次可触发轮次，可选。
* `condition_hit_count`：条件命中次数，可选。
* `created_at`：创建时间。
* `updated_at`：更新时间。
建议约束：`graph_run_id + event_id` 唯一。即同一个图运行实例里，同一个事件只能有一条状态记录。
### 8. `condition_subscriptions`：条件订阅表
用于支持订阅制触发。它是程序根据事件的解锁条件和触发条件自动生成/维护的动态索引表。事件进入待检查范围后，系统根据其关心的条件建立订阅；变量系统、时间系统、地点系统、用户输入或事件状态发生变化时，只唤醒相关事件。
建议字段：
* `id`：订阅编号。
* `graph_run_id`：所属事件图运行实例编号。
* `event_id`：对应事件编号。
* `condition_type`：条件类型。
* `condition_key`：条件键。
* `condition_value`：条件值或辅助信息。
* `created_at`：创建时间。
* `updated_at`：更新时间。
说明：订阅表服务运行实例，因此建议带 `graph_run_id`，避免不同角色或不同上下文之间互相污染。
### 9. `pending_injections`：待注入提示词表
用于记录事件触发后产生、尚未过期的提示词注入任务。
建议字段：
* `id`：注入任务编号。
* `graph_run_id`：所属事件图运行实例编号。
* `event_id`：来源事件编号，可为空。
* `injection_content`：注入内容。
* `injection_position`：注入位置。
* `duration_type`：持续类型。
* `duration_value`：持续值。
* `remaining_turns`：剩余轮次。
* `expires_at`：过期时间。
* `clear_condition`：清除条件。
* `is_active`：是否仍有效。
* `created_at`：创建时间。
* `updated_at`：更新时间。
说明：注入持续时间属于注入任务，而不是事件状态。注入结束后，只清除或失效该注入任务，不改变事件流程状态。
## 六、部分字段的值类型及简要说明
### 1. 事件流程状态 `event_states.status`
建议枚举值：
* `locked`：未解锁。
* `pending`：已解锁，等待触发条件满足。
* `active`：事件已触发，正在等待结果或后续判断。
* `completed`：本次触发流程已完成。
* `cancelled`：事件被取消。
说明：`completed` 表示该事件在当前图运行实例中已经结束，不再进入常规触发检查。可重复事件在一次触发结束后通常回到 `locked` 或 `pending`，再由冷却、触发次数、触发条件和结果状态决定是否能再次触发。
### 2. 事件结果状态 `event_states.result_status`
建议枚举值：
* `none`：无特殊结果。
* `success`：成功。
* `failed`：失败。
* `interrupted`：被打断。
* `unresolved`：未定或暂未得到明确结果。
说明：结果状态用于后续事件判断和短摘要注入。例如表白事件成功后，触发条件可以写“结果状态不等于 success”，从而阻止再次触发。
### 3. 最大触发次数 `events.max_triggers`
建议类型：整数，可为空。
* `null`：不限制触发次数。
* `1`：最多触发一次。
* 正整数 N：最多触发 N 次。
说明：不再单独保留 `repeatable`。最大触发次数已经能表达是否可重复。
### 4. 连接关系 `event_connections.relation_type`
建议枚举值：
* `unlock`：上游事件完成后，检查下游事件的解锁条件。
* `trigger`：上游事件完成后，检查下游事件的触发条件。
说明：连接关系不直接绕过目标事件自身条件。复杂前置条件应写在目标事件的 `unlock_conditions` 或 `trigger_conditions` 中。
### 5. 冷却类型 `events.cooldown_type`
建议枚举值：
* `none`：无冷却。
* `time`：按真实时间或世界时间冷却。
* `turn`：按对话轮次冷却。
* `condition_hit`：按条件命中次数冷却或延迟触发。
说明：冷却规则属于事件定义。冷却进度属于事件状态，例如 `next_available_at` 或 `next_available_turn`。
### 6. 注入位置 `pending_injections.injection_position`
可根据现有 prompt 结构定义，例如：
* `before_character`：角色提示词前。
* `after_character`：角色提示词后。
* `status_section`：状态区。
* `history_section`：历史区。
* `deep_position`：指定深度位置。
说明：注入位置用于控制事件产生的提示词内容进入 AI 上下文的位置。
### 7. 注入持续类型 `pending_injections.duration_type`
建议枚举值：
* `once`：只注入一次。
* `turns`：持续若干轮。
* `time`：持续到某个时间。
* `until_condition`：持续到清除条件满足。
* `permanent`：永久有效，慎用。
说明：持续类型控制 pending injection 的生命周期。持续轮次和过期时间记录在注入任务里。
## 七、补充运行规则
### 1. 可重复事件结束后的状态回流
可重复事件触发结束后，不应直接标记为 completed。若事件仍可能再次触发，应根据其解锁条件重新判断：
* 如果解锁条件仍然满足，回到 `pending`。
* 如果解锁条件不再满足，回到 `locked`。
也就是说，事件结束后可以先重新检查解锁条件，再决定回流状态。若简单地回到 locked，下一次全图检查时也可以再次把它放入 pending，但即时检查会更及时、更符合调度逻辑。
### 2. 事件状态初始化
事件图运行实例第一次运行时，建议为图内所有事件生成事件状态记录，默认 `locked`。随后进行一次初始化检查：
* 无解锁条件，或解锁条件满足的事件，进入 `pending`。
* 进入 `pending` 后立即检查触发条件。
* 若触发条件为空或满足，则进入候选池。
这样不需要在事件定义里增加“默认状态”字段。默认都是 locked，初始化检查负责把应当可用的事件推进到 pending 或候选池。
极端情况下，如果某事件没有任何解锁限制，也没有任何触发限制，可以理解为：解锁条件为空，触发条件为空。初始化时它会进入 pending，并立刻进入候选池。若不希望它无限连续触发，需要通过最大触发次数、冷却或条件限制控制。
### 3. 事件结果标记方式
事件结果标记方式应属于事件定义的一部分。建议在事件定义中增加 `result_marking_mode` 字段，用于决定事件结果由谁标记。
建议值：
* `none`：不需要结果标记。
* `ai`：由角色回复中的 sys/xml 更新块自动标记。
* `user`：由用户手动标记。
* `ai_suggest_user_confirm`：AI 给出建议结果，但由用户确认。
当前建议：需要用户决定成功/失败的事件，使用 `user` 或 `ai_suggest_user_confirm`。程序可在最后一轮注入后，在界面顶部弹出结果标记弹窗，由用户选择结果。
### 4. effects 第一版最小格式
第一版 effects 主要用于生成提示词注入，但字段语义应保留扩展空间。
建议 effects 使用数组，每一项是一个效果对象。当前最小效果类型为 `inject_prompt`。
示例结构：
```json
[
  {
    "type": "inject_prompt",
    "content": "路西刚刚经历了一次表白失败。他表面平静，但对 Agnes 的反应非常在意。",
    "position": "status_section",
    "duration_type": "turns",
    "duration_value": 3,
    "visibility": "internal"
  }
]
```
字段含义：
* `type`：效果类型。当前主要是 `inject_prompt`。
* `content`：注入内容。
* `position`：注入位置。
* `duration_type`：持续类型，例如 once、turns、time、until_condition。
* `duration_value`：持续值，例如 3 轮。
* `visibility`：可见性，例如 internal 表示只进入 AI 上下文，不直接展示给用户。
未来可扩展的效果类型包括改变量、设标记、写历史、改地点、触发系统动作等。
### 5. 条件体系
条件由变量系统统一提供。能进入条件判断的对象都应注册为变量或变量空间中的值，包括数值、时间、地点、事件状态、事件结果、用户输入解析结果等。
事件的解锁条件和触发条件都基于这些已注册变量和值进行判断。
条件 JSON 第一版采用两层结构，不做无限嵌套：
* 条件组之间有组间逻辑：`and` 或 `or`。
* 每个条件组内部有组内逻辑：`and` 或 `or`。
* 每一条具体条件是一行。
这样可以表达：
* `A AND B`
* `A OR B`
* `(A AND B) OR (C AND D)`
* `(A OR B) AND (C OR D)`
不做无限嵌套，避免编辑器和人工填写过于复杂。
条件检查应基于本轮调度开始时的状态快照。本轮触发产生的新状态，会写入数据库，但不反向影响本轮已经开始的条件判断。它会在下一次调度中生效。
### 6. 事件状态 / 订阅 / 注入的实际代码流程
第一版运行流程可以按以下顺序实现。
#### 6.1 创建事件图运行实例
当某个角色、某个功能、某个预设启用某张事件图时，创建一条 `event_graph_runs`。
系统读取该事件图中的所有事件，为每个事件生成一条 `event_states`，默认状态为 `locked`。
#### 6.2 初始化检查
运行实例第一次启动时，对图内所有事件做一次初始化检查。
对每个事件：
1. 读取事件定义。
2. 读取该运行实例下对应的事件状态。
3. 检查 `unlock_conditions`。
4. 若无解锁条件，或解锁条件满足，则把状态从 `locked` 改为 `pending`。
5. 进入 `pending` 后立刻检查 `trigger_conditions`。
6. 若触发条件为空或满足，则进入本轮候选池。
初始化后，系统根据仍需等待的条件生成或刷新 `condition_subscriptions`。
#### 6.3 生成条件订阅
`condition_subscriptions` 是程序自动生成和维护的动态索引表。
系统解析事件的 `unlock_conditions` 和 `trigger_conditions`，找出它依赖哪些变量，例如：
* 关系数值。
* 时间段。
* 地点。
* 天气。
* 事件状态。
* 事件结果。
* 用户输入解析结果。
然后为该事件在当前 `graph_run_id` 下写入订阅记录。
订阅表不代替完整条件判断。它只负责“状态变化时该唤醒哪些事件”。事件被唤醒后，仍然要检查完整条件 JSON。
#### 6.4 状态变化时唤醒事件
当变量系统、时间系统、地点系统、用户输入解析、事件状态或事件结果发生变化时，系统根据变化的变量 key 查询 `condition_subscriptions`。
查到相关事件后，根据事件当前状态分别处理：
* 如果是 `locked`，检查解锁条件。满足则进入 `pending`，并立即检查触发条件。
* 如果是 `pending`，检查触发条件。满足则进入候选池。
* 如果是 `active`，等待结果标记或后续处理，不参与普通触发检查。
* 如果是 `completed` 或 `cancelled`，默认不进入常规触发检查。
#### 6.5 处理事件连接
当某个事件触发结束，或者被标记结果后，系统检查 `event_connections` 中从该事件出发的连线。
* `unlock`：通知系统检查下游事件的解锁条件。
* `trigger`：通知系统检查下游事件的触发条件。
连接本身不绕过目标事件的完整条件、冷却、触发次数和状态检查。
#### 6.6 候选池与执行
系统把所有通过触发条件的事件放入候选池。
候选池处理规则：
1. 基于本轮调度开始时的状态快照检查条件。
2. 同一事件同一轮只进入候选池一次。
3. 按优先级排序。
4. 优先级相同时按稳定顺序排序，例如事件编号或更新时间。
5. 根据本轮触发上限选择要执行的事件。
6. 执行事件 effects。
7. 更新事件状态、触发次数、冷却进度。
本轮执行 effects 产生的新变量变化，不反向影响本轮已经开始的条件判断。它会在下一次调度中生效。
#### 6.7 执行 effects 并生成 pending injections
第一版 effects 主要实现 `inject_prompt`。
当事件 effect 为 `inject_prompt` 时，系统根据 effect 定义生成一条 `pending_injections`。
`pending_injections` 记录：
* 来源运行实例。
* 来源事件。
* 注入内容。
* 注入位置。
* 持续类型。
* 持续值。
* 剩余轮次或过期时间。
* 是否有效。
如果注入持续 3 轮，意思是它实际进入上下文并通过 API 发给 AI 三次。每成功发送一次，剩余轮次减 1。达到次数后失效或删除。
#### 6.8 生成 AI 上下文
每次准备调用 AI API 前，系统查询当前角色、当前功能、当前预设对应的活跃 `event_graph_runs`。
然后查询该运行实例下有效的 `pending_injections`，按照注入位置和顺序拼入上下文。
AI 回复后，系统解析 sys/xml 更新块，更新变量、事件状态或事件结果。若某事件定义要求用户标记结果，则在对应注入结束后由前端弹窗让用户选择。
#### 6.9 一轮结束后的清理
一轮 API 调用结束后，系统处理注入寿命：
* `once` 类型：使用一次后失效。
* `turns` 类型：每实际注入并发送一次，剩余轮次减 1。
* `time` 类型：超过过期时间后失效。
* `until_condition` 类型：清除条件满足后失效。
同时，系统根据事件的冷却规则更新 `next_available_at` 或 `next_available_turn`。
### 7. 同轮多事件触发顺序
同一轮允许多个事件进入候选池。第一版建议采用稳定、有限深度的调度规则：
1. 收集本轮被唤醒的事件。
2. 检查解锁条件，必要时推进 locked 到 pending。
3. 检查 pending 事件的触发条件。
4. 将满足条件的事件放入候选池。
5. 按优先级排序；优先级相同时按更新时间或事件编号保证稳定顺序。
6. 按本轮触发上限选择要执行的事件。
7. 执行选中事件的 effects。
8. 更新事件状态、触发次数、冷却、结果等待状态。
9. 处理由事件完成带来的 `event_connections` 通知。
10. 第一版建议不做无限递归触发；若要处理连锁触发，最多允许一层或设置最大调度深度。
核心原则：同一事件同一轮只执行一次；重复 unlock/trigger 通知只触发一次检查；effect 引发的变量变化可以进入下一轮或下一次调度，不建议第一版在同一轮无限滚动。
### 7. 注入轮次计算
注入持续 N 轮，表示该注入内容实际被放入上下文并通过 API 发给 AI N 次。每成功发送一次，剩余轮次减 1。发送次数达到 N 后，该 pending injection 失效或删除。
### 8. 活跃图运行实例约束
同一角色、同一功能、同一时间只允许一个活跃事件图运行实例。该约束应通过程序检查或数据库唯一约束保证。
### 9. 日志
系统已有日志功能。事件系统接入后，应记录事件检查、触发、状态变化、注入生成、结果标记等关键行为，便于调试。
## 八、当前保留与暂不处理的内容
当前保留：
* 订阅制触发。
* 多事件同轮触发。
* 多步骤事件用事件链表达。
* 互斥优先通过触发条件表达。
* 注入持续由 pending injection 管理。
暂不处理：
* 事件模板与图节点实例拆分。
* 强互斥组。
* 事件组/同时激活组。
* 复杂事务回滚。
* AI intent planner。
* 单独事件日志表。
这些不是被否定，而是当前阶段不急。当前优先目标是让事件定义、事件图、运行实例、事件状态和提示词注入的边界清楚，并让第一版系统稳定跑起来。
## 九、修订决策（2026-05-03 定稿）
本节为代码评审后的修订定稿。与正文有冲突时以本节为准。
### 1. 冷却字段统一使用快照
取消 `events.cooldown_remaining` 与 `events.condition_cooldown_remaining`。冷却进度由 `event_states` 中的字段表达：
* `next_available_at`（ISO 时间戳）：用于 `cooldown_type='time'`，与 `new Date()` 比较。
* `next_available_turn`（整数）：用于 `cooldown_type='turn'`，与所属图运行实例的 `turn_counter` 比较。
* `condition_hit_count`（整数）：用于 `cooldown_type='condition_hit'`，累计达到 `cooldown_value` 才允许触发。
`event_graph_runs` 增加 `turn_counter` 字段，每轮 `chat_end` 时 +1。调度器判断冷却时直接读快照，不再每轮 tick 减 1。
### 2. 取消独立的 `condition_cooldown` 字段
不再保留 `events.condition_cooldown`。原“条件命中 N 次才触发”语义改用 `cooldown_type='condition_hit'` + `cooldown_value=N` 表达，运行态用 `event_states.condition_hit_count`。冷却字段从两套合并为一套。
### 3. 取消 `repeatable` 字段
只保留 `events.max_triggers`，语义为：
* `null`：不限触发次数。
* `1`：最多触发一次。
* 正整数 N：最多触发 N 次。
### 4. 第一版去掉的字段/枚举
* `effects` 项的 `visibility` 字段（后续按需再引入，第一版注入默认进入 AI 上下文，不直接展示给用户）。
* `result_marking_mode` 的 `ai_suggest_user_confirm` 值（只保留 `none / ai / user`）。
### 5. `event_result` 进入订阅类型
`condition_subscriptions.condition_type` 新增枚举值 `event_result`。当某事件的 `result_status` 改变时，系统按 `(graph_run_id, condition_type='event_result', condition_target=event_id)` 唤醒下游事件。砍掉 `branch` 边后，“按 outcome 分支”改由下游事件的 `trigger_conditions` 包含“事件结果等于 success”等判断表达，订阅唤醒由该字段保证精确性。
### 6. `trigger` 边不绕过目标事件条件
`event_connections` 中 `relation_type='trigger'` 的语义为“通知系统检查下游事件的触发条件”，不直接调用 `fireEvent`。下游事件仍需经过完整的冷却、最大次数、状态、触发条件、概率检查。
### 7. `trigger_event` effect 的作用域
`trigger_event` effect 默认作用于发出该 effect 的事件所在的 `graph_run_id`，不跨 run。如未来需要跨 run 触发，再在 effect 参数中显式增加目标 graph_run 字段。
### 8. 数据迁移策略
当前为开发期，事件系统数据全部清库重来。落地时直接 drop 旧表（`events / event_connections / condition_subscriptions / pending_injections`），按新 schema 重建，不写数据迁移脚本。
