# 变量、状态、触发效果与角色主动性设计思路 v1

**日期**：2026-05-02
**来源**：Agnes 与 Codex 关于 ICS 角色连续性、变量系统、Trigger/Effect、角色主动行为的讨论沉淀
**定位**：思路文档，不是实现规格。用于后续拆分专项设计文档或与 CC 克协作前统一概念。

---

## 1. 核心结论

本次讨论确认：ICS 里的“角色鲜活感”不应只依赖聊天回复和记忆检索，而应来自一套可读、可写、可触发、可沉淀的运行结构。

本项目不是单纯复刻 SillyTavern 或其他“小手机”项目。它的独立价值在于：把角色扮演中长期靠用户经验、提示词技巧和手工维护支撑的隐形连续性，转成可调度、可解释、可沉淀的系统。

这里的目标不是“比已有工具功能更多”，而是：

```text
让角色知道为什么某条记忆此刻该出现。
让系统知道为什么某个事件改变了关系。
让变量和事件支撑角色长期变成什么样。
让角色在稳定范围内产生意外，而不是每轮重生。
```

可以先按以下职责区分：

```text
变量系统：现在是什么
变量管道：把“现在是什么”读出来注入提示词
Trigger/Effect：当条件满足时，自动做什么
事件系统：把一组触发和效果编排成有剧情意义的流程
记忆系统：把经历沉淀为可被未来重新调度的过去
工具/动作系统：让角色在 app 内创造可见产物
```

一句话原则：

```text
变量描述状态，Trigger/Effect 制造因果，事件系统包装剧情意义。
```

补充原则：

```text
角色连续性不是全知检索。
角色主动性不是乱用工具。
角色成长不是机械解锁。
角色鲜活不是更吵，而是更有分寸、更有时间感、更能留下痕迹。
```

## 2. 角色主动性的定位

只在 app 里的角色要变得鲜活，不只是“多回复几句话”，而是要拥有聊天之外的主动表达通道。

这些能力可以统称为：

```text
角色主动表达通道 / Presence Actions
```

示例：

- 主动弹窗消息
- 写信
- 留便签
- 准备礼物
- 生成梦境碎片
- 创建未发送草稿
- 更新生活状态
- 埋下延迟触发的伏笔
- 创建可保存的纪念物或文本物件

这些不是变量本身，而是动作、工具或 Effect。变量只记录角色是否倾向于这样做、是否具备能力、是否已有相关产物。

例如“写信”相关概念应拆开：

```text
技能变量：letter_expression_skill = basic / intimate / reflective
驱动变量：repair_relationship = high
状态变量：action_posture = drafting_unsent_letter
Flag：unsent_letter_exists = true
Effect/action：create_letter_draft / send_letter
```

### 2.1 主动表达的三层结构

角色主动行为不应设计成“AI 想调用什么工具就调用什么工具”。更稳的拆法是三层：

```text
Action：角色能做什么
Intent：角色为什么做
Delivery：什么时候、以什么强度、是否打扰用户
```

Action 示例：

```text
send_message
write_letter
create_gift
create_memory_anchor
schedule_event
update_status
create_dream_fragment
create_unsent_draft
```

Intent 示例：

```text
安慰
试探
撒娇
道歉
逃避
纪念
求关注
隐藏真实想法
兑现承诺
制造惊喜
修复关系
确认自己是否被在意
```

Delivery 示例：

```text
立即弹窗
静默进入信箱
下次打开 app 展示
延迟到晚上
需要用户主动发现
只记录不发送
只生成草稿不投递
```

因此完整链路更像：

```text
角色形成意图
        ↓
系统根据状态、技能、场景、冷却判断允许哪些 Action
        ↓
系统选择或限制 Delivery
        ↓
AI 生成具体内容
        ↓
产物存表、投递或延迟
        ↓
写回事件/变量/记忆
```

这能避免角色变成“会乱按按钮的客服机器人”。角色可以有冲动，但 app 要有礼仪。

### 2.2 主动存在的几类方式

角色主动性可以先按存在方式分，而不是按工具名称分。

第一类：主动靠近用户。

示例：

- 弹窗
- 短消息
- 未读便签
- 语音留言
- 状态更新

关键不是频率，而是“为什么现在来”。如果只是定时问候，会像运营推送；如果来自昨晚事件、梦境残留、关系张力、用户久未打开、角色驱动升高，就更像角色自己冒出来。

第二类：制造可保存的物件。

示例：

- 信
- 便签
- 礼物
- 梦境碎片
- 手账页
- 一段被封存的回忆
- 一个私人称呼
- 一个只有双方知道含义的纪念物

聊天是流动的，物件是沉积的。物件可以成为记忆锚点、事件入口或后续触发条件。

第三类：改变自己的状态。

角色不一定总要对用户做什么。它也可以更新自己的生活状态、心理姿态或未完成动作。

示例：

- 今天没有主动说话，但状态变成“把那封信删了又重写”
- 梦境之后对白天某个话题更敏感
- 冲突后开始回避某个称呼
- 角色给自己立了一个小目标，之后可能成功或失败

这会让角色不是等用户输入才“开机”。

第四类：延迟回应和伏笔。

AI 角色最容易缺的是时间感。延迟回应可以让角色显得真的记得自己还有债。

示例：

- “我明天再告诉你”真的明天出现
- 争吵后隔一段时间写信，而不是当场和好
- 角色当时没反应，之后在梦境里反应
- 用户送过的东西在后续生活事件里被提到
- 某句玩笑后来变成事件种子

### 2.3 角色也可以隐藏东西

不要只设计角色能发给用户什么，也要设计角色能藏起什么。

示例：

- 没发出的信
- 删掉又重写的草稿
- 梦里说出口但醒来不承认的话
- 准备了但没送出的礼物
- 只写进日记、不进入聊天的想法
- 以后才会被发现的物件

角色的鲜活不只是表达，也包括犹豫、保留、错过和延迟兑现。

## 3. 变量系统的修正版理解

本项目当前变量机制大致是：

```text
当前变量值 + 值描述提示词
        ↓
变量规则提示词
        ↓
AI 在回复中输出结构化变量更新块
        ↓
系统读取、校验并更新变量
```

因此变量分类不应按“听起来像什么”来分，而应按运行职责分：

- 谁能改
- 改动频率多高
- 是否进入提示词
- 是否能触发后续效果
- 是否允许 AI 提议更新
- 是否需要系统自动衰减或冷却

重要边界：

```text
变量 = 可读取的状态、事实、倾向或能力
变量值 = 该变量当前处于的数值、枚举或阶段
值描述提示词 = 这个值对角色表现意味着什么
更新规则提示词 = AI 如何判断是否提议变化
Trigger/Effect = 真正执行变化或动作的因果层
Action/Tool = 写信、送礼、弹窗、创建记忆等可见能力
```

这个机制的优点是让 AI 可以用角色理解参与变量更新；风险是 AI 同时成了演员、计分员和规则解释者。后续设计应尽量让 AI 输出“更新提案”，系统负责裁剪、校验、记录来源和执行副作用。

推荐变量更新块至少保留：

```text
变量名
旧值
新值或 delta
更新理由
来源事件/对话
变量类型
是否只是提案
```

变量分类的意义不只是整理文档，而是决定：

```text
更新权限
更新频率
提示词注入方式
副作用范围
是否允许 AI 提议
是否需要 Trigger/Effect 后续处理
```

## 4. 建议变量类型

### 4.1 世界变量

记录客观环境和舞台条件。

示例：

```text
date
time_of_day
weather
location
world_phase
```

作用：

- 限制角色能感知和能做的事
- 影响场景气氛
- 作为事件、Trigger、世界书激活条件

更新方式：

- 主要由系统、日历、时间流逝或用户显式设定更新
- AI 通常只读，不应随意改写

### 4.2 场景/模式变量

记录当前互动形态，与 ConversationMode 强相关。

示例：

```text
conversation_mode = chat / life / dream / letter / event
privacy_level = public / semi_private / private
interaction_surface = bubble_chat / longform / popup / inbox
```

作用：

- 决定提示词模板和注入位置
- 决定前端展示方式
- 限制可用动作通道
- 限制某些事件或数值规则是否生效

更新方式：

- 由用户切换、系统流程或事件推进更新
- 不建议由普通对话 AI 直接改

### 4.3 角色特质变量

记录相对稳定的人格、表达底色和习惯。

示例：

```text
directness
self_control
pride
empathy
curiosity
possessiveness
conflict_style
romantic_expression_style
```

作用：

- 作为状态生成和行为解释的底色
- 决定同一情绪在不同角色身上的表现差异

例如：

```text
同样是想道歉：
高 pride + 高 self_control -> 绕弯、写信、送小东西
低 self_control -> 当场爆发或直接追问
高 empathy -> 先照顾用户感受
```

更新方式：

- 低频更新
- 通常由长期事件、成长节点或技能树变化影响
- 不应每轮聊天漂移

### 4.4 关系变量

记录角色与用户或其他对象之间的长期关系结构。

示例：

```text
affection
trust
dependency
familiarity
tension
boundary_comfort
power_distance
public_reputation
expert_reputation
```

作用：

- 决定长期距离、开放程度、主动靠近概率
- 决定冲突后的修复方式
- 决定某些事件和主动表达通道是否可用

说明：

`public_reputation` 和 `expert_reputation` 建议分开。一个角色可能大众评价高但专业领域不认可，也可能百姓不理解但官员或同行敬畏。

更新方式：

- AI 可提议小幅变化
- 系统应限制步长、记录来源
- 重大变化应走事件或阶段节点

### 4.5 情绪变量

记录当前可连续波动的情绪仪表。

示例：

```text
joy
anxiety
irritation
sadness
shame
security
loneliness
expression_desire
defensiveness
```

作用：

- 影响语气、句长、敏感点、主动性
- 为状态变量提供底层数据

说明：

情绪变量负责“量”，不负责完整叙事解释。

例如：

```text
irritation = 70
self_control = 80
```

这组数值可以推导出或支持一个状态：

```text
emotional_posture = suppressed_irritation
```

情绪参数/心情饼图与状态变量的分工：

```text
情绪参数/心情饼图 = 底层情绪仪表盘
状态变量 = 对情绪的叙事化解释和行为约束
```

例如：

```text
irritation = 70
security = 35
expression_desire = 65
self_control = 85
```

可以支撑一个状态：

```text
emotional_posture = suppressed_irritation
```

它的描述不是简单重复数值，而是说明这组数值在该角色身上意味着什么。

更新方式：

- 高频变化
- 可由 AI 提议更新
- 系统可做衰减、回归基线、上下限裁剪

### 4.6 状态变量

状态变量记录短期叙事姿态，回答“角色今天/此刻怎么了”。

状态不是临时改人设，而是运行时姿态。

可以用三层比喻理解：

```text
设定是骨骼：角色是谁，核心性格和背景是什么
记忆是经历：角色如何携带过去
状态是此刻姿态：角色当前怎么运行
```

状态可以受记忆、事件、变量影响，也可以回写成事件或记忆，但它本身不应轻易改写底层设定。

建议做成“状态槽”，而不是无限创建布尔变量：

```text
emotional_posture
relationship_posture
action_posture
dream_residue
social_energy_state
```

示例值：

```text
emotional_posture =
  calm / suppressed_irritation / defensive / withdrawn / softened

relationship_posture =
  open / cautious / avoiding_reconciliation / seeking_reassurance

action_posture =
  idle / drafting_unsent_letter / preparing_gift / waiting_to_speak

dream_residue =
  none / faint / vivid / disturbing
```

作用：

- 把情绪、事件、特质翻译成角色可表演的短期姿态
- 影响回复风格、主动行为、工具倾向

状态值描述提示词示例：

```text
suppressed_irritation：
角色表面维持平静，但对被忽视、被敷衍更敏感。
减少直白示弱，增加短句、反问、回避或事后写信倾向。
```

更新方式：

- AI 可提议创建、切换、降低强度或结束
- 系统应要求来源、强度、有效期、退出条件
- 衰减和到期最好由规则或 Trigger/Effect 处理

可以概括为：

```text
角色变量回答“他是谁”。
状态变量回答“他今天怎么了”。
驱动变量回答“所以他倾向于做什么”。
```

### 4.7 驱动变量

驱动变量记录角色当前想做什么或避免什么，回答“所以角色倾向于做什么”。

示例：

```text
approach_user
repair_relationship
seek_reassurance
avoid_conflict
hide_vulnerability
create_surprise
protect_user
maintain_dignity
```

作用：

- 影响是否触发主动表达通道
- 影响事件分支
- 影响工具/action 的候选排序

说明：

`create_surprise = high` 是变量。

`create_gift` 是 Effect/action。

更新方式：

- AI 可提议变化
- 系统结合技能、冷却、权限、场景模式决定是否执行实际动作

### 4.8 能力/技能变量

记录角色表达或行动能力的成熟度，可与技能树系统对接。

示例：

```text
letter_expression_skill = none / basic / intimate / reflective
gift_preparation_skill = none / awkward / thoughtful
conflict_repair_skill = avoidant / basic / mature
memory_reflection_skill = shallow / associative / self_aware
delayed_expression_skill = none / simple / nuanced
```

作用：

- 限制角色能不能做某类行为
- 决定行为复杂度和质量
- 支持角色成长，而不是机械解锁按钮

说明：

技能树不应设计成“亲密度到 5 解锁写信”这种简单菜单，而应是角色逐渐学会更复杂地表达、克制、修复关系、保存记忆。

高级技能也可以是“学会不做什么”：

- 学会克制打扰
- 学会把话放到明天说
- 学会在用户累的时候只留下安静陪伴
- 学会把伤心写进未发送的信

技能树可以按角色存在方式分支，而不是按功能按钮分支：

```text
表达系：短消息、便签、信件、复杂道歉、含蓄告白
关系系：记住偏好、识别情绪、修复冲突、建立私人仪式
生活系：状态更新、日常片段、日程、准备礼物、维护个人空间
记忆系：简单回忆、关联旧事件、形成执念、创建记忆锚点
事件系：小事件、多阶段事件、延迟伏笔、长期剧情线
梦境/潜意识系：梦境碎片、梦境反映现实压力、梦里泄露白天不承认的东西
```

技能树的味道应接近“表达能力、关系能力、生活能力、叙事能力的成长轨迹”，而不是“功能解锁列表”。

更新方式：

- 不建议由 AI 每轮直接改
- 由成长事件、长期关系推进、技能树系统更新

### 4.9 Flag/进度变量

记录事实、伏笔、承诺、产物存在状态。

示例：

```text
promised_letter_tomorrow = true
unsent_letter_exists = true
conflict_unresolved = true
gift_prepared_not_sent = true
anniversary_seed_created = true
secret_known_by_user = true
```

作用：

- 给事件系统、Trigger/Effect 和延迟任务提供条件
- 让角色有时间感和承诺感

说明：

“明天再告诉你”不应只是一句台词，应留下 flag 或 scheduled event。

更新方式：

- 通常由事件、Effect、工具动作创建或消费
- AI 可提议，但应由规则或系统确认

### 4.10 统计变量

记录累积量。

示例：

```text
interaction_count
days_known
letters_sent_count
conflicts_resolved_count
gifts_created_count
dreams_shared_count
```

作用：

- 作为事件解锁条件
- 支持成长和技能树判断
- 支持长期关系阶段判断

更新方式：

- 主要由系统自动更新
- AI 不直接修改

## 5. 变量、值、动作的边界

本次讨论中特别需要避免的混淆：

```text
affection / trust / irritation
  更像变量名

suppressed_irritation / avoiding_reconciliation
  更像状态变量的值

write_letter / send_popup / create_gift
  更像动作、工具或 Effect，不是变量本体
```

判断原则：

```text
凡是“角色能做什么”，优先不是变量，而是 action/tool/channel。
凡是“角色当前倾向于怎么做”，才是变量或变量值。
凡是“这件事是否已经发生/存在”，是 flag。
凡是“角色做这件事的能力成熟度”，是技能变量。
```

## 6. Trigger/Effect 的角色

Trigger/Effect 是共享基础设施层，不是变量系统的一类。

它负责：

```text
当某条件满足时，执行某个效果。
```

变量系统和 Trigger/Effect 的关系：

```text
变量管道只读。
Trigger/Effect 只写。
两者通过数据库和子系统接口间接联系。
```

示例：

```text
变量：
emotional_posture = suppressed_irritation

Trigger：
该状态持续 3 轮

Effect：
创建 pending_injection “角色仍在压着火”
提高 repair_relationship 驱动
解锁事件“别扭的道歉”
```

如果没有 Trigger/Effect，状态只会影响提示词；有了 Trigger/Effect，状态才能产生后果。

Trigger/Effect 也适合承接“AI 按协议发出结构化意图，系统按规则执行”的模式。

例如：

```text
作者层：定义规则，声明 trigger 模式和 effect 动作
作者层：在提示词里告诉 AI 可以使用某种格式达成某种效果
AI 层：在恰当时机输出该格式
规则层：检测匹配，执行对应 Effect
```

AI 看似主动影响数据库，实际是在作者搭好的协议框架里说话。没声明的能力，AI 做不到；格式不匹配，规则不触发。

## 7. 事件系统与 Trigger/Effect 的关系

事件系统不是万能规则引擎，也不是最后一层。它更像剧情容器。

Trigger/Effect 是原子因果。

事件系统是剧情编排。

关系示例：

```text
普通规则 Trigger
  ↓
Effect：解锁事件 / 触发事件 / 创建事件实例
  ↓
事件进入 pending / active
  ↓
事件步骤自己的 Trigger 判断是否推进
  ↓
事件步骤自己的 Effect 执行具体后果
```

因此：

- Effect 可以触发事件
- 事件步骤内部也可以声明 Trigger/Effect
- 事件完成后也可以触发新的 Effect
- 事件触发事件需要 parent_event/source_rule 和循环保护

应避免无意识循环：

```text
事件 A completed -> 触发事件 B
事件 B completed -> 触发事件 A
```

除非它被明确设计为循环日常事件，并设置冷却。

## 8. AI 写状态与系统守门

AI 可以参与状态创建，但不应绕过系统直接写数据库或永久规则。

推荐方式：

```text
AI 输出状态提案
        ↓
系统校验
        ↓
写入变量/状态表
        ↓
Trigger/Effect 处理后续变化
```

状态提案应包含：

```text
state_slot
state_value
label
intensity
source_event
prompt_hint
exit_conditions
duration
decay_rule
reason
```

示例：

```json
{
  "state_slot": "emotional_posture",
  "state_value": "suppressed_irritation",
  "label": "压抑的烦躁",
  "intensity": 62,
  "source_event": "用户连续两次回避角色的认真提问",
  "prompt_hint": "说话仍然克制，但更容易使用短句、回避直接示弱。",
  "exit_conditions": ["用户认真回应", "角色独处到次日清晨", "角色写下未发送的道歉信"],
  "duration": "until_resolved_or_24h",
  "decay_rule": "每轮轻微下降，若再次被忽视则上升",
  "reason": "角色感到被敷衍，但由于自尊和自控没有直接爆发"
}
```

程序负责：

- 过期
- 衰减
- 冷却
- 最大值/最小值
- 互斥状态
- 来源记录

AI 负责：

- 命名
- 解释
- 叙事化状态
- 给出角色味道的 prompt hint
- 提议退出条件

可以概括为：

```text
程序管账，AI 管诗意。
```

Effect：展示信件入口或注入提示词

## 10. 对记忆系统的启发

本次讨论里，“记忆好”不等于“全知”。

对角色扮演来说，全知反而可能损害沉浸感。真实的关系连续性不是数据库式精确召回，而是有偏向、有遗漏、有执念、有误会、有突然想起。

一个角色如果每次都精准引用十天前第三句话，短期会惊艳，长期可能像后台检索系统在说话，而不是一个人延续到现在。

角色扮演的连续性更像：

- 有偏向地想起
- 有时遗忘
- 有时回避
- 有时误记
- 在恰当时刻忽然注意到某个旧点
- 让过去影响现在，但不把数据库检索感暴露给用户

因此记忆系统可被理解为：

```text
连续性调度系统
```

它关心的不只是存了什么，还包括：

- 什么时候该想起
- 想起多少
- 以什么情绪想起
- 想起后是否改变行为
- 什么时候不想起反而更像角色
- 什么内容沉淀为记忆，什么内容只作为状态消散

状态变量和 Trigger/Effect 可以为记忆系统提供入口：

```text
重大状态持续
  -> Trigger
  -> Effect 写入记忆候选
  -> 记忆系统决定是否巩固、扭曲、压缩或沉淀
```

记忆系统应和游戏性共存：

```text
稳定：角色核心连续，不每轮重生
新鲜感：同一关系状态下仍有不同表达路径
意外：角色逻辑内的偏航，而不是随机噪声
分寸：有些事想起，有些事不想起，有些事暂时压下去
```

文游经验给 ICS 的启发是：变量、事件、flag、隐藏条件本质上都是状态化叙事。但 AI 角色扮演多了一层生成不确定性，所以系统不只要记录状态，还要对生成过程施加叙事压力。

```text
变量不是为了展示数值，而是为了让角色有稳定倾向。
事件不是为了流水账，而是为了改变之后的可发生范围。
记忆不是为了全量召回，而是为了在某些时刻制造连续感。
遗忘不是失败，而是体验设计的一部分。
意外不是随机噪声，而是角色逻辑内的偏航。
```

## 12. 暂定原则

1. 变量系统不承担所有逻辑，只负责可读取状态。
2. 变量管道只读，不做副作用。
3. Trigger/Effect 负责自动化因果，不伪装 UI 显式操作。
4. 事件系统负责剧情编排，不重复实现通用规则引擎。
5. 动作、工具、渠道不是变量；倾向、能力、产物存在状态才进入变量或 flag。
6. AI 可以提议变化，但系统必须守门。
7. 角色鲜活不等于更吵、更主动、更全知，而是更有分寸、更有时间感、更能留下痕迹。
