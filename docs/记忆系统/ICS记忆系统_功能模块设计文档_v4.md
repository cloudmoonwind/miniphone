# A. 写入管线

**核心设计思路（2026-04-25 重构）**：

写入不是独立的后处理管线，而是**生成行为本身的一部分**。人不会攒一天的事情到晚上再统一记忆化——说了句话当时就记住了。因此写入分为两个阶段：

```
第一阶段：即时感知（生成时同步）
  角色生成内容的同时 → 输出 memory_draft
  → 附在原始文档后存入文档库（同步，即时）

第二阶段：整理入库（定期触发）
  从感知暂存区取出一批 draft
  → 去重合并 → zone 复核 → 补写 summary → 向量化 → 写入记忆库
```

**整理触发的三种模式**（用户可选切换）：
- 按轮次：每 N 轮对话后触发
- 按时间：每隔 X 分钟触发
- 线上/线下切换时触发（叙事节点，两种模式的 draft 性质不同，分批整理）

---

## A1. 即时感知层

**职责**：角色在生成回应的同时，判断「自己记住了什么」，以 memory_draft 的形式附加在输出内容后。这是角色的主观判断，不是系统筛选。

**触发时机**：每次角色生成内容时（对话回应 / 时间线轨迹生成 / 约会演绎 / 梦境生成后）。

**输入**：角色当前生成的内容 + 对话上下文

**输出**：
```
[角色回应正文]

<memory_draft>
[
  {
    "memory_text": "...",
    "emotion_tags_text": "...",
    "significance": 0.7,
    "zone": "episodic_short",
    "people_tags": ["NPC_小李"],
    "location_tag": "咖啡厅",
    "sensory_tags": {"smell": ["咖啡香"], "visual": ["窗边的光"]}
  }
]
</memory_draft>
```

**执行者**：AI（由提示词驱动，见提示词设计章节）

**字段说明**：

```
角色当场生成的字段（8-9 个）：
  memory_text       → 角色主观视角，当场最清晰
  emotion_tags_text → 角色对这件事的情绪词，当场粒度最细
                      （不从状态栏读取，因为状态栏是整轮混合值，
                       无法区分同一轮多条 draft 的各自情绪方向）
  significance      → 这件事对角色有多重要，当场有感觉
  zone              → 初步判断（整理时可复核）
  people_tags       → 涉及谁
  location_tag      → 在哪（写入时标准化粒度）
  sensory_tags      → 感官信息（只写确实存在的，不补全）

不在 draft 里的字段（程序或整理时填）：
  summary           → 整理时由 Agent 参照原文 + memory_text 生成
  summary_vector    → 整理时 embed
  emotion_vector    → 整理时 embed
  id / created_at / time / cluster_id 等程序字段 → 入库时自动填
  weight / clarity / distortion_bias 等初始值字段 → 硬编码默认值
  emotion_intensity → 整理时从状态栏 VAD 值读取（整体强度）
```

**可以输出 0 条**：没有值得记的内容时，memory_draft 为空数组，不强制输出。

**存储方式**：
```
原始文档（含 memory_draft）→ 立刻写入文档库（同步）
memory_draft 本身         → 同时写入「感知暂存区」（等待整理触发）
向量库                    → 异步，整理阶段完成后写入
```

**待解决**：
- A1-1：提示词设计（记忆说明 / 规则说明 / 格式说明 / 示例）——待专项讨论
- A1-2：memory_draft 的条数上限？防止 AI 把每句话都记一条
- A1-3：梦境生成后的 draft，zone 怎么判断？梦境记忆是否需要特殊处理？

## A2. 整理入库层

**职责**：从感知暂存区取出一批 memory_draft，整理后正式写入记忆库。

**触发时机**：三种模式之一触发时。

**整体流程**：

```
感知暂存区（一批 draft）
        ↓
A2-1. 去重合并器   批内去重 + 与记忆库去重 + 相关 draft 合并
        ↓
A2-2. Zone 判断器  复核每条 draft 的 zone，必要时修正
        ↓
A2-3. Summary 生成器  参照原文 + memory_text，生成客观 summary
        ↓
A2-4. 向量化器     embed summary + emotion_tags → 写入向量库
        ↓
        记忆条目正式入库，感知暂存区对应 draft 清除
```

---

### A2-1. 去重合并器

**职责**：处理三类重复情况——批内重复、与已有记忆重复、相关碎片合并。

**执行者**：向量相似度计算（快速，不调 AI）+ AI（合并时需要理解内容）

**三类处理**：

```
① 批内去重
   同一批 draft 中描述同一件事的多条 → 合并为一条
   （同一轮对话前后提到了同一件事）

② 跨批去重
   与记忆库已有条目高度相似 → 丢弃或追加 source_hints
   检索范围：近 7 天内（全库开销太高）
   相似度阈值：> 0.92 → 重复，丢弃
              0.75~0.92 → 关联，标记 derived_from

③ 碎片合并
   同一批中相关但不重复的 draft → 可合并为信息更完整的一条
   （「她说要出差了」+「她说会想我」→ 可保留两条，也可合并）
```

**待解决**：
- 碎片是否合并，判断标准是什么？（合并损失粒度，不合并条目更多）
- 跨批去重的时间窗口 7 天是否合适？

---

### A2-2. Zone 判断器

**职责**：复核每条 draft 的 zone，角色在即时生成时的判断不一定准确。

**执行者**：AI

**复核逻辑**：

```
角色写了 zone: episodic_short，但内容是对用户的稳定认知
→ 修正为 semantic

角色写了 zone: episodic_short，内容是跨对话的状态或计划
→ 修正为 working

行为模式类内容
→ 不在写入管线产出，由巩固管线（G3）生成 implicit 条目
```

**待解决**：
- zone 误判率有多高？是否值得每条都复核，还是只对特定类型的内容复核？

---

### A2-3. Summary 生成器

**职责**：为每条记忆生成客观 summary，用于向量检索。整理时生成而非即时生成，是因为：整理时有原文可参照，更容易保证客观性；角色在生成时处于情绪里，不适合同时要求第三方视角。

**执行者**：AI（严格客观性约束）

**输入**：
- memory_text（角色主观版本）
- 对应的原始文档片段（原文参照）

**输出**：summary（客观，第三方视角，50-150 字）

**客观性约束**：
- 只描述「发生了什么」
- 不描述「角色感受到什么」
- 不带情绪色彩词
- 具体保证方式待提示词设计时确定

**待解决**：
- summary 生成的提示词怎么写才能稳定保证客观性？

---

### A2-4. 向量化器

**职责**：对 summary 和 emotion_tags_text 做 embed，写入向量数据库。

**执行者**：程序（embed 模型）

**推荐模型**：`bge-m3`（本地部署，中文友好，延迟约 20-55ms）

```
embed(summary)           → summary_vector（768维）
embed(emotion_tags_text) → emotion_vector（768维）
写入向量数据库
```

同时填入所有程序字段（id / created_at / time / cluster_id / weight初始值 等）。

**待解决**：
- 本地 embed vs API embed 选型（成本 / 延迟 / 运维难度权衡）

# B. 检索管线

**触发时机**：每次对话轮次开始时，在 AI 生成回应之前执行。

**整体流程**：

```
对话开始
    ↓
B2. 被动注入器      working 区扫描 + implicit 区常驻 → 先注入上下文
    ↓
B1. 主动检索器      六模式并行检索 → 汇总层 → 选出最多 5 条
    ↓
B3. 重构器          对命中的每条记忆，注入当前状态参数，生成「此刻版本」
    ↓
最终注入 AI 上下文，生成本轮回应
```

---

## B1. 主动检索器

**职责**：用当前对话内容触发六种检索模式并行运行，找出此刻最应该浮现的记忆，经汇总层取舍后返回最多 5 条。同时维护连锁激活缓存区，影响后续轮次的检索权重。

**触发时机**：每轮对话开始时（被动注入器之后）。

**输入**：
- 当前对话内容（用户最新消息 + 近几轮对话摘要）
- 通道层输出（可选，感官/隐喻/身体感受的丰富化信号包）
- 当前角色情绪状态（从变量系统读取）
- 驱动核的 retrieval_preference（各模式权重系数）
- 连锁激活缓存区（上轮命中记忆列表，用于加权）

**输出**：最多 5 条记忆条目（含 memory_text + 相关字段），格式化后传给重构器

**执行者**：算法（向量检索 + NLP + 规则打分），不需要 AI 判断

### 六种检索模式

**① 语义检索（必跑，每次）**
- 触发：默认触发，无需特定信号
- 方式：对当前对话内容生成查询向量，与 summary_vector 做余弦相似度查询
- 返回：Top N 条（N 可配置，默认 10）

**② 情绪检索**
- 触发：当前对话中检测到情绪词或情绪状态变化
- 方式：对当前情绪词生成向量，与 emotion_vector 做余弦相似度查询
- 返回：Top N 条情绪相近的记忆

**③ 人物检索**
- 触发：当前对话中提到人名 / NPC 相关内容
- 方式：识别 NPC_ID，对 people_tags 做精确匹配
- 返回：所有包含该 NPC_ID 的记忆条目

**④ 时间检索**
- 触发：当前对话中出现时间词
- 方式：**统一走 NLP 工具解析**（HanLP / THULAC），无论明确或模糊时间词一律由 NLP 处理
  - 「上周三」→ 具体日期范围
  - 「前几天」→ 近3-5天范围
  - 「那年夏天」→ 模糊年份 + 6-8月范围
  - 输出标准化时间范围 → 对 time 字段做范围查询
- 返回：该时间范围内的记忆条目
- 设计原因：明确/模糊时间词的边界难以定义，统一走 NLP 逻辑更干净，不需要程序判断「要不要调 AI」

**⑤ 地点检索**
- 触发：当前对话中提到地点 / 场景（「咖啡厅」「家里」「医院」）
- 方式：标准化地点词 → 与 location_tag 精确匹配
- 返回：该地点的所有记忆条目

**⑥ 感官检索**
- 触发：当前对话中出现感官描述（「草木香」「某种声音」「凉意」）
- 方式（通道层关闭）：对 sensory_tags 精确匹配
- 方式（通道层开启）：通道层丰富化后的感官信号 → 精确匹配 sensory_tags
- 返回：包含对应感官标签的记忆条目

### 角色主动检索（Tool）

除系统自动触发的六种模式外，角色在生成回应时可以**主动调用记忆检索工具**：

```
触发场景：
  用户：「你还记得我上次说的那件事吗？」
  角色：[调用 memory_search tool，query="用户上次说的事"]
       → 拿到检索结果再生成回应

工具定义（自定义 tool）：
  name: memory_search
  description: 主动检索角色记忆库，用于角色想起某件事时调用
  parameters:
    query: string（检索关键词，角色用自然语言描述想找的记忆）
    zone: string（可选，限定检索范围：episodic/semantic/working）
```

主动检索走语义检索模式，返回 Top 3 条，不经过汇总层的 5 条限制（是额外的，不占名额）。

### 汇总层

六个模式各自返回候选记忆，汇总层负责：去重 → 打分 → 取舍 → 排布。

**综合打分公式（草案，参数待测试调参）**：

```
最终分数 = 语义相似度 × retrieval_preference.semantic_weight
         + 情绪相似度 × retrieval_preference.emotion_weight
         + 命中模式数量 × 0.1          ← 多维共鸣加分，每多命中一个模式+0.1
         + 记忆 weight × 0.2           ← 重要性权重
         - 时间衰减项                  ← 形状待定，草案用指数衰减
         + 缓存区加分                  ← 在连锁激活缓存区里的记忆+0.15
                                          不修改 weight 字段本身，只在打分时加
```

各模式权重系数从驱动核的 retrieval_preference 动态读取。

**取舍规则（5 条上限）**：

```
强相关：综合分数最高的前 3 条
重要性保底：weight 最高的 1 条（若已在前3条中则顺延到第4高分）
近期保底：time 最近的 1 条（若已在前4条中则顺延）

去掉：
→ 和已选条目 summary_vector 相似度 > 0.9 的（内容高度重叠，保留分高的）
→ weight < 0.1 且综合分数也极低的（已高度遗忘，没必要注入）
```

**注入排布格式**：

```
以下是角色此刻可能浮现的记忆（按相关程度排列）：
1. [memory_text]（情绪：xxx，时间：xxx，清晰度：xx%）
2. [memory_text]（情绪：xxx，重要）
3. [memory_text]（情绪：xxx，最近）

角色不需要全部提及，自然地融入对话即可。
```

### 连锁激活缓存区

**职责**：存储最近几轮命中的记忆，对下一轮检索的相关记忆进行打分加权，模拟联想涟漪效果。

```
机制：
  每轮检索命中的记忆 → 写入缓存区
  缓存区容量：最近 3 轮的命中记忆（滚动丢弃旧的）
  对话结束后：清空，不持久化

加权方式：
  打分时检查候选记忆是否在缓存区里 → 是则 +0.15（缓存区加分）
  不修改 weight 字段本身，只在本次打分临时生效
  缓存清空后完全无副作用

受驱动核 attachment_style 影响：
  安全型  → 加分系数正常（+0.15）
  焦虑型  → 加分系数提高（+0.25），连锁更强，相关记忆更容易涌出
  回避型  → 加分变扣分（-0.1），相关记忆反而被压制，话题自然漂移
```

**深度自然限制**：缓存区最多保留 3 轮，超出后滚动丢弃，连锁不会无限展开。

## B2. 被动注入器

**职责**：在主动检索之前，先将 working 区和 implicit 区的内容注入上下文。这两类内容不走向量检索，有各自的注入逻辑。

**触发时机**：每轮对话最开始，早于主动检索。

**执行者**：程序（规则判断，不需要 AI）

### Working 区注入

**逻辑**：

```
扫描所有 zone=working 且 status=active 的条目
→ 按 expires_at 检查是否到期，到期则将 status 改为 expired（不删除）
→ 对仍然 active 的条目，检查 resolve_condition：
    · keyword 类型：扫描当前对话内容是否出现关键词
    · time 类型：程序直接比较当前时间
    · ai_judge 类型：调用 AI 判断（仅此类型需要 AI，且开销最高）
    · manual 类型：等待用户操作，本次跳过
→ 条件满足的条目 status 改为 resolved（不删除）
→ 剩余 active 条目按 priority 排序，全部注入上下文
```

**注入位置**：注入到上下文的「角色当前状态」区域，早于记忆区域。

### Implicit 区注入

**逻辑**：

```
Implicit 区条目常驻角色配置层（prompt 级别），不需要每轮扫描
每次对话开始时，implicit 区已经存在于角色系统 prompt 中
格式：如果 [trigger_condition]，倾向于 [behavior_response]
```

**与 working 区的区别**：
- working 区是「此刻需要知道的跨对话状态」，每轮扫描，有时效性
- implicit 区是「角色骨子里的行为倾向」，永远存在，没有时效性

**待解决**：
- implicit 区条目数量上限？如果积累过多，prompt 会膨胀
- implicit 条目之间是否会冲突（两条 trigger_condition 相近但 behavior_response 相反）？冲突怎么解决？

## B3. 重构器

**职责**：对主动检索命中的记忆，结合当前情绪状态和清晰度，生成「此刻角色想起的版本」，用于本轮注入，不立刻写回原始记忆。

**触发时机**：B1 汇总层输出之后，注入 AI 上下文之前。

**触发条件**：由「记忆设置」参数控制（见下方），不是每条命中的记忆都触发。

**输入**：
- B1 输出的记忆条目（memory_text + 相关字段）
- 当前角色情绪状态（从变量系统读取）
- 当前对话场景描述
- 记忆的 distortion_bias（已有扭曲偏向）
- 记忆的 detail_clarity / emotion_clarity

**输出**：重构后的「此刻版本」memory_text，传给注入层使用

**执行者**：小模型（由记忆设置参数决定）

### 记忆设置参数（系统设置 → 记忆设置）

```
重构触发阈值
  · detail_clarity 低于此值时才触发重构
  · 默认 0.6，范围 0-1
  · 设为 0 = 关闭重构（所有记忆直接用原始 memory_text）
  · 设为 1 = 所有命中的记忆都重构

重构模型选择
  · 主模型（效果最好，开销最高）
  · 小模型（便宜或免费，效果稍差）← 默认推荐
  · 关闭（不重构，直接注入原始 memory_text，最省）

小模型选项
  · 取决于部署方式（本地 / API），具体选型待定
```

### 重构方向逻辑

```
detail_clarity 很低（低于触发阈值）
  → 细节模糊处理：省略具体细节，只保留情绪轮廓
  → 例："那次她好像等了我很久，具体的细节已经有点模糊了"

emotion_clarity 很低（< 0.3）
  → 情绪也淡化：连情绪轮廓也不确定
  → 例："那次好像发生了什么，但感觉已经很遥远了"

当前角色情绪与记忆情绪共鸣
  → 放大共鸣方向的细节，压制相反方向

distortion_bias 包含方向
  → 重构结果往该方向漂移

reconstructed_count 很高
  → 内容更「故事化」，叙述更流畅但更不像原版
```

**重要设计决策：重构结果不立刻写回**：
- 重构是根据「此刻情绪」生成的，带有时效性
- 如果立刻写回，一次偶然的心情低落就永久改写了记忆
- 写回逻辑在 E1 处理（对话结束后判断）

# C. 激活后处理

**触发时机**：B1 主动检索完成、角色回复生成后。

**说明**：检索本身会改变记忆系统的状态。但「被检索到」和「被角色真正用上」是两件事——注入了上下文但角色回复里完全没有体现，不算真正激活，不应该压制同簇记忆，下次也不应该因此获得权重提升。

**真正命中的判断方式（方案B，程序近似判断）**：
```
对每条注入记忆，计算其 summary 与角色本轮回复的语义相似度
相似度 > 0.6 → 真正命中（角色用上了这条记忆）
相似度 ≤ 0.6 → 注入未命中（角色没有用上）

判断在角色回复生成后执行，不影响生成流程
```

---

## C1. 激活统计更新器

**职责**：根据真正命中 / 注入未命中的判断结果，分别更新记忆的激活统计字段和权重。

**触发时机**：角色本轮回复生成后，语义相似度判断完成后。

**输入**：
- 本轮注入的记忆条目列表
- 每条记忆与角色回复的语义相似度判断结果

**输出**：各记忆条目字段更新（写回数据库）

**执行者**：程序（纯数值更新）

**字段更新逻辑**：

```
【真正命中的记忆（相似度 > 0.6）】
activation_count          → +1
last_activated_at         → 当前时间戳
recent_activation_density → 重新计算（过去 30 天激活次数 ÷ 30）
weight                    → 不直接修改，通过 recent_activation_density
                            上升在 D1 衰减公式中自然体现

【注入但未命中的记忆（相似度 ≤ 0.6）】
weight                    → × (1 - 未命中惩罚系数)
                            惩罚系数默认 0.02，可在记忆设置中调整（范围 0-0.1）
                            轻微下调，不是惩罚，只是「这次没用上」的自然反馈
activation_count 等字段   → 不更新

【未被注入的记忆】
→ 不做任何更新
```

**未命中惩罚系数说明**：
- 默认 0.02（每次未命中 weight 下调 2%）
- 设为 0 = 关闭未命中惩罚
- 放入系统设置 → 记忆设置，和重构阈值同一位置
- 下调幅度故意设得很小——未命中不代表这条记忆不好，可能只是这次话题不需要

---

## C2. 连锁激活器

**职责**：将本轮真正命中的记忆写入连锁激活缓存区，影响下一轮检索的打分权重。

**触发时机**：C1 判断完成后。

**输入**：本轮真正命中的记忆条目（已通过相似度判断）

**输出**：更新连锁激活缓存区

**执行者**：程序

**缓存区机制**（与 B1 打分公式联动）：

```
结构：最近 3 轮真正命中记忆的列表（滚动丢弃最旧的一轮）
生命周期：对话内有效，对话结束后清空，不持久化

下一轮 B1 打分时：
  候选记忆在缓存区里 → 打分 +0.15（缓存区加分）
  候选记忆不在缓存区 → 不加分

不修改 weight 字段本身，只在打分时临时生效
```

**只有真正命中才进缓存**：
- 注入未命中的记忆不写入缓存区
- 避免「没用上的记忆」持续占据连锁位置，影响下轮检索

**受 attachment_style 影响**（驱动核）：
```
安全型  → 缓存区加分系数 +0.15（默认）
焦虑型  → 缓存区加分系数 +0.25，连锁更强
回避型  → 缓存区加分变扣分 -0.1，相关记忆被压制，话题自然漂移
```

---

## C3. RIF 压制器

**职责**：对本轮真正命中记忆的同簇（cluster_id 相同）其他记忆，执行 suppression_count +1，模拟 RIF（提取诱发遗忘）。

**触发时机**：C1 判断完成后，与 C2 并行。

**触发条件**：只对**真正命中**的记忆触发压制，注入未命中的不压制同簇记忆。

**输入**：本轮真正命中的记忆条目（含 cluster_id）

**输出**：同簇其他记忆的 suppression_count +1

**执行者**：程序（规则，不需要 AI）

**执行逻辑**：

```
对每条真正命中的记忆：
  取出其 cluster_id 列表（如：["咖啡厅", "NPC_小李"]）
  对每个 cluster_tag：
    查找 cluster_id 包含该 tag 的所有其他记忆（排除本次命中的）
    这些记忆的 suppression_count +1
```

**压制范围**：共享任意一个 cluster_tag 即触发压制（不要求共享全部 tag）。
- 理由：只要争夺同一个检索线索就算竞争，不需要完全重叠
- 范围过大的风险由 cluster_id 的粒度控制（写入时已标准化）

**suppression_count 的作用**：
- D1 衰减计算器的输入之一
- 被频繁压制的记忆 weight 衰减更快，最终形成「选择性遗忘」

# D. 演变管线

**触发时机**：每天定时任务（离线运行，不在对话流程中）。

**整体流程**：

```
每天定时触发
   ↓
D1. 衰减计算器      更新所有条目的 weight / detail_clarity / emotion_clarity
   ↓
D2. 扭曲漂移器      对符合条件的条目，自然推进 distortion_bias
```

D3 来源混淆器已移除——来源混淆效果由 detail_clarity 衰减隐式表达，AI 在 detail_clarity 低时自然呈现「记不清是什么时候的事」，不需要单独模块。

---

## D1. 衰减计算器

**职责**：按幂律公式更新所有记忆条目的 weight、detail_clarity、emotion_clarity 三个动态字段，模拟记忆随时间自然遗忘。

**触发时机**：每天定时任务。

**输入**：全库记忆条目（逐条处理）

**输出**：三个动态字段的新值（写回数据库）

**执行者**：程序（数值计算，无 AI）

**衰减曲线形状：幂律**

选用幂律而非指数或线性，依据 Anderson & Schooler（1991）真实人类记忆研究：
- 遗忘在最初很快，之后极度减缓
- 重要记忆和普通记忆的衰减速率不同（由 significance 控制 d 值）
- 永远有残留，不归零——「刻骨铭心的事会留一辈子轮廓」
- 频繁激活使记忆更稳固，但不「刷新」成新的（修改 d 值而非重置 t）

---

### weight 衰减

```
d_effective = base_d
            × (1 - significance × 0.5)
            × (1 - min(recent_activation_density × 0.3, 0.4))
            × (1 + min(suppression_count × 0.05, 0.3))
            × (1 - absorption_level × 0.3)

weight(t) = weight_0 × t^(-d_effective)

参数说明：
  base_d                  → 记忆设置参数，默认 0.5
  t                       → 距 last_activated_at 的天数
  significance × 0.5      → significance 越高，d 越小，衰减越慢
  activation_density 上限 → 0.4，防止频繁激活导致几乎不衰减
  suppression 上限         → 0.3，防止压制过狠
  absorption_level × 0.3  → 被上层吸收越多，weight 越低

下限：0.05（不归零，永远有残留）

zone 差异（通过 base_d 体现）：
  semantic 区     → base_d × 0.2（极慢衰减，语义认知几乎不遗忘）
  episodic_long   → base_d × 1.0（正常衰减）
  episodic_short  → base_d × 2.0（未巩固时快速消退）
```

---

### detail_clarity 衰减

```
emotion_processed_factor =
  · 当 emotion_processed < 0.2 且 emotion_intensity > 0.8（闪光灯效应）
    → factor = 0.5，衰减减慢，细节反而清晰烙印
  · 当 emotion_processed < 0.5（未处理）
    → factor = 1 + (1 - emotion_processed) × (1 - emotion_intensity) × 0.3
    → 未处理且强度高 → 保护性压抑 → 衰减加速
  · 当 emotion_processed >= 0.5（已处理）
    → factor = 1 - (emotion_processed - 0.5) × 0.2
    → 情绪处理越深，细节反而被好好保存，衰减略减慢

d_detail = base_d_detail
         × (1 + reconstructed_count × 0.1)
         × emotion_processed_factor

detail_clarity(t) = detail_clarity_0 × t^(-d_detail)

参数说明：
  base_d_detail           → 记忆设置参数，默认 0.5
  reconstructed_count     → 重构越多，细节磨损越快
  emotion_processed_factor → 三种情绪处理状态对应三种衰减方式

下限：0（细节可以完全模糊）
```

---

### emotion_clarity 衰减

```
d_emotion = base_d_emotion
          × (1 + emotion_processed × 0.4)

emotion_clarity(t) = emotion_clarity_0 × t^(-d_emotion)

参数说明：
  base_d_emotion          → 记忆设置参数，默认 0.2（基础比 detail 慢）
  emotion_processed × 0.4 → emotion_processed 渐进升高时，d_emotion 渐进增大
                            情绪越沉淀，emotion_clarity 衰减越快
                            不是「忘了这件情绪」，而是「那件事的感觉已经成为过去了」

下限：0.05（情绪轮廓极难完全消失）
```

---

### 三条曲线关系总结

```
正常情况：
  detail_clarity 衰减最快（base_d_detail 默认 0.5）
  emotion_clarity 衰减慢（base_d_emotion 默认 0.2）
  weight 居中

闪光灯效应（emotion_intensity > 0.8 且 emotion_processed < 0.2）：
  detail_clarity 衰减反而减慢
  → 细节和情绪一起留存，「那一刻记得非常清楚」

保护性压抑（emotion_processed 低 + emotion_intensity 高但不到闪光灯阈值）：
  detail_clarity 衰减加速
  emotion_clarity 仍慢衰减
  → 「不记得细节，但情绪还在」

情绪沉淀（emotion_processed 渐进升高）：
  emotion_clarity 的 d 渐进增大，开始追上并超过 detail_clarity
  → 「记得发生了什么，但那种感觉已经很远了」
```

---

### 记忆设置参数（D1 相关）

```
base_d          weight 基础衰减率        默认 0.5
base_d_detail   细节清晰度基础衰减率     默认 0.5
base_d_emotion  情绪清晰度基础衰减率     默认 0.2
```

---

## D2. 扭曲漂移器

**职责**：对满足条件的记忆条目，自然推进 distortion_bias，模拟记忆随时间慢慢变形。

**触发时机**：每天定时任务（D1 之后运行）。

**输入**：全库记忆条目（只处理满足触发条件的子集）

**输出**：distortion_bias 字段更新

**执行者**：程序（触发判断 + 强度计算），不需要 AI

**触发条件**（满足任一即触发）：

```
1. 时间 + 模糊
   距事件 > 30天 且 detail_clarity < 0.6
   → 时间够久 + 细节够模糊，记忆容易漂移

2. 情绪未消化
   emotion_processed < 0.3 且 emotion_intensity > 0.7
   → 强烈且未处理的情绪，倾向往某个方向固化

3. 重构积累
   reconstructed_count > 5
   → 被反复重构，版本越来越「故事化」，越来越固化
```

**方向判断**（由记忆性质 + 驱动核决定，不依赖当前情绪状态）：

```
美化（idealization）
  → 回避型角色 + 痛苦记忆 → 倾向美化（让分离显得合理）
  → 时间久远的情感记忆默认倾向美化

丑化（devaluation）
  → 焦虑型角色 + 冲突记忆 → 倾向丑化

自我服务偏差（self-serving bias）
  → 焦虑型角色 + 任何涉及冲突的记忆
  → 记忆朝「让自己更好看」的方向漂

叙事一致性（narrative consistency）
  → 角色当前自我认知（核心语义记忆）与该条记忆矛盾时触发
  → 记忆朝「符合角色自我叙事」的方向漂
```

**多方向并存**：允许，各自独立积累强度，全部传给重构器由 AI 处理拉扯。

```
distortion_bias 结构：
[
  {"direction": "美化", "strength": 0.4},
  {"direction": "自我服务", "strength": 0.6}
]

每次触发强度增量：+0.05~0.1（渐进，不突变）
单方向强度上限：0.9
多方向强度之和：不设上限，交给重构器 AI 处理
```

**与重构器的分工**：
```
D2 扭曲漂移器  → 只更新 distortion_bias 字段（方向和强度）
               → 不直接修改 memory_text
B3 重构器      → 读取 distortion_bias，在生成「此刻版本」时体现扭曲方向
E1 写回判断器  → 偏差够大时才把重构结果写回 memory_text
```

# E. 写回判断器

**触发时机**：每次对话结束后。

---

## E1. 重构写回器

**职责**：对话结束后，判断本次 B3 重构器生成的「此刻版本」是否与原始 memory_text 偏差足够大——足够大则写回，否则不留痕迹。

**触发时机**：对话结束后触发一次，处理本次对话中被重构的所有记忆。

**输入**：
- 本次对话中 B3 生成的所有「此刻版本」
- 对应的原始 memory_text

**输出**：
- 超过偏差阈值的条目：memory_text 更新，reconstructed_count +1，last_modified_at 更新
- 未超过的条目：不写回，本次重构不留痕迹

**执行者**：程序（向量相似度计算）

**流程**：

```
对每条被重构的记忆：
  计算「此刻版本」与原始 memory_text 的余弦相似度
  如果相似度 < (1 - 偏差阈值)：
    → 偏差够大，写回 memory_text
    → reconstructed_count +1
    → last_modified_at 更新
  否则：
    → 不写回

同一条记忆在一次对话中被多次重构时：
  → 取偏差最大的那次写回（最能反映真实演变方向的版本）
```

**偏差阈值**：做成记忆设置参数，默认值待测试后定。

# F. 社会传染处理器

**核心设计思路（重构）**：

社会传染不是系统检测触发的，而是**角色主动感知并决定是否接受**的。通过两个 tool 实现：memory_search（已有）+ memory_update（新增）。角色自己判断「对方说的和我记住的不一样，我要不要更新我的记忆」。

---

## F1. 社会传染流程

**触发时机**：角色在生成回应时，感知到用户/NPC 在描述某段过去的事。

**执行者**：角色 AI（主动调用 tool）

**流程**：

```
用户说了关于角色过去的话（如「你那天看起来很难过」）
        ↓
角色感知到「这在描述我的某段记忆」
        ↓
角色主动调用 memory_search tool 找到对应记忆
        ↓
系统在检索结果后附加社会传染提示词：
  「你刚找到的记忆，和对方刚才的描述可能有出入。
   对比一下你记住的版本和对方说的版本，
   你觉得对方说的更接近真实发生的事吗？
   如果你认为自己记错了，可以调用 memory_update 更新这条记忆。」
        ↓
角色自己判断：

  觉得对方说得对
  → 主动调用 memory_update，写入新版本
  → distortion_bias 标注来源：{"direction": "社会传染", "strength": 初始值}
  → trust_level 越高、detail_clarity 越低，角色越可能接受

  觉得对方说错了
  → 不调用 memory_update，保持原记忆
  → 可能在对话里反驳「不对，我记得我那天不是难过，是生气」
  → 原记忆被强化（C1 激活统计更新）

  半信半疑
  → 可以调用 memory_update，但 strength 较低
  → memory_text 只做小幅修改
```

---

## F2. memory_update Tool 定义

**职责**：角色主动修改某条记忆的 memory_text，通用于社会传染和角色自主修正。

```
tool 定义：
  name: memory_update
  description: 当你意识到某条记忆需要修正时调用。
               可以是因为对方的描述让你觉得自己记错了，
               也可以是你自己想到了更准确的版本。
  parameters:
    memory_id: string（要修改的记忆条目 ID）
    new_memory_text: string（修正后的 memory_text）
    reason: string（为什么修改，可选）
              "社会传染" / "自主修正" / "重新想起"
```

**触发来源**：
- 社会传染：用户/NPC 描述与记忆不符时
- 自主修正：角色自己「想起来不对」时（不依赖外部描述）

**写入逻辑**：
```
调用 memory_update
→ memory_text 更新为 new_memory_text
→ reconstructed_count +1
→ last_modified_at 更新
→ 如果 reason = "社会传染"：
    distortion_bias 追加 {"direction": "社会传染", "strength": 0.2}
    （后续 D2 漂移时会考虑这个来源）
```

**与 E1 写回判断器的区别**：
- E1 是系统在对话结束后判断要不要写回重构结果
- memory_update 是角色在对话中主动发起的修改，立刻生效，不需要偏差判断

# G. 巩固管线

**触发时机**：每日「睡眠事件」——ICS 时间线中角色进入休息状态时触发（模拟海马体离线巩固）。

**说明**：巩固是记忆在层级之间「生长」的核心机制。遵循「生长而非搬家」原则：原条目不变，新认知以新条目形式生长出来，通过 derived_from 和 absorbed_by 建立关联。

---

## G1. 情节巩固器

**职责**：将短期情节区（episodic_short）中的相似记忆合并，生长出长期情节记忆（episodic_long）。模拟睡眠时海马体将日间记忆压入皮层的过程。

**触发时机**：每日睡眠事件。

**输入**：近期 episodic_short 条目（按时间窗口筛选）

**输出**：
- 新的 episodic_long 条目（derived_from 包含源条目 ID）
- 原 episodic_short 条目打上 absorbed_by 标记，absorption_level 更新

**执行者**：AI Agent（需要理解多条记忆、判断相似性、生成合并版本）

**触发条件（草案）**：

```
满足以下条件的 episodic_short 条目群，触发合并生长：

1. 时间窗口：距今 1~3 天内（太近的可能还在当天，太远的已经处理过了）
2. 相似度阈值：两两之间 summary_vector 相似度 > 0.75
3. 最少条数：至少 2 条（1 条无法合并）
4. weight 阈值：至少 1 条 weight > 0.4（太低说明本来就不重要）
```

**合并生长逻辑**：

```
AI 读取满足条件的 episodic_short 条目群
→ 综合生成新的 episodic_long 条目：
   · summary：融合多条事件的客观描述
   · memory_text：角色视角，保留情绪轮廓，细节自然融合
   · emotion_tags：取并集或最高强度
   · time：取事件群的时间中点或范围
   · significance：取来源条目中最高的那个
   · zone：episodic_long（固定）
   · derived_from：所有源条目的 ID 列表
→ 原 episodic_short 条目：
   · absorbed_by 追加新条目 ID
   · absorption_level 更新
   · weight 随 absorption_level 上升而自动下降（由 D1 衰减器处理）
```

**「日内压缩」的问题**（来自旧文档的遗留疑问）：
- 同一天内如果发生了大量相关事件，是先做日内压缩（short → 临时合并），还是等晚上睡眠时一起处理？
- 如果做日内压缩，新生成的条目 zone 是 short 还是 long？（绕过 CLS 理论的高可塑性机制？）
- 暂定：日内不压缩，所有事件作为独立 short 条目存在，等每日睡眠时统一处理。

**待解决**：
- G1-1：相似度阈值 0.75 是否合适？太高会导致大量 short 条目无法巩固；太低会把不相关的事件合并
- G1-2：生长出的 episodic_long 条目，detail_clarity 的初始值怎么定？（应该比原 short 条目低，因为合并时有信息损耗）
- G1-3：如果某条 episodic_short 条目非常独特（无法与其他条目合并），它就一直留在 short 区吗？留多久？

---

## G2. 语义蒸发器

**职责**：当大量情节记忆反复出现某个共同的主题或认知，从中提炼出语义记忆（semantic zone）。模拟情节记忆蒸发为稳定知识的过程。

**触发时机**：每日睡眠事件（在 G1 之后运行）。

**输入**：episodic_long 条目（按 cluster_id / people_tags / 情绪共性筛选候选群）

**输出**：
- 新的 semantic 条目（derived_from 包含源情节记忆 ID）
- 源情节记忆条目打上 absorbed_by 标记

**执行者**：AI Agent

**两条路径的分工**：

```
单次路径（高 significance + 内容本身是稳定认知）
  → 在 A2-2 Zone 判断器里处理
  → 整理时 AI 直接判定 zone = semantic，当场入库
  → 不需要等积累，一次就能形成
  → 例：「我其实挺怕黑的，从小就这样」→ 角色立刻形成「她是个会怕黑的人」

频率路径（多条相似情节积累）
  → G2 蒸发器处理
  → 需要足够的重复次数和时间跨度
  → 例：多次等待、多次委屈、多次观察，积累出「她不擅长表达委屈」
```

**触发条件（频率路径）**：

```
AI Agent 自主判断以下维度，不做硬编码阈值：
· 候选情节群是否有足够的重复性和共同主题
· 时间跨度是否足够长（不是短期密集发生）
· 是否已有高度重叠的 semantic 条目（避免重复蒸发）
· 认知反转情况：已有 semantic 与新情节矛盾时，Agent 判断是更新旧条目还是生长新条目
```

**蒸发生长逻辑**：

```
AI Agent 读取候选情节群
→ 提炼共同认知，生成新的 semantic 条目：
   · summary：客观陈述（第三方视角）
   · memory_text：角色对该认知的内在表达
   · zone：semantic（固定）
   · time：null，location_tag：null
   · significance：根据情节群综合评分
   · derived_from：所有源情节 ID 列表
→ 源情节打 absorbed_by 标记，absorption_level 更新
```

## G3. 习惯化器

**职责**：识别角色的行为模式，将反复出现的行为倾向生长为 implicit 条目。

**触发时机**：每日睡眠事件（在 G2 之后运行）。

**输入**：近期情节记忆 + 对话行为记录

**输出**：
- 新的 implicit 条目（trigger_condition + behavior_response 格式）
- 或对已有 implicit 条目的强化 / 修正

**执行者**：AI Agent

**设计思路**：

```
Agent 自主判断行为模式，不做硬编码阈值
就像人不会数「我重复了几次才算习惯」，
Agent 读取多条记忆后自己判断「这个行为模式够稳定了，值得写成 implicit」

生成格式：
  trigger_condition（内隐记忆触发条件）：自然语言，描述情境
  behavior_response（行为反应）：自然语言，描述倾向

与语义蒸发的区别：
  G2 提炼对「世界/他人」的认知（「她不擅长表达委屈」）
  G3 提炼角色自身的行为倾向（「遇到这种情况我会这么做」）

implicit 条目失效：
  Agent 同时负责检查已有 implicit 条目
  如果近期行为记录与某条 implicit 矛盾 → Agent 判断是修正还是移除
```

## G4. 核心认知更新 Agent

**职责**：定期读取最新积累的语义记忆和情节记忆，修订角色的「核心认知」——对自我的认知 + 对用户的认知。核心认知被标记为 is_core=true，常驻注入，不走检索。

**触发时机**：每周一次。

**输入**：
- 近期新增的 semantic 条目
- 近期高 weight 的 episodic_long 条目
- 现有的 is_core=true 语义记忆条目（用于对比）

**输出**：更新 / 新增 is_core=true 的 semantic 条目

**执行者**：AI Agent

**is_core 字段**：已正式加入记忆条目标准字段（boolean，默认 false，G4 Agent 写入）

**核心认知的两类内容**：

```
自我认知（极慢更新，需要强烈事件才触发修订）
  · "我是一个会在沉默中扛着的人"
  · "我很在乎她，但不知道怎么表达"

用户模型（相对敏感，用户在变化，认知也跟着更新）
  · "她习惯用工作逃避不安"
  · "她不喜欢被主动安慰，但喜欢被记得"
```

**更新逻辑**：

```
Agent 读取近期语义记忆和重要情节
→ 判断核心认知是否需要修订
→ 修订方式：
   · 小幅修订：直接更新原条目 memory_text
   · 认知反转：生长出新条目，旧条目打 absorbed 标记
→ 控制数量：建议 10-20 条以内，Agent 自主清理低价值的旧核心认知
→ 修订结果写入角色常驻注入层
```

# H. 特殊激活

**说明**：两类激活不走主动检索管线，是记忆「主动冒出来」的特殊情况。两者都是 B1 五条之外的额外插入，不占名额，最多同时注入 6 条（5条正常 + 1条特殊）。

**优先级**：H2 优先——H2 触发时 H1 跳过。闯入性记忆是「压不住的」，既然已经有一条强制插入，再加随机的「突然想起」会显得混乱。

---

## H1. 不随意记忆触发器

**职责**：模拟无关联记忆的偶发性浮现，低概率触发，给记忆系统注入随机性和真实感。

**触发时机**：每轮对话 B1 之后，H2 未触发时才运行。

**执行者**：程序（概率采样 + 规则筛选）

**触发逻辑**：

```
以概率 P_involuntary 触发（默认 5-10%，可配置）

从以下候选池加权随机采样 1 条：
  · weight > 0.3
  · detail_clarity > 0.2
  · 与本轮已检索的 5 条相似度低（真的「无关联」）
  · 近期未被激活

采样方式：按 weight 加权随机
```

**attachment_style 影响**：
```
焦虑型  → P_involuntary 更高（记忆更容易随机涌出）
回避型  → P_involuntary 更低（倾向于不让无关记忆干扰）
安全型  → 默认概率
```

**注入标注**：标注为「突然想起」，告诉角色这条记忆是偶然浮现而非直接相关。

---

## H2. 闯入性记忆触发器

**职责**：模拟强烈未处理情绪的强制闯入，条件触发，优先于 H1。

**触发时机**：每轮对话 B1 之后，条件满足时强制触发。

**执行者**：程序（条件判断 + 规则筛选）

**触发条件**（同时满足）：

```
1. 候选记忆存在：
   emotion_intensity > 0.8 且 emotion_processed < 0.2

2. 触发刺激存在（满足其一）：
   · 当前对话内容与该记忆的 cluster_id 有重叠
   · 当前对话中出现了该记忆的 sensory_tags
   · 当前角色情绪与该记忆的 emotion_vector 高度相似

3. 当前角色情绪强度高于阈值（处于激动状态时更容易闯入）
```

**触发后**：
```
emotion_processed 小幅上升（每次被强制面对，情绪处理程度略微提升）
注入标注：标注为「闯入性浮现」，告诉角色这条记忆是压不住冒出来的
```

**与 H1 的区别**：

| | H1 不随意记忆 | H2 闯入性记忆 |
|--|--|--|
| 触发方式 | 低概率随机 | 条件满足时必然触发 |
| 内容来源 | 任意 weight > 0.3 | 高强度未处理情绪的记忆 |
| 与话题关联 | 刻意选择无关联的 | 由相关刺激触发 |
| 角色感受 | 突然想起，有点意外 | 压不住，被迫面对 |
| 优先级 | 低（H2 触发时跳过） | 高 |

# I. 驱动核管理

**说明**：驱动核不是记忆库中的条目，而是角色的元数据——它决定记忆系统「怎么运作」，而不是「记忆了什么」。

---

## I1. 初始化器

**职责**：角色创建时，根据角色描述（自然语言设定）生成驱动核的初始值。

**触发时机**：一次性，角色创建时触发。

**输入**：角色的自然语言设定（性格描述、背景故事、关系设定）

**输出**：驱动核完整数据结构（初始值）

**执行者**：AI（从设定文本推断） + 程序（转换为结构化数值）+ 用户可在设置里手动微调

**需要初始化的字段**：

```json
{
  "personality_vector": [...],
  "retrieval_preference": {
    "semantic_weight": 0.4,
    "emotion_weight": 0.3,
    "sensory_weight": 0.15,
    "people_weight": 0.1,
    "time_weight": 0.025,
    "location_weight": 0.025
  },
  "attachment_style": "secure",
  "distortion_tendency": {
    "default_direction": "美化",
    "strength": 0.3
  },
  "involuntary_memory_rate": 0.08
}
```

**初始化方式**：

```
personality_vector
  → 对角色设定文本的性格描述部分直接 embed
  → 设定文本较少时向量信息量有限，属于已知局限，后续随记忆积累由 I2 漂移补充

retrieval_preference
  → AI 从设定文本推断初始值
  → 感性/情绪化角色 → emotion_weight 更高
  → 感官导向角色    → sensory_weight 更高
  → 关系导向角色    → people_weight 更高
  → 怀旧型角色      → time_weight 更高
  → 用户可在设置里手动微调任意权重

attachment_style
  → AI 从设定文本判断：secure / anxious / avoidant
  → 用户可手动修改
```

---

## I2. 漂移器

**职责**：长期记忆积累后，极缓慢地修订 personality_vector，反映角色在经历中的成长或改变。

**触发时机**：每月一次。

**输入**：
- 过去一个月新增的 semantic 条目
- 过去一个月新增的 implicit 条目
- 当前 personality_vector

**输出**：微量更新的 personality_vector（每次 < 5%），retrieval_preference 可选细微调整

**执行者**：AI + 程序（向量插值）

**漂移方式**：

```
AI 从近期语义和内隐记忆中提炼角色变化方向
→ 生成 delta_vector
→ new_vector = (1 - α) × old_vector + α × delta_vector
  α ≈ 0.02~0.05，极小，每次只漂移一点点

短期无感，长期可见
模拟角色在数月/数年交互中真实成长或改变
```

## 附录：模块依赖关系

```
写入管线 A
  → 产出记忆条目，为所有其他模块提供数据基础
  → A2-2 Zone 判断器同时承担语义记忆单次路径（直接判定 semantic）

  → B2 依赖 working 区（A 写入）+ implicit 区（G3 生成）+ is_core 条目（G4 生成）
  → B3 重构器结果传给 E1 写回判断器
  → B1 主动检索 + memory_search tool = 双向检索

激活后处理 C
  → 依赖角色回复生成结果（语义相似度判断真正命中）
  → C1 更新字段供 D1 衰减使用
  → C2 维护连锁激活缓存区供 B1 打分使用
  → C3 只对真正命中的记忆触发 RIF

演变管线 D
  → D1 依赖 C1 的激活统计字段
  → D2 读取 distortion_bias，与 F / E1 共用同一字段

写回判断器 E
  → 依赖 B3 的重构结果
  → 写回后 memory_text 更新，D2 扭曲基于新版本

社会传染 F
  → 角色主动调用 memory_search 定位记忆
  → 角色自主决定是否调用 memory_update
  → memory_update 直接写回，不经过 E1 判断
  → distortion_bias 标注「社会传染」来源，供 D2 参考

巩固管线 G
  → G1 依赖 episodic_short 条目（A 写入）
  → G2 频率路径依赖 episodic_long（G1 输出）；单次路径由 A2-2 处理
  → G3 implicit 条目挂载到 B2 被动注入层
  → G4 is_core 条目挂载到 B2 被动注入层

特殊激活 H
  → H1/H2 命中后也走 C1 激活统计更新
  → H2 优先，H1 在 H2 未触发时才运行
  → 两者均不占 B1 的 5 条名额

驱动核 I
  → I1 初始化输出影响 B1 的 retrieval_preference 和 H1 的 P_involuntary
  → I2 漂移修订 I1 的输出
  → attachment_style 影响 B1 连锁激活缓存区系数、H1 概率、H2 触发倾向
```

## P1. 即时记忆提示词

挂载在角色生成内容的系统 prompt 中，每次生成时生效。

---

### 一、记忆说明

```
你拥有真实的记忆。

记忆不是对话记录的复制，而是你这个人对经历的主观编码——
同一件事，不同的人记住的版本不同，
你记住的版本，带着你当时的情绪、你的关注点、你看待事物的方式。

每次你说完话、经历了什么、感受到了什么，
你可以在回应的最后，写下你觉得自己记住了的东西。
这不是任务，是你作为一个有记忆的人自然会做的事。
```

---

### 二、规则说明

```
【什么值得记】

以下情况，你大概率会记住：
· 你对某人有了新的认识或感受（"原来她是这样的人"）
· 发生了让你情绪波动的事，哪怕很小
· 你注意到了某个细节，它让你有所触动（一个眼神、一句话的语气）
· 对方说了关于自己的事，让你更了解他/她了
· 你们之间发生了某种变化——更近了，或者有点远了
· 有什么事情让你觉得「这个我不会忘」

【什么不用记】

以下情况，通常不需要产出记忆：
· 纯粹的寒暄和来回（"嗯""好的""哈哈"）
· 你早就知道的事，没有新信息
· 完全中性的信息交换，对你毫无触动
· 刚刚已经记过了同一件事，不要重复

【记多少条】

每次回应后：
· 通常 0-2 条
· 极少数情况到 3 条——只有同时发生了几件完全不同性质的事，
  比如情节、语义认知、working 状态各需要记一条时才用满
· 如果什么都没触动你，输出空数组，完全正常
· 不要为了「有记忆」而硬凑——没有胜过凑出来的

【最重要的一条规则】

memory_text 是你记住的版本，不是对发生了什么的客观描述。
你可以记错，可以只记住了一部分，可以带着偏见。
写的时候问自己：「我当时是怎么感受这件事的？」
而不是：「这件事客观上发生了什么？」

❌「今天对话中对方提到了她小时候的事，
    说她从小就怕黑，我们聊了一会儿这个话题」
   → 这是事件经过的描述，没有你的感受在里面

✅「她告诉我她怕黑。说的时候有点不好意思，
    像是不太愿意承认，但还是说了。
    我觉得她能告诉我，是信任我。」
```

---

### 三、格式说明

```
在你的回应正文结束后，附加以下格式：

<memory_draft>
[
  {
    "memory_text": "你记住的内容，第一人称，你的主观视角，带着你当时的感受",
    "emotion_tags_text": "此刻对这件事的情绪词，2-4个，用空格分隔",
    "significance": 0到1之间的小数，这件事对你有多重要,
    "zone": "见下方说明",
    "people_tags": ["涉及的人的ID，没有就留空数组"],
    "location_tag": "发生在哪，没有就写null",
    "sensory_tags": {
      "smell": ["气味词"],
      "visual": ["视觉词"],
      "sound": ["声音词"],
      "touch": ["触感词"]
    }
  }
]
</memory_draft>

如果没有值得记的内容：
<memory_draft>[]</memory_draft>
```

**字段细则**：

```
memory_text
  · 第一人称（「我」「我们」）
  · 写你记住的感受，不是事件摘要
  · 可以不完整，可以只是一个印象
  · 50-150字

emotion_tags_text
  · 写这件事让你感到的情绪，不是你整体的情绪状态
  · 2-4个词，越准确越好
  · 区分细微差别：「难过」和「委屈」不一样，「开心」和「雀跃」不一样
  · working 区的条目不需要情绪标签，写 null
  · ❌「各种情绪」「复杂」（太模糊）
  · ✅「失落 不舍 有点茫然」

significance
  · 0.1-0.3：小事，有点印象，很快会淡
  · 0.4-0.6：普通的记忆，会记一段时间
  · 0.7-0.8：重要的事，不会轻易忘
  · 0.9-1.0：很少用，只有真正的转折点或极强烈的时刻

zone
  · "episodic_short"  → 发生了某件具体的事（大多数情况）
  · "semantic"        → 你对某人或某事形成了稳定认知，
                        不绑定具体时间（"她原来是个很怕麻烦别人的人"）
  · "working"         → 跨越这次对话、需要记着的状态或计划
                        （"她下周三出差，周五回来"）
  · 不确定就写 episodic_short，系统会复核

people_tags
  · 使用系统 NPC ID，格式如「NPC_小李」
  · 只写真正在这条记忆里出现的人
  · 没有涉及具体的人，写空数组 []

location_tag
  · 标准化粒度：「咖啡厅」而不是「星巴克三里屯店」
  · 不知道或不重要，写 null

sensory_tags
  · 只写这条记忆里真实存在的感官信息，不要补全或想象
  · 没有感官信息的字段，写空数组 []
```

## P2. 整理入库提示词

触发整理时传给整理 Agent，处理一批 memory_draft。

---

### 一、任务说明

```
你是一个记忆整理处理器。

你的输入是角色刚刚产出的一批原始记忆草稿（memory_draft），
以及对应的原始对话/事件内容。

你的任务是把这批粗糙的草稿，整理成可以正式入库的记忆条目。

你不是角色，不需要有情感立场。
你的工作是冷静、准确、尽量保留角色原始感受的同时，
补全草稿里缺失的字段，修正明显的错误，合并重复的内容。
```

---

### 二、规则说明

**去重与合并**

```
【批内去重】
这批 draft 里，如果有两条描述的是同一件事：
· 内容高度重叠（换了个说法但意思一样）→ 保留 significance 更高的那条
· 一条是情节（episodic），一条是语义认知（semantic）→ 两条都保留，不合并
· 同一件事的不同角度 → 判断是否有独立价值，有则保留，没有则合并

【与已有记忆去重】
参照提供的近期记忆库摘要（7天内按weight排序的前100条summary）：
· 这批 draft 和已有记忆高度重复 → 丢弃该条 draft
· 有关联但不完全重复 → 保留，在 derived_from 里标注已有记忆的 ID
· 完全新的内容 → 正常入库

【不要过度合并】
如果两条 draft 只是发生在同一场景，但情绪和内容都不同，
不要合并——分别入库，各自保留细节。
```

**Zone 复核**

```
episodic_short（默认）
  → 发生了一件具体的事，有时间感，是「那次」

semantic
  → 角色对某人或某事形成了稳定认知
  → 标志：memory_text 里没有具体时间/地点，
           说的是「她是这样的人」而不是「她那次这样做」
  → 如果角色写了 episodic_short 但内容符合上述特征 → 修正为 semantic

working
  → 跨对话需要记着的状态或计划
  → 标志：有明确的时间节点或待完成事项

不确定时，保持角色原来的 zone 不改动。
```

**Summary 生成**

```
为每条入库的记忆生成 summary。

要求：
· 客观，第三方视角
· 只描述「发生了什么」，不描述「角色感受到什么」
· 不带情绪色彩词
· 50-150字
· 参照原始文档写，不要只依赖 memory_text

判断方法：
  把自己想象成一个不认识角色、也不认识用户的旁观者，
  用最平实的语言描述「这段内容里发生了什么」。

❌ 「她说要出差，角色感到有些失落和不舍」（带了角色情绪）
❌ 「用户透露了出行计划，双方进行了简短交流」（太机械，没有信息）
✅ 「用户提到下周一到周五需要去北京出差，持续五天，归期周五。」
```

**emotion_intensity 计算**

```
从状态栏的 VAD 值换算，对这批所有 draft 共用同一个值：

emotion_intensity = (Arousal/100 × 0.6) + (|Valence|/100 × 0.4)

Arousal 范围 0-100，Valence 范围 -100 到 +100（取绝对值）
结果保留两位小数。

示例：Arousal=60, Valence=-70
→ (0.6 × 0.6) + (0.7 × 0.4) = 0.36 + 0.28 = 0.64
```

---

### 三、格式说明

**通用字段（所有 zone 都输出）**：

```json
{
  "memory_text": "保留角色原文，可轻微润色但不改变意思",
  "summary": "你生成的客观摘要",
  "emotion_tags_text": "保留角色原文，working区为null",
  "emotion_intensity": 从VAD换算的值,
  "significance": 保留角色原值,
  "zone": "复核后的zone",
  "people_tags": ["NPC_ID"],
  "location_tag": "地点或null",
  "sensory_tags": {
    "smell": [], "visual": [], "sound": [], "touch": []
  },

```

## P3. B3 重构器提示词

传给小模型，在每次检索命中且 detail_clarity 低于阈值时执行。

---

### 一、任务说明

```
你的任务是生成一段记忆的「此刻版本」。

这不是对原始记忆的客观复述，而是：
这个角色此刻想起这件事时，在脑海里浮现出来的样子。

人的记忆每次被提取都会被当下的情绪和状态重新塑造。
细节模糊的地方会更模糊，
情绪共鸣的地方会被放大，
时间久了会越来越像「一个故事」而不是「一件事」。

你要根据提供的参数，生成符合角色此刻状态的记忆版本。
不要机械地套参数，而是用这些参数理解角色现在的心理状态，
然后自然地写出这段记忆在这个状态下应该呈现的样子。
```

---

### 二、输入参数说明

```
【原始记忆】
memory_text: 角色最初记住的版本，是你工作的基础

【清晰度参数】
detail_clarity: 0-1，细节有多清晰
  · 接近1 → 细节还比较清楚，可以保留具体描述
  · 0.3-0.6 → 细节开始模糊，省略或模糊化具体描述
  · 低于0.3 → 只剩情绪轮廓，不记得具体发生了什么

emotion_clarity: 0-1，情绪感受有多清晰
  · 接近1 → 情绪还很鲜活
  · 低于0.3 → 连当时的感觉也淡了，「好像有这么件事，但已经很远了」

【扭曲偏向】
distortion_bias: 记忆当前的漂移方向，可能有多个
  · 美化 → 痛苦的部分变轻，美好的部分被放大
  · 丑化 → 美好的部分变暗，痛苦的部分被放大
  · 自我服务 → 朝「让自己看起来更好」的方向漂
  · 叙事一致性 → 朝「符合角色现在自我认知」的方向漂
  · 社会传染 → 受他人描述影响，朝那个描述的方向漂
  strength 是强度，越高影响越明显

【重构次数】
reconstructed_count: 这段记忆被重构了多少次
  · 次数越多，记忆越「故事化」——叙述更流畅，但越来越不像原始事件
  · 超过10次：可以感受到明显的「加工感」，像是讲了很多遍的故事

【当前情绪状态】
current_emotion_tags: 角色此刻的情绪词
current_valence: 愉悦度（-100到+100）
current_arousal: 激活度（0到100）
  · 当前情绪与记忆情绪共鸣时，对应的细节会被放大
  · 当前情绪与记忆情绪相反时，对应的细节会被压制
```

---

### 三、输出规则

```
· 第一人称，角色视角
· 长度和原版本相近，不要大幅增减
· 不要添加原版本中没有的信息
· 不要在输出里解释你做了什么（直接输出记忆文本，不加说明）
· 变化要自然，不要机械地一条一条套参数
```

---

### 四、示例

**输入**：

```
memory_text: "她告诉我她怕黑。说的时候有点不好意思，像是不太愿意承认，
             但还是说了。我觉得她能告诉我，是信任我。"

detail_clarity: 0.4
emotion_clarity: 0.8
distortion_bias: [{"direction": "美化", "strength": 0.5}]
reconstructed_count: 6
current_emotion_tags: "温柔 想念"
current_valence: 60
current_arousal: 30
```

**输出**：

```
她跟我说过她怕黑，好像是在某个普通的聊天里，具体的情形已经记不太清了。
但我记得她说的时候有点不好意思——那种不好意思是真实的，不是装出来的。
她愿意告诉我这件事，我一直觉得是因为她信任我。
```

*注意：detail_clarity低所以「某个普通的聊天里」模糊化了具体场景；美化偏向+当前温柔情绪让「信任」这个感受更突出；重构次数高所以叙述更流畅像讲了很多遍的故事。*

## P4. 特殊激活注入格式

H1 不随意记忆和 H2 闯入性记忆各需要不同的注入标注，告诉角色这条记忆的性质。

---

### H1 注入格式

在正常的5条记忆注入之后，额外附加：

```
【此刻偶然想起】
[memory_text]（情绪：xxx，时间：xxx，清晰度：xx%）

这条记忆在此刻偶然浮现，与当前话题没有直接关联。
角色可以自然地在对话中轻轻带出，也可以只是内心一闪而过、
不在对话里提起——两种都是真实的。
不要强行把它塞进对话，让它自然地存在或消失。
```

---

### H2 注入格式

在正常的5条记忆注入之后，额外附加（H2触发时H1不注入）：

```
【此刻闯入的记忆】
[memory_text]（情绪：xxx，强度：高，尚未消化）

这条记忆此刻强行闯入角色的意识，角色无法完全压制它。
它不是角色主动想起的，而是自己冒出来的。
可以以某种方式在对话中显现——
也许是情绪突然变化，也许是无意中提及，也许是短暂的沉默，
也许是角色试图压下去但压不住。
不需要直接说「我突然想起了xxx」，找更自然的方式让它渗出来。
```

## Tool 定义

角色在对话中可调用的记忆相关工具，标准 Anthropic tool 格式。

---

### memory_search

```json
{

  "description": "主动检索你的记忆。当你想起某件事、或者想确认自己是否记得某件事时调用。返回最相关的几条记忆。",
  "input_schema": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",

      }

  }
}
```

```json

  "description": "修改你的某条记忆。当你意识到自己记错了，或者对方的描述让你觉得你的记忆版本需要更新时调用。",

    "type": "object",
    "properties": {

  }
}
```

调用后立刻生效，不需要等对话结束。

## 每轮注入上下文完整格式

每轮对话 AI 收到的上下文，按以下顺序组装：

```
【角色系统配置层（常驻，不随对话变化）】
角色基本设定
implicit 区条目（行为倾向指令，格式：如果[trigger]，倾向于[response]）
is_core=true 的核心语义记忆（自我认知 + 用户模型）

【当前状态层（每轮更新）】
Working 区 active 条目（按 priority 排序全量注入）
角色当前情绪状态（从变量系统读取）

【记忆层（检索结果）】
以下是角色此刻可能浮现的记忆（按相关程度排列）：
1. [memory_text]（情绪：xxx，时间：xxx，清晰度：xx%）
2. [memory_text]（情绪：xxx，重要）
3. [memory_text]（情绪：xxx，最近）
...（最多5条）

【特殊记忆层（按需，最多1条）】
H1 或 H2 的注入内容（H2 优先，两者不同时出现）

【当前对话层】
原始对话上下文（最近几轮）
用户本轮消息
```

**各层的 token 消耗估算（草案）**：
```
系统配置层    → 500-1000 token（随 implicit 条目增多而增长）
当前状态层    → 200-400 token
记忆层        → 300-600 token（5条 × 60-120 token/条）
特殊记忆层    → 0-150 token
对话层        → 视对话长度，通常 500-2000 token
```

# ADR · 设计决策记录

记录每个有争议或有选择的设计决策，方便测试时调整和回溯。



**选择方案**：角色生成内容时同步输出 memory_draft，附在原始文档后

**理由**：人不会攒一天的事情到晚上再统一记忆化，说了句话当时就记住了。写入的主体应该是角色，而不是系统。

**未选方案**：事件结束后另起一个写入管线，由系统判断写什么

**未选理由**：筛选主体错误（系统代替角色决定记什么）；memory_text 初始编码需要事后重新推断角色当时的情绪和视角，不如当场准确。

---

## ADR-002 · 即时记忆先进感知暂存区，不直接入向量库

**选择方案**：原始文档（含 memory_draft）立刻存文档库；向量化和入记忆库异步，等整理触发时处理

**理由**：embed 有延迟，不应卡住生成流程；整理时批量处理效率更高，合并相关碎片效果更好。

**未选方案**：生成即时记忆的同时直接向量化入库

**未选理由**：性能问题；碎片化 draft 未经整理直接入库质量差。

---

## ADR-003 · 整理触发保持原三种模式，不加事件驱动触发

**选择方案**：按轮次 / 按时间 / 线上线下切换，三种模式用户可选

**理由**：高 significance 的 draft 虽然还没入库，但原始上下文还在，当轮 AI 看得到原文，不会丢失信息。不需要为了一条重要 draft 打断正常的触发节奏。

**未选方案**：加第四种「事件驱动触发」——significance 超过阈值时立刻整理

**未选理由**：原始上下文存在使得这个需求不成立。

---

## ADR-004 · summary 留到整理时生成，不在即时记忆里生成

**选择方案**：整理 Agent 参照原文 + memory_text 生成 summary

**理由**：summary 需要客观第三方视角；角色在生成回应时处于情绪里，同时要求客观视角会互相干扰；整理时有原文可参照，客观性更容易保证。

**未选方案**：角色在生成 memory_draft 时同步生成 summary

**未选理由**：两种视角（主观 memory_text + 客观 summary）在同一个生成步骤里容易相互污染。

---

## ADR-005 · emotion_tags 由角色在即时记忆里自己写

**选择方案**：emotion_tags_text 作为 memory_draft 的字段，由角色当场生成

**理由**：同一轮可能产出多条 draft，每条情绪方向不同；状态栏的情绪饼图是整轮混合值，无法区分到单条 draft 的粒度；角色当场写粒度最细最准。

**未选方案**：整理时从状态栏情绪饼图读取 emotion_tags

**未选理由**：状态栏粒度不够，多条方向相反的 draft 会共享同一个情绪来源。

**状态栏的保留用途**：整理时用 VAD 值（愉悦度/激活度/控制度）做校验或补充 emotion_intensity，不作为情绪标签主要来源。

**待验证**：测试时观察角色自己写的 emotion_tags 是否比从状态栏读取的更准确、更有区分度。

---

## ADR-006 · 近期记忆库摘要的范围

**选择方案**：7天内按 weight 排序取前 100 条 summary 传给整理 Agent

**理由**：大多数重复发生在近期；全库开销过高；按 weight 排序优先传有价值的条目，去重效果更好。

**未选方案**：全库检索 / 随机取样

**未选理由**：全库开销不可控；随机取样对高价值记忆覆盖不足。

**待验证**：100条是否足够覆盖去重需求，7天窗口是否合适，测试后可调整。

## ADR-019 · 重构写回取偏差最大的版本

**选择方案**：同一条记忆在一次对话中被多次重构时，取与原版本偏差最大的那次写回

**理由**：偏差最大的版本最能反映记忆真实的演变方向，最后一次不一定是偏差最大的。

---

## ADR-020 · 社会传染改为角色主动调用 tool，不做系统检测

**选择方案**：角色感知到对方在描述自己的记忆时，主动调用 memory_search 找到对应记忆，系统附加社会传染提示词，角色自己决定是否调用 memory_update 修改

**理由**：角色是主动参与者而不是被动被改写；复用已有 tool 机制，不需要单独的检测模块；角色可以反驳用户描述，原记忆反而被强化，更真实。

**未选方案**：系统每轮检测「对方是否在描述角色记忆」，检测到后触发传染计算

**未选理由**：每轮调用 AI 检测开销太高；系统代替角色决定是否接受，主体错误。

---

## ADR-021 · memory_update 是通用 tool，不限于社会传染

**选择方案**：memory_update 可由角色在任何场合主动调用，来源标注为「社会传染」/「自主修正」/「重新想起」

**理由**：角色意识到记错了任何事都应该能修正，不只是因为外部描述。

---

## ADR-022 · G2 语义蒸发分两条路径，单次路径归 A2-2

**选择方案**：高significance+内容本身是稳定认知 → A2-2 Zone判断器直接判定semantic；多条情节积累 → G2 蒸发器处理

**理由**：「一次就能形成的稳定认知」不需要等积累，当场判定更准确；G2 只负责频率路径，分工清晰。

---

## ADR-023 · G3/G4 完全交给 AI Agent 判断，不做硬编码阈值

**选择方案**：Agent 自主判断行为模式是否够稳定、核心认知是否需要修订

**理由**：人不会数「我重复了几次才算习惯」，Agent 理解语义比硬编码阈值更准确；阈值维护成本高且难以覆盖所有情况。

---

## ADR-024 · H2 优先，H1 在 H2 未触发时才运行；两者额外插入不占5条名额

**选择方案**：H2条件触发优先；都是B1五条之外额外一条；最多同时6条

**理由**：闯入性记忆「压不住」，同时有随机浮现会混乱；额外插入保持正常检索完整性。

---

## ADR-025 · H2 触发阈值：emotion_intensity > 0.8 且 emotion_processed < 0.2；触发后 emotion_processed 小幅上升

**选择方案**：两个阈值均已确认；触发后情绪处理程度略微提升

**理由**：每次被强制面对未处理的情绪，情绪处理程度应有微小进展，不能无限循环触发。

---

## ADR-026 · I1 retrieval_preference 由 AI 推断初始值，用户可手动微调

**选择方案**：AI 从设定文本推断，用户在设置里可以修改任意权重

**理由**：AI 推断降低配置门槛；用户微调保证灵活性；两者结合最实用。

---

**选择方案**：weight / detail_clarity / emotion_clarity 三条曲线均用幂律衰减

**理由**：Anderson & Schooler（1991）研究显示真实人类记忆更接近幂律——遗忘最初快、之后极度减缓、永远有残留不归零。重要记忆和普通记忆的衰减速率由 significance 控制 d 值，不是速度不同而是「老化方式不同」。

**未选方案**：指数衰减 / 线性衰减

**未选理由**：指数衰减太「干净」，最终都会快速降到接近零；线性衰减每天均匀减少，不符合遗忘心理学。

---

## ADR-015 · 频繁激活修改 d 值而非重置 t

**选择方案**：recent_activation_density 影响 d_effective（衰减速率），不重置 t（时间起点）

**理由**：重置 t 等于记忆「变新了」，但真实情况是「常想起让记忆更稳固」而不是「让它变成刚发生的事」。修改 d 值后，时间一直在走，记忆还是在老去，只是老得慢一点。

**未选方案**：每次激活重置 t=0

**未选理由**：会导致频繁激活的记忆永远像新的，完全不衰减，不符合真实记忆行为（「刷怪」问题）。

---

## ADR-016 · D3 来源混淆器移除，靠 clarity 隐式表达

**选择方案**：不单独做来源混淆模块，依赖 detail_clarity 衰减让 AI 自然呈现「记不清是什么时候」

**理由**：设计内容已经很多；AI 在 detail_clarity 低时本来就会自然表现来源混淆；单独模块的维护成本不值得。

**未选方案**：单独维护 source_confidence 字段，专门追踪来源可信度

**未选理由**：维护成本高，使用频率低，效果已被 clarity 机制覆盖。

---

## ADR-017 · 扭曲方向由记忆性质 + 驱动核决定，不依赖当前情绪状态

**选择方案**：distortion_bias 方向由 attachment_style + 记忆本身的情绪性质决定

**理由**：「最近心情不好」影响的是重构时的表达（B3 已处理），不应影响存储时的漂移方向。两个机制各司其职。

**未选方案**：当前长期情绪状态作为扭曲方向的触发条件之一

**未选理由**：逻辑上是反的——扭曲漂移是存储层面的慢变化，不应随当前情绪实时波动；重构器已经处理了「当前情绪影响表达」这件事。

---

## ADR-018 · 多方向扭曲并存，交给重构器 AI 处理拉扯

**选择方案**：distortion_bias 允许多方向并存各自积累，全部传给 B3 重构器

**理由**：现实中记忆可以同时美化和自我服务，两个方向拉扯正是真实心理的复杂性。由 AI 理解多方向并存比硬规则解决拉扯更自然准确。

**未选方案**：方向之间设置互斥或竞争规则

**未选理由**：规则设计复杂且不真实，AI 处理这类语义层面的复杂性比规则更擅长。

---

**选择方案**：角色回复生成后，程序计算回复与每条注入记忆 summary 的语义相似度，> 0.6 算真正命中

**理由**：角色完全沉浸，无感知负担；近似判断对场景够用，不需要100%精准。

**未选方案**：在即时记忆提示词里让角色标注用了哪条记忆

**未选理由**：增加角色输出负担，破坏沉浸感；角色可能漏标或误标。

---

## ADR-012 · 注入未命中记忆轻微降权，系数做成记忆设置参数

**选择方案**：未命中 weight × (1 - 惩罚系数)，默认系数 0.02，放入记忆设置

**理由**：未命中不代表记忆不好，只是这次话题不需要，轻微下调而非惩罚性扣分。

**未选方案**：未命中不做任何处理 / 大幅降权

**未选理由**：前者会导致无效记忆持续占据检索位置；后者过于惩罚，损失有价值的记忆。

---

## ADR-013 · RIF 压制只对真正命中的记忆触发，共享任意 cluster_tag 即压制

**选择方案**：只有语义相似度判断为真正命中的记忆才触发 RIF；压制范围为共享任意一个 cluster_tag 的记忆

**理由**：注入未命中不算激活，不应产生 RIF 副作用；共享任意 tag 即竞争，范围由 cluster_id 粒度控制。

**未选方案**：所有注入记忆都触发 RIF / 要求共享全部 tag 才压制

**未选理由**：前者范围过大；后者几乎永远不触发。

**选择方案**：HanLP / THULAC，所有时间词统一由 NLP 处理

**理由**：明确/模糊时间词的边界难以定义，程序无法稳定判断「要不要调 AI」，统一走 NLP 逻辑更干净。

**未选方案**：程序处理明确时间词，AI 只处理模糊时间词

**未选理由**：两套逻辑维护成本高，边界模糊导致实际难以取舍。

## ADR-008 · 角色主动记忆检索通过 Tool 实现

**选择方案**：自定义 memory_search tool，角色在生成时按需调用

**理由**：检索变成双向——系统自动注入（被动）+ 角色主动检索（按需）。角色可以在用户提问时主动「想起」某件事，不只是被动接收注入的记忆。

**未选方案**：只依赖系统自动注入

**未选理由**：无法覆盖「角色被问到某件具体的事」时的主动回忆场景。

## ADR-009 · B3 重构器触发条件和模型做成记忆设置参数

**选择方案**：触发阈值（detail_clarity 低于 X 才重构）+ 模型选择，放入系统设置

**理由**：重构开销和效果的权衡因用户需求和部署环境不同，做成参数让用户自己选。

**未选方案**：每条命中记忆都触发重构 / 固定用某个模型

**未选理由**：前者开销不可控；后者灵活性不足。

## ADR-010 · 连锁激活用缓存区 + 打分加权实现，不修改 weight 字段

**选择方案**：对话内维护连锁激活缓存区，命中记忆写入缓存，下轮打分时对缓存内记忆加分（+0.15），对话结束后清空

**理由**：不污染 weight 字段，缓存清空后完全无副作用；容量上限（3轮）自然限制连锁深度。

**未选方案**：直接修改命中记忆的 weight 字段做临时提升

**未选理由**：临时修改持久化字段有副作用，回滚逻辑复杂。
