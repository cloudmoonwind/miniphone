# ICS 记忆系统 - 检索架构设计文档

日期：2026-04-22

---

## 一、整体架构：五层结构

```
原始信息层
    ↓
通道层（可选扩展）
    ↓
检索层（六种模式并行）
    ↓
汇总层（去重 / 加权 / 取舍 / 排布）
    ↓
记忆库（双向：被检索，也被写入）
```

不开扩展功能时为四层六模式。

### 每层存在的原因

**原始信息层**：用户对话、角色生活事件、系统触发——这些是记忆的原材料。不是所有输入都会进入记忆（筛选在写入机制里处理），这一层只是信号的来源。

**通道层（可选）**：存在条件是"原始信号不够用，需要AI转化才能被检索层利用"。感官描述（"草木香"）、隐喻（"像刺猬一样"）、身体感受（"胃突然紧"）——这些输入的字面含义不足以触发有效检索，需要AI丰富化后才有意义。

关键区分：**通道层不是所有输入的必经入口**，只是部分输入的前置处理。人名、时间词、语义内容直接进检索层，不需要经过通道层。关闭通道层时这些输入完全不受影响，只有感官类输入降级为精确匹配。

**检索层**：六种模式并行运行，因为人的联想是多维度的——同一段记忆可以被情绪、被人名、被地点、被感官同时触发。并行而非串行，是因为多模式同时命中的记忆正是最该浮现的记忆（多维共鸣）。

**汇总层**：token有限，不能把所有命中的记忆都注入。汇总层的工作是：去重（多个模式可能返回同一条）、打分（综合相似度+weight+时效）、取舍（5条上限）、排布（强相关+重要+近期的组合）。

**记忆库**：双向——被检索时读出，演变和激活时写入。写入和读出走不同的机制，互不干扰。

## 二、三个核心概念的区分

这是整个系统的基础，必须先分清楚：

| 概念 | 归属 | 视角 | 说明 |
|------|------|------|------|
| **原文** | 系统 | 上帝视角或客观记录 | 时间线条目、聊天记录、梦境记录等原始内容，完整保存 |
| **摘要** | 系统 | 客观第三方 | 对原文的客观压缩，用于语义检索，不带角色主观色彩 |
| **记忆** | 角色 | 角色自身视角 | 角色对经历的主观编码，受性格/情绪/日记影响，可能不准确 |

**关键特性**：
- 记忆不一定对应单一原文，可能是多个事件融合后的产物
- 摘要用于检索（找到哪条记忆），记忆用于生成（角色怎么描述这件事）
- 原文永久保存，记忆会演变，两者独立存在

## 三、记忆库结构

### 3.1 记忆条目字段

```json
{
  "id": "M_001",

  // ── 检索用（客观，永远不扭曲）──
  "summary": "那天堵车两小时，很烦躁但朋友圈没发",
  "summary_vector": [...],              // 摘要向量，768维，语义检索用，embed程序生成

  // ── 生成用（主观，属于角色，会随演变漂移）──
  "memory_text": "那天堵在路上，车里放着她喜欢的歌，反而有点难受",
  // summary永远不扭曲，是客观锚点
  // memory_text随重构/扭曲缓慢变化
  // 提取时附带clarity参数，AI根据清晰度呈现，不直接改文本

  // ── 情绪 ──
  "emotion_tags_text": "焦虑 压抑 委屈",  // 情绪词拼接，AI写入
  "emotion_vector": [...],               // 情绪词向量，768维，embed程序生成
  "emotion_intensity": 0.7,             // 情绪强度，从变量系统读取，写入时定，静态
  "emotion_processed": 0.0,             // 情绪处理程度，0-1连续值，初始固定为0
                                         // 随时间、写日记、对话提及缓慢上升
                                         // 上升速度由变量系统情绪强度+情绪类型占比决定
                                         // 低值+高emotion_intensity → 闯入性记忆、回避行为概率上升
                                         // （多变量联合计算，无硬编码阈值）

  // ── 感官 ──
  "sensory_tags": {                      // 感官标签，AI写入，精确检索
    "smell": ["松木", "潮湿"],
    "visual": ["金色光斑", "绿色"],
    "sound": ["虫鸣", "风声"],
    "touch": ["凉意", "绵软"]
  },
  // ~~sensory_vector~~ 已删除
  // 删除原因：sensory_tags精确匹配已足够，模糊向量检索在角色扮演场景性价比不足，后期有需求再加

  // ── 索引标签 ──
  "people_tags": ["NPC_小李"],           // 人物ID列表，AI写入，精确匹配
  "location_tag": "咖啡厅",              // 标准化地点名，AI写入时统一粒度
                                         // 如"星巴克三里屯"→"咖啡厅"，cluster_id从此生成
  "time": "2026-03-10T18:00:00",         // 事件时间戳，程序写入，范围查询
                                         // semantic区此字段为null

  // ── 分区（写入时定，永远不变）──
  "zone": "episodic_long",
  // episodic_short → 外显短期，高可塑，未巩固，轻量筛选后拼入上下文
  // episodic_long  → 外显长期情节，低可塑，有时间地点，检索召回
  // semantic       → 外显长期语义，从情节蒸发，无时间地点，检索召回
  // implicit       → 内隐行为指令，不走检索，常驻角色配置层
  // working        → 当前状态和近期计划，有时效性，每次对话自动全量注入
  // ~~zone_strength~~ 已删除
  // 删除原因：层级转化触发条件可用weight+activation_count判断，此字段没有独立计算来源

  // ── 重要性（静态，AI写入，不变）──
  "significance": 0.8,                  // 事件本身重要性，AI写入
                                         // weight初始值 = significance
                                         // clarity初始值默认为1
                                         // significance低于阈值时初始clarity自动下调

  // ── 检索权重（动态，程序计算）──
  "weight": 0.8,                        // 当前检索权重，初始值=significance
                                         // 随recent_activation_density上升，随时间/压制/吸收下降

  // ── 清晰度（动态，程序计算）──
  "detail_clarity": 1.0,               // 细节清晰度，初始1，衰减快
  "emotion_clarity": 1.0,              // 情绪清晰度，初始1，衰减慢
                                        // 衰减速度由变量系统情绪强度+情绪类型占比调节
                                        // detail_clarity低 → AI呈现"细节模糊"
                                        // emotion_clarity低 → AI呈现"感觉也淡了"

  // ── 演变追踪（动态）──
  "distortion_bias": [],               // 扭曲偏向数组，初始为空，程序/AI更新
                                        // 结构：[{direction: "美化", strength: 0.6}]
                                        // direction枚举：美化/丑化/自我服务/叙事一致性
                                        // 允许多个方向并存
  "reconstructed_count": 0,           // 被重构次数，程序计数，影响detail_clarity衰减加速

  // ── 激活统计（动态，程序计算）──
  "activation_count": 0,               // 总激活次数，辅助参数
  "recent_activation_density": 0.0,   // 近期激活密度（过去30天激活次数÷30）
                                        // weight衰减公式的核心输入
  "last_activated_at": "...",          // 上次激活时间，程序写入
  "last_modified_at": "...",           // 上次内容修改时间，程序写入
                                        // 重构写回时更新，不是激活时更新
                                        // clarity衰减从此字段算起

  // ── 层级转化（动态）──
  "derived_from": [],                  // 来源条目ID列表，程序在巩固/蒸发时写入
  "absorbed_by": [],                   // 被吸收的上层条目ID，程序写入
  "absorption_level": 0.0,            // 被吸收程度，0-1，高时weight自动下降

  // ── 网络效应（动态）──
  "cluster_id": [],                    // 所属记忆簇列表，从标准化tags自动生成
                                        // 如 ["咖啡厅", "等待", "NPC_小李"]，同簇记忆RIF时互相竞争
  // ~~linked_memories~~ 已删除
  // 删除原因：连锁激活改为实时相似度计算，无需维护静态关联列表
  // ~~association_count~~ 已删除
  // 删除原因：linked_memories删除后失去意义，连锁激活已改为实时计算
  "suppression_count": 0,             // 被RIF压制次数，程序计数，加速weight衰减
  // ~~interference_level~~ 已删除
  // 删除原因：灾难性干扰效果由detail_clarity低时的模糊重构自然模拟，专门追踪开销过高
  // ~~interfered_by~~ 已删除
  // 删除原因：同上

  // ── 来源关联（可为空）──
  "source_hints": [                    // 来源线索，AI+程序写入，主要用于调试和溯源
    {"type": "timeline", "time_range": "2026-03-10"},
    {"type": "chat", "time_range": "2026-03-09 to 2026-03-10"}
  ],
  // ~~source_confidence~~ 已删除
  // 删除原因：维护成本高，使用频率低，来源混淆效果已由clarity自然体现

  // ── implicit区专用（其他zone留空）──
  "trigger_condition": null,           // 触发条件，AI生成，自然语言
  "behavior_response": null,          // 行为反应指令，AI生成，自然语言

  // ── working区专用（其他zone留空）──
  "relevance_type": null,             // current_state / near_plan / pending_action
  "expires_at": null,                 // 时效截止时间，到期后status→expired
  "resolve_condition": null,          // 解决触发条件，结构化：
                                       // {type: "keyword", value: "感冒好了"}
                                       // {type: "time", value: "2026-04-28"}
                                       // {type: "manual"}
                                       // {type: "ai_judge"}（最贵，慎用）
  "priority": null,                   // 注入优先级 0-1
  "status": null,                     // active / expired / resolved，失效后保留记录

  "created_at": "..."                 // 创建时间，程序写入
}
```

### 字段统计

删除的7个字段及原因：

| 字段 | 删除原因 |
|------|---------|
| zone_strength | 层级转化可用weight+activation_count判断，无独立计算来源 |
| sensory_vector | 精确匹配已足够，后期有需求再加 |
| linked_memories | 连锁激活改为实时相似度计算 |
| association_count | linked_memories删除后失去意义 |
| interference_level | 灾难性干扰由detail_clarity模糊重构模拟 |
| interfered_by | 同上 |
| source_confidence | 维护成本高，使用频率低 |

保留字段：30个（含id）

### 3.2 字段说明补充

**summary vs memory_text**：
- summary永远客观，用于检索，永远不扭曲——它就是"客观版本"的保留
- memory_text是角色主观版本，会随重构/扭曲缓慢漂移
- 不需要单独存original_memory_text，summary已经承担了这个职责

**weight vs significance**：
```
significance  → 写入时定，这件事本身的重要程度，不变
weight        → 动态，当前检索权重
               初始值 = significance
               之后随激活/衰减/压制独立演变
```

**detail_clarity vs emotion_clarity**：
```
两条不同的衰减曲线：
detail_clarity   → 衰减快，受reconstructed_count和时间影响
emotion_clarity  → 衰减慢，主要受emotion_processed影响
```

**zone永远不变**：
```
记忆不会"搬家"，层级转化是生长出新条目
原条目通过absorbed_by + absorption_level标记被吸收程度
absorption_level高到一定程度 → weight自动下降
用户可审查absorbed条目后手动清理，支持回退
```

**层级转化示意**：
```
episodic_short M_001 "今天堵车很烦"
episodic_short M_002 "昨天堵车也很烦"
        ↓ 巩固
episodic_long  M_010 "最近上班路上经常堵车，心情很差"
               derived_from: ["M_001", "M_002"]

M_001 → absorbed_by: ["M_010"], absorption_level: 0.9
M_002 → absorbed_by: ["M_010"], absorption_level: 0.9
```

**implicit区格式特殊**：
```
trigger_condition + behavior_response 替代 summary/memory_text的功能
这两个字段让AI知道这是行为指令，不是事件描述
其他zone这两个字段留空
```

**zone注入方式**：
```
episodic_short / episodic_long / semantic → 检索召回，注入对话上下文
implicit  → 长期挂在角色prompt配置层，不走检索
working   → 每次对话自动扫描，未过期未resolved的全部注入，不走检索
```

### 3.3 向量维度选择

| 字段 | 推荐维度 | 说明 |
|------|---------|------|
| summary_vector | 768维 | 轻量模型够用，省存储 |
| emotion_vector | 768维 | 情绪词语义空间 |
| sensory_vector | 768维 | 感官词语义空间 |

推荐本地模型：`bge-m3`（中文友好，免费，可本地部署）

### 3.4 三类存储对应关系

| 类型 | 存储方式 | 原因 |
|------|---------|------|
| 语义 | 向量 | 模糊匹配，"委屈"能找到"难过" |
| 情绪 | 向量 | 情绪词模糊性强 |
| 感官 | 向量 | "松木香"≈"木香" |
| 时间 | 规则 | 精确匹配，快速 |
| 地点 | tag精确匹配 | 语义相似无意义 |
| 人物 | tag精确匹配（ID） | 独立实体 |

### 【检索层】

---

#### `id` · 唯一标识

| 属性 | 值 |
|------|-----|
| 类型 | 字符串 |
| 写入方 | 程序 |
| 静/动 | 静态 |
| 适用zone | 全部 |

**定义**：每条记忆的全局唯一标识符。

**值**：系统自动生成，格式如 `M_001`，不重复。

**设计原因**：derived_from、absorbed_by等字段需要引用其他记忆条目，必须有稳定的引用锚点。

**关联字段**：被 `derived_from` / `absorbed_by` 引用。

---

#### `summary` · 客观摘要

| 属性 | 值 |
|------|-----|
| 类型 | 字符串 |
| 写入方 | AI生成（加客观性约束） |
| 静/动 | 静态（永远不修改） |
| 适用zone | episodic_short / episodic_long / semantic |

**定义**：对这条记忆所对应事件的客观压缩描述，用于向量化检索。

**值**：自然语言字符串，50-150字，第三方视角，不带情绪色彩。

**设计原因**：检索需要客观（summary），生成需要主观（memory_text）。如果用扭曲后的memory_text做向量检索，被美化或丑化的记忆会失去正确的语义位置，检索结果会偏移。summary是永久的客观锚点，无论memory_text怎么漂移，都能找回这条记忆。

**关联字段**：
- 驱动 `summary_vector` 的生成
- 是 `memory_text` 的"客观原版"对照

**变化规则**：写入后永不修改。

**使用场景**：语义检索时embed后与查询向量计算相似度。

---

#### `summary_vector` · 摘要向量

| 属性 | 值 |
|------|-----|
| 类型 | 浮点数组 |
| 写入方 | embed程序（如bge-m3） |
| 静/动 | 静态 |
| 适用zone | episodic_short / episodic_long / semantic |

**定义**：summary的语义向量表示，768维浮点数组。

**值**：768个浮点数，由embed模型生成，人类不可读。

**设计原因**：向量相似度检索（余弦距离）能实现模糊语义匹配——"委屈"能匹配"难过"，"那次吵架"能匹配"争执"。纯文本检索做不到这一点。

**关联字段**：由 `summary` 生成。

**变化规则**：summary不变则不变。

**使用场景**：语义检索模式，每次对话必跑。

---

### 【生成层】

---

#### `memory_text` · 角色主观记忆

| 属性 | 值 |
|------|-----|
| 类型 | 字符串 |
| 写入方 | AI生成（初始），后续由重构/扭曲机制更新 |
| 静/动 | 动态（随演变缓慢漂移） |
| 适用zone | episodic_short / episodic_long / semantic |

**定义**：角色对这段经历的主观描述，是"此刻角色想起时脑海里浮现的版本"。

**值**：自然语言字符串，角色第一人称视角，带情绪色彩，可能不准确。

**设计原因**：同一件事，角色半年后想起和刚发生时想起，描述应该不同——细节模糊、情绪变化、可能被美化。如果每次都返回固定文本，失去了真实感。memory_text负责承载这种"随时间漂移的主观感受"，而summary保持客观不变。

**关联字段**：
- 受 `distortion_bias` 控制漂移方向
- 受 `detail_clarity` / `emotion_clarity` 控制注入AI时的呈现方式
- `reconstructed_count` 记录被重构的次数
- `last_modified_at` 在写回时更新

**变化规则**：
- 扭曲（被动）：存储期间自然漂移，沿distortion_bias方向缓慢改变
- 重构（主动）：提取时由小模型根据当前状态生成当下版本，对话结束后视偏差程度决定是否写回

**使用场景**：检索命中后，连同clarity参数一起注入AI上下文，让角色自然地"想起"。

### 【情绪层】

---

#### `emotion_tags_text` · 情绪词文本

| 属性 | 值 |
|------|-----|
| 类型 | 字符串 |
| 写入方 | AI写入 |
| 静/动 | 动态（情绪演变时可能更新） |
| 适用zone | episodic_short / episodic_long / semantic |

**定义**：描述这条记忆情绪色彩的词汇集合，空格分隔。

**值**：如 `"焦虑 压抑 委屈"`，3-5个情绪词，可以有多个。

**设计原因**：情绪向量化需要文本来源。拼接后embed，比单个词embed更能体现情绪的复合性。

**关联字段**：驱动 `emotion_vector` 的生成；与 `distortion_bias` 配合——扭曲发生时emotion_tags_text可能更新（如美化后"委屈"变为"平静"）。

**变化规则**：社会传染或扭曲积累到一定程度时AI更新。

**使用场景**：情绪检索模式。

---

#### `emotion_vector` · 情绪词向量

| 属性 | 值 |
|------|-----|
| 类型 | 浮点数组 |
| 写入方 | embed程序 |
| 静/动 | 动态（emotion_tags_text更新时重新生成） |
| 适用zone | episodic_short / episodic_long / semantic |

**定义**：emotion_tags_text的语义向量，768维。

**设计原因**：情绪词之间存在语义相似性——"委屈"和"难过"应该能互相检索到。向量比精确匹配更符合情绪联想的模糊性。

**关联字段**：由 `emotion_tags_text` 生成。

**使用场景**：情绪检索模式，检测到情绪信号时触发。

#### `emotion_intensity` · 情绪强度

| 属性 | 值 |
|------|-----|
| 类型 | 浮点数 |
| 写入方 | 从变量系统读取，写入时定 |
| 静/动 | 静态 |
| 适用zone | episodic_short / episodic_long |

**定义**：这条记忆形成时的情绪强烈程度。

**值**：0-1，从ICS数值变量系统的情绪相关变量计算得出。

**设计原因**：情绪强度影响记忆在多个维度的行为：
1. 影响 `detail_clarity` 初始值（极高强度下，未处理时大脑保护性压抑细节）
2. 影响 `emotion_processed` 的上升阻力（强度越高越难消化）
3. 与 `emotion_processed` 联合决定激活时的行为（闯入性记忆概率）

**与emotion_processed的区别**：
- `emotion_intensity`：这件事有多强烈（静态，客观）
- `emotion_processed`：这件事被消化了多少（动态，过程）
- 例：签一份重要合同——significance高，emotion_intensity可能不高；吵一架——emotion_intensity高，significance可能不高

**关联字段**：影响 `detail_clarity` 初始值、`emotion_processed` 上升速度、激活行为计算。

**变化规则**：不变。

**使用场景**：clarity初始值计算、emotion_processed上升速度计算、激活行为多变量计算。

#### `emotion_processed` · 情绪处理程度

| 属性 | 值 |
|------|-----|
| 类型 | 浮点数 |
| 写入方 | 初始固定，后续程序+变量系统计算更新 |
| 静/动 | 动态（缓慢上升） |
| 适用zone | episodic_short / episodic_long / semantic |

**定义**：这条记忆的情绪被角色"消化"的程度，0=完全未消化，1=完全沉淀为过去。

**值**：0-1，初始固定为0。

**设计原因**：情绪处理是一个过程，不是状态。同样的事情，刚发生时情绪鲜活，一年后可能已经成为"那段经历"。这个渐进过程影响：
- 记忆的衰减方式：未处理时detail_clarity快速衰减（保护性模糊），emotion_clarity慢衰减（情绪留存）
- 扭曲的方向倾向：未处理的痛苦更容易向美化漂移
- 激活时的行为：与emotion_intensity联合计算闯入性记忆概率

**普通情绪 vs 强烈情绪的处理**：任何情绪都能被处理，只是速度不同。强烈痛苦的处理阻力由变量系统计算（情绪强度+情绪类型占比），不存在"永远无法处理"，只有极慢速上升。极端情况：emotion_processed长期极低+反复激活 → 角色系统健康值持续下降，可能触发质变事件（崩溃/和解）。

**喜悦的处理**：emotion_processed=1的喜悦意味着从"当下的兴奋"变成"温暖的怀念"——情绪沉淀为认知，可能进入语义记忆。方向不同但机制相同。

**关联字段**：
- 上升速度由变量系统的情绪强度+情绪类型占比决定（不存在记忆条目里）
- 影响 `detail_clarity` / `emotion_clarity` 的衰减速度
- 与 `emotion_intensity` 联合影响激活行为

**变化规则**：
- 触发上升：时间流逝（慢）、角色在日记写到这件事（中）、对话中主动谈起（快）
- 上升速度：变量系统实时计算处理阻力，阻力高则上升慢
- 无法下降（情绪处理是单向的）

**使用场景**：clarity衰减公式、扭曲方向计算、激活行为多变量计算。

#### `location_tag` · 地点标签

| 属性 | 值 |
|------|-----|
| 类型 | 字符串 |
| 写入方 | AI写入，写入时标准化粒度 |
| 静/动 | 静态 |
| 适用zone | episodic_short / episodic_long |

**定义**：记忆发生的地点，AI写入时统一标准化粒度。

**值**：`"咖啡厅"`，而不是 `"星巴克三里屯店"`

**设计原因（粒度问题）**：RIF需要同簇记忆互相竞争，而"簇"的定义来自cluster_id，cluster_id部分来自location_tag。如果粒度太细（每家店独立），RIF几乎不发生；太粗（所有室内场所），压制范围过大。解决方案：AI在写入时就做标准化，把"星巴克三里屯"写成"咖啡厅"，粒度在写入时统一控制，不是检索时处理。

**关联字段**：自动生成 `cluster_id` 的来源之一。

**使用场景**：地点检索模式；cluster_id生成。

**注**：semantic区此字段为null（语义记忆无时间地点）。

### 【分区层】

---

#### `zone` · 记忆所在区

| 属性 | 值 |
|------|-----|
| 类型 | 字符串枚举 |
| 写入方 | AI判断写入 |
| 静/动 | **静态（永远不变）** |
| 适用zone | 全部（这个字段本身定义zone） |

**定义**：这条记忆属于哪个分区，决定其存储形态和注入方式。

**值**：`episodic_short` / `episodic_long` / `semantic` / `implicit` / `working`

**五个分区的含义**：

| zone | 中文 | 特征 | 注入方式 |
|------|------|------|---------|
| episodic_short | 短期情节 | 高可塑，未巩固，细节完整 | 轻量筛选后拼入上下文 |
| episodic_long | 长期情节 | 低可塑，已巩固，有时间地点 | 检索召回 |
| semantic | 语义记忆 | 从情节蒸发，无时间地点，稳定认知 | 检索召回 |
| implicit | 内隐指令 | 行为倾向，不走检索 | 常驻角色配置层 |
| working | 当前状态 | 有时效性，跨对话存在 | 每次对话自动全量注入 |

**设计原因（为什么永远不变）**：层级转化是"产生新认知"，不是"同一件事变稳定"——旧记忆积累后在更高层生长出新条目，原条目打吸收标记保留，支持用户审查和回退。如果zone会改变，历史轨迹就消失了，无法溯源，也无法回退。

**分区的理论依据**：
- 可塑性维度（CLS理论）：短期高可塑，长期低可塑，两者不能共存于同一系统
- 意识参与维度：外显（有意识，走检索）vs 内隐（无意识，绕过检索）
- working区独立于这两条切割线，是实用性补充

**关联字段**：决定哪些专用字段有效（implicit区用trigger_condition，working区用status等）；影响注入方式。

### 【重要性与权重层】

---

#### `significance` · 事件本身重要性

| 属性 | 值 |
|------|-----|
| 类型 | 浮点数 |
| 写入方 | AI写入（结合变量系统数据） |
| 静/动 | **静态（永远不变）** |
| 适用zone | episodic_short / episodic_long / semantic |

**定义**：这件事客观上有多重要，与情绪强度无关，写入时定。

**值**：0-1，AI综合判断：事件的客观影响范围、是否涉及用户、与角色核心关切的相关度。

**与emotion_intensity的区别**：
```
significance   = 事件本身的重量（客观）
emotion_intensity = 事件触发的情绪强度（主观，当时的感受）

重要会议：significance=0.9，emotion_intensity=0.3（角色很平静）
争吵：significance=0.4，emotion_intensity=0.9（情绪很激烈）
```

**设计原因**：weight需要一个稳定的起点，如果weight从第一天就开始衰减，所有记忆的初始状态会不一致。significance作为永久不变的"出生重量"，让weight的演变有可追溯的起点。

**关联字段**：
- `weight` 的初始值 = significance
- `detail_clarity` / `emotion_clarity` 的初始值受significance影响（低于阈值时自动下调）

**变化规则**：不变。

---

#### `weight` · 当前检索权重

| 属性 | 值 |
|------|-----|
| 类型 | 浮点数 |
| 写入方 | 程序计算 |
| 静/动 | **动态** |
| 适用zone | 全部 |

**定义**：这条记忆当前有多容易被检索到。

**值**：0-1，初始值=significance，后续由公式驱动。

**与significance的区别**：significance是出生时的固定重量，weight是当前的浮动状态。同一件事，六个月前的检索权重和今天应该不同。

**变化规则（影响因素）**：

推高weight：
- `recent_activation_density` 高（近期频繁被想起）
- `association_count`（实际已删，通过连锁激活计算代替）

压低weight：
- 时间流逝（自然衰减）
- `suppression_count` 高（被RIF压制）
- `absorption_level` 高（被更高层记忆吸收后自动下降）

**使用场景**：汇总层打分（weight是三条打分维度之一）；高absorption_level时自动下调。

### 【清晰度层】

---

#### `detail_clarity` · 细节清晰度

| 属性 | 值 |
|------|-----|
| 类型 | 浮点数 |
| 写入方 | 程序计算 |
| 静/动 | 动态（衰减快） |
| 适用zone | episodic_short / episodic_long |

**定义**：角色想起这条记忆时，具体细节有多清晰。

**值**：0-1，初始通常为1，significance低于阈值时初始值自动下调。

**为什么和weight分开**：两个维度可以独立变化：
- weight低+clarity高：很难想起，但一旦想起细节清晰（被压抑的记忆）
- weight高+clarity低：容易浮现，但只剩模糊感受（时间久了的重要记忆）

**为什么和emotion_clarity分开**：细节（具体对话、地点、顺序）和情绪轮廓（当时的感受）衰减速度不同。真实人类记忆里，"我记不清说了什么，但我记得当时很难过"是常见体验。

**变化规则**：
- 衰减从 `last_modified_at` 算起（不是created_at——上周被重构过的记忆，衰减重置）
- `reconstructed_count` 越高，衰减越快（反复重构"磨损"细节）
- `emotion_processed` 低时：大脑保护性压抑，detail_clarity衰减加速
- `emotion_processed` 高+`emotion_intensity` 高：可能形成闪光灯效应，detail_clarity反而较高

**使用场景**：注入AI时附带的参数——低clarity时AI呈现"细节模糊，说不清楚"；重构时的输入参数之一。

---

#### `emotion_clarity` · 情绪清晰度

| 属性 | 值 |
|------|-----|
| 类型 | 浮点数 |
| 写入方 | 程序计算 |
| 静/动 | 动态（衰减慢） |
| 适用zone | episodic_short / episodic_long / semantic |

**定义**：角色想起这条记忆时，情绪轮廓有多清晰。

**值**：0-1，初始通常为1。

**衰减速度**：比detail_clarity慢，由变量系统的情绪强度+情绪类型占比调节——情绪越强烈、越单一，emotion_clarity越持久。

**使用场景**：注入AI时附带参数——低clarity时AI呈现"连当时的感觉也淡了"；重构输入参数之一。

### 【演变追踪层】

---

#### `distortion_bias` · 扭曲偏向

| 属性 | 值 |
|------|-----|
| 类型 | 对象数组 |
| 写入方 | 初始为空，程序/AI更新 |
| 静/动 | 动态 |
| 适用zone | episodic_short / episodic_long / semantic |

**定义**：这条记忆当前正在向哪些方向漂移。

**值**：`[{direction: "美化", strength: 0.6}, {direction: "自我服务", strength: 0.3}]`

direction枚举：`美化` / `丑化` / `自我服务` / `叙事一致性`

**设计原因（为什么是数组）**：扭曲可以同时朝多个方向发生——一段痛苦的感情既可以被美化（"他其实还是爱我的"），又在自我服务（"分手是他的错"）。单一枚举无法表达这种复合性，改为带strength的数组结构。

**四个方向的含义**：
- `美化/丑化`：情感色彩的偏移，往更好或更坏的方向扭转
- `自我服务`：朝"让角色自己看起来更好"的方向漂移
- `叙事一致性`：改成符合角色当前自我认知的版本
- `图式驱动`：被角色已有的认知框架拉着走（已归入叙事一致性）

**扭曲 vs 重构的区别**：
- 扭曲：被动，存储期间自然发生，慢慢漂移
- 重构：主动，提取时由小模型生成当下版本，结果不一定写回

**变化规则**：触发条件、幅度、方向判断逻辑待设计（见待设计模块）。

**使用场景**：重构时的方向参数；扭曲机制的执行依据。

#### `reconstructed_count` · 被重构次数

| 属性 | 值 |
|------|-----|
| 类型 | 整数 |
| 写入方 | 程序计数 |
| 静/动 | 动态（单调递增） |
| 适用zone | episodic_short / episodic_long |

**定义**：这条记忆被小模型重构并写回的次数。

**设计原因**：一条被反复重构的记忆，像被传抄了很多遍的手稿，会越来越"像故事"，越来越不像原始事件。reconstructed_count越高，detail_clarity衰减越快，反映这种磨损效应。

**变化规则**：每次重构写回时+1；不会减少。

**使用场景**：detail_clarity衰减速度计算。

### 【激活统计层】

---

#### `activation_count` · 总激活次数

| 属性 | 值 |
|------|-----|
| 类型 | 整数 |
| 写入方 | 程序计数 |
| 静/动 | 动态（单调递增） |
| 适用zone | 全部 |

**定义**：这条记忆被检索到并注入上下文的总次数。

**与recent_activation_density的区别**：
- activation_count：总历史（一条被激活100次vs5次的记忆，即使近期密度相同，前者更根深蒂固）
- recent_activation_density：近期趋势（过去30天的活跃程度）
- 两者共同反映激活历史，缺一不可

**变化规则**：每次激活+1；不会减少。

**使用场景**：weight衰减公式的辅助参数；层级转化触发条件之一。

#### `recent_activation_density` · 近期激活密度

| 属性 | 值 |
|------|-----|
| 类型 | 浮点数 |
| 写入方 | 程序计算 |
| 静/动 | 动态 |
| 适用zone | 全部 |

**定义**：过去30天内被激活的频率（过去30天激活次数÷30）。

**值**：0-1，0=近期完全未被激活，1=每天都被激活。

**设计原因**：比activation_count更能反映"当前阶段的活跃程度"。一条两年前频繁被想起、近期没怎么想的记忆，weight应该已经下降；反之，一条最近频繁涌现的记忆，weight应该较高。

**变化规则**：程序定时重新计算（每天/每次激活后），滑动窗口30天。

**使用场景**：weight衰减公式的核心输入。

| 适用zone | episodic_short / episodic_long / semantic |

#### `derived_from` · 来源条目ID列表

| 属性 | 值 |
|------|-----|
| 类型 | 字符串数组 |
| 写入方 | 程序在巩固/蒸发时写入 |
| 静/动 | 静态（写入后不变） |
| 适用zone | episodic_long / semantic |

**定义**：这条记忆是由哪些下层条目生长出来的。

**使用场景**：溯源（这条长期记忆来自哪些短期记忆）；用户审查时可以查看来源链。

---

#### `absorbed_by` · 被吸收的上层条目ID

| 属性 | 值 |
|------|-----|
| 类型 | 字符串数组 |
| 写入方 | 程序写入 |
| 静/动 | 动态（可能被多次吸收） |
| 适用zone | 全部（被吸收的条目） |

**定义**：这条记忆被哪些更高层的记忆吸收了。

**设计原因（为什么不删除被吸收的记忆）**：
1. 用户可能想审查被吸收的条目，确认新条目是否准确
2. 支持回退操作
3. 历史记录完整保留，系统可追溯
4. 与working区的status逻辑一致——失效的东西打标记保留，不删除

**变化规则**：每次有新条目吸收它时追加；weight随absorption_level高低自动下降，不再被主动检索。

**使用场景**：层级转化溯源；weight自动下降的依据。

---

#### `absorption_level` · 被吸收程度

| 属性 | 值 |
|------|-----|
| 类型 | 浮点数 |
| 写入方 | 程序计算 |
| 静/动 | 动态 |
| 适用zone | 全部（被吸收的条目） |

**定义**：这条记忆被上层记忆覆盖的程度，高时weight自动下降。

**值**：0-1，0=未被吸收，1=完全被覆盖。

**变化规则**：由derived_from条目数量和各条目weight加权计算（具体公式待设计）。

**使用场景**：weight自动下调的触发依据；用户审查时判断是否可以清理。

### 【网络效应层】

---

#### `cluster_id` · 所属记忆簇

| 属性 | 值 |
|------|-----|
| 类型 | 字符串数组 |
| 写入方 | 从标准化tags自动生成 |
| 静/动 | 静态 |
| 适用zone | episodic_short / episodic_long / semantic |

**定义**：这条记忆属于哪些"竞争簇"，同簇记忆在RIF时互相竞争。

**值**：`["咖啡厅", "等待", "NPC_小李"]`，从location_tag + people_tags + 核心情绪词自动提取。

**RIF的逻辑**："同类"不是内容相似，是争夺同一个检索线索的记忆。"咖啡厅"被检索时，所有cluster_id包含"咖啡厅"的记忆互相竞争——命中的那条得分，其余suppression_count+1。

**粒度控制**：粒度在写入时由AI统一控制（location_tag已标准化），不在检索时处理。一条记忆可属于多个cluster。

**使用场景**：RIF机制——每次检索命中时，同簇其他记忆的suppression_count+1。

---

#### `suppression_count` · 被RIF压制次数

| 属性 | 值 |
|------|-----|
| 类型 | 整数 |
| 写入方 | 程序计数 |
| 静/动 | 动态（单调递增） |
| 适用zone | episodic_short / episodic_long / semantic |

**定义**：这条记忆因为RIF机制被同簇竞争记忆压制的总次数。

**设计原因**：RIF（提取诱发遗忘）是真实记忆现象——检索到A，和A共享检索线索的同类记忆B、C会被主动压制。经常被压制的记忆weight加速下降，造成选择性遗忘。

**变化规则**：每次同簇其他记忆被检索命中时+1。

**使用场景**：weight衰减公式的输入（压制次数越多，weight衰减越快）。

---

### 【来源关联层】

---

#### `source_hints` · 来源线索

| 属性 | 值 |
|------|-----|
| 类型 | 对象数组 |
| 写入方 | AI+程序写入 |
| 静/动 | 静态 |
| 适用zone | episodic_short / episodic_long / semantic |

**定义**：这条记忆可能来自哪些原文，提供模糊的溯源线索。

**值**：`[{"type": "timeline", "time_range": "2026-03-10"}, {"type": "chat", "time_range": "2026-03-09"}]`

**设计原因**：记忆不一定对应单一原文（可能是多个事件融合），找不到对应原文是正常的——记忆独立于原文存在。source_hints只是调试和溯源的辅助，不是系统运行的依赖。

**注**：source_confidence（来源可靠程度）已删除——维护成本高，使用频率低，来源混淆效果已由clarity自然体现。

### 【implicit区专用字段】

---

#### `trigger_condition` · 触发条件

| 属性 | 值 |
|------|-----|
| 类型 | 字符串 |
| 写入方 | AI生成 |
| 静/动 | 动态（随经历更新） |
| 适用zone | **仅implicit** |

**定义**：什么情况下这条内隐记忆被触发，影响角色行为。

**值**：自然语言，如 `"对方长时间沉默"` / `"用户迟到或让角色久等"`

**设计原因**：内隐记忆不走检索，但需要告诉角色配置层"在什么情境下这条指令生效"。自然语言比枚举更灵活，AI在生成角色行为时可以判断当前情境是否匹配。

#### `behavior_response` · 行为反应

| 属性 | 值 |
|------|-----|
| 类型 | 字符串 |
| 写入方 | AI生成 |
| 静/动 | 动态（随经历更新） |
| 适用zone | **仅implicit** |

**定义**：触发条件满足时角色的行为倾向。

**值**：自然语言，如 `"不追问，先转移话题，等她自己开口"` / `"表现平静，但话会变少"`

**与episodic记忆的区别**：implicit区不是"角色想起了某件事"，而是"角色不知道为什么就这样做了"——行为绕过意识直接发生。

### 【working区专用字段】

---

#### `relevance_type` · 记忆类型

| 属性 | 值 |
|------|-----|
| 类型 | 字符串枚举 |
| 写入方 | AI判断 |
| 静/动 | 静态 |
| 适用zone | **仅working** |

**定义**：这条working记忆属于哪种类型。

**值**：
- `current_state`：当前进行中的状态（用户感冒、角色在赶项目）
- `near_plan`：近期确定的计划（明天出差、周末约会）
- `pending_action`：待处理的事项（角色答应帮忙查的事、用户想看的电影）

#### `expires_at` · 过期时间

| 属性 | 值 |
|------|-----|
| 类型 | 时间戳或null |
| 写入方 | AI或程序写入 |
| 静/动 | 静态 |
| 适用zone | **仅working** |

**定义**：这条working记忆的时效截止时间，到期后status自动→expired。

**值**：时间戳或null（无明确截止时间时为null，依赖resolve_condition触发）

---

#### `resolve_condition` · 解决触发条件

| 属性 | 值 |
|------|-----|
| 类型 | 对象 |
| 写入方 | AI写入 |
| 静/动 | 静态 |
| 适用zone | **仅working** |

**定义**：什么条件触发这条working记忆变为resolved状态。

**值（结构化，不是自然语言）**：
```json
{"type": "keyword", "value": "感冒好了"}    // 对话中出现关键词
{"type": "time", "value": "2026-04-28"}     // 特定时间
{"type": "manual"}                           // 用户手动标记
{"type": "ai_judge"}                         // AI每轮判断（最贵，慎用）
```

**设计原因（为什么结构化而不是自然语言）**：自然语言条件无法自动判断是否触发——AI每轮都要解读，开销大。结构化后keyword类型可以精确匹配，time类型程序直接比较，manual类型用户操作触发，只有ai_judge才需要AI每轮判断。

---

#### `priority` · 注入优先级

| 属性 | 值 |
|------|-----|
| 类型 | 浮点数 |
| 写入方 | AI或程序写入 |
| 静/动 | 静态 |
| 适用zone | **仅working** |

**定义**：这条working记忆注入上下文时的优先级，working区全量注入但按此排序。

**值**：0-1

---

#### `status` · 状态

| 属性 | 值 |
|------|-----|
| 类型 | 字符串枚举 |
| 写入方 | 程序写入 |
| 静/动 | 动态 |
| 适用zone | **仅working** |

**定义**：这条working记忆的当前状态。

**值**：
- `active`：有效，每次对话自动注入
- `expired`：时间到期，不再注入，保留记录
- `resolved`：事件触发解决，不再注入，保留记录

**设计原因（为什么失效后保留）**：与absorbed_by逻辑一致——失效的东西打标记保留，不删除，用户可查看历史working记忆，支持审查和回退。

## 四、记忆分层

### 4.1 两条独立切割线

```
① 可塑性维度    高可塑（短期）↔ 低可塑（长期）
                本质是学习机制的不同，不是时间的不同
                可塑性和稳定性不能共存于同一系统（CLS理论）

② 意识参与维度  外显（有意识）↔ 内隐（无意识）
                两套完全独立的硬件，互不干扰
```

两条切割线交叉，产生五个分区。

---

### 4.2 五个记忆分区

**① 外显-短期（情节缓冲区）**

```
切割线：高可塑 + 外显
特征：未经巩固，细节完整，随时可被覆盖
巩固机制：睡眠时海马体反复激活，压入皮层，进入长期区
          未被巩固的自然消退

存的内容示例：
"今天下午她在咖啡厅等了我二十分钟，
没有抱怨，但一直看门口"
→ 原始、完整、带细节、有时间戳

注入方式：轻量筛选后直接拼入上下文
```

**② 外显-长期-情节记忆**

```
切割线：低可塑 + 外显 + 有时间地点自我视角
特征：巩固后，细节模糊，情绪轮廓保留
     越抽象的内容越稳定，细节衰减快

存的内容示例：
"有一次她在咖啡厅等我，我迟到了，
她没说什么但眼睛是红的"
→ 细节模糊了，情绪轮廓保留，仍有时间地点

注入方式：通过检索召回
```

**③ 外显-长期-语义记忆**

```
切割线：低可塑 + 外显 + 无时间地点，从情节蒸发
特征：从大量情节中抽象出来，脱离具体事件后独立存在
     最稳定，几乎不衰减
     知道结论，不记得是哪次发现的

存的内容示例：
"她不擅长表达委屈，习惯自己吞下去"
→ 没有时间地点，是稳定认知，是角色对人的理解

注入方式：通过检索召回，也可常驻角色配置
```

**④ 内隐记忆**

```
切割线：无意识，不依赖海马体，独立于可塑性维度之外
特征：无法被语言化描述，只能被"表演"出来
     不走检索，直接影响行为
     意识介入反而干扰

AI扮演的实现方式：
存的不是事件描述，而是行为倾向指令

存的内容示例：
"遇到对方长时间沉默时，
不要追问，先说别的，等她自己开口"
→ 格式接近提示词片段，经历积累后动态更新

注入方式：长期挂在角色prompt配置里
```

**⑤ Working区（当前进行状态与近期计划）**

**设计决策：为什么废弃buffer区，重新设计为working区**

原来的buffer区设计是"启动缓存/神经预热"——当前对话里刚提到某个词，这个词的相关记忆在后台保持激活状态，影响后续回应。这是一个脑科学概念的直接移植。

但这个设计从根本上就错了：**AI对话本身就是这样工作的**。AI每次生成回应时会读取完整的对话历史，"刚刚提到了咖啡厅"这件事已经在上下文里——不需要单独存一个zone来处理"当前对话的预热状态"，这是重复设计。

真正跨对话、需要单独存储的是完全不同的东西：**几天前说的事情、近期的计划、还没有解决的状态**。用户上周说"明天要出差"，这件事不在当前对话上下文里，但角色应该知道。这才是需要专门存储的内容。

```
本质：跨对话的"此刻需要记着的事"，不是过去事件，是当下状态和近期计划

和AI对话上下文的区别：
  上下文 → 这次对话里说了什么（AI已自动处理，不需要单独存）
  working → 跨多次对话都需要知道的状态和计划

三种内容类型（relevance_type）：
  current_state  → 用户/角色当前进行中的状态
                   "用户这几天感冒，昨天说有好转"
                   "角色最近压力很大，在赶一个项目"

  near_plan      → 近期已确定的计划或约定
                   "用户明天要出差，周五回来"
                   "这周末有约会，角色在准备"

  pending_action → 待处理的事项
                   "上次用户说想看某部电影，还没看"
                   "角色答应帮用户查一件事，还没做"

时效管理：
  expires_at       → 时间到期自动将status改为expired（如出差结束日期）
  resolve_condition → 结构化条件触发status改为resolved
  status           → active / expired / resolved
                     失效后不删除，打标签保留，不再注入上下文

注入方式：不走检索，每次对话开始时自动扫描所有active条目
         按priority排序，全部注入上下文（条目数量一般不多）
```

---

### 4.3 原始上下文（不是记忆）

```
最新对话原文，直接拼入上下文，不经过任何记忆流程
与短期情节缓冲区的区别：
```

| | 原始上下文 | 短期情节缓冲区 |
|--|---------|---------|
| 内容 | 最新对话原文 | 近期记忆条目（角色视角） |
| 处理方式 | 直接拼入 | 轻量筛选后拼入 |
| 格式 | 对话消息 | 记忆条目（有tags/weight） |
| 来源 | 只来自对话 | 来自各种原文整理后的记忆 |

---

### 4.4 记忆科学依据（偷来的）

```
RIF提取诱发遗忘    → 检索本身改变权重分布
                    检索到A，同类记忆B/C被主动压制
                    不同线索存储的记忆（D）不受影响

灾难性干扰         → 相似的新经历不是新增一条记忆
                    而是腐蚀、污染已有的长期记忆
                    两段记忆开始混淆

记忆重构           → 每次提取都是重建，不是播放录像
                    当前情绪/场景会污染提取结果
                    实现：检索命中后，小模型注入当前状态重构memory_text

皮层分级存储       → 越抽象越稳定
                    细节（感官/具体对话）衰减快
                    情绪轮廓衰减慢
                    语义认知几乎不衰减
```

- 短期情节缓冲区的时间窗口和日内压缩机制
- 情节→语义的蒸发触发条件
- 内隐记忆的生成和更新规则

## 五、演变概念梳理

### 5.1 机制 vs 方向参数

演变不是一堆并列概念，有层级：

```
机制层（怎么发生）：
  重构  → 提取时发生，每次取出都重新生成，结果写回memory_text
  扭曲  → 存储时自然漂移，存着存着就变了

方向层（往哪个方向变，是扭曲的参数，不是独立机制）：
  美化/丑化      → 情感色彩的方向
  自我服务偏差   → 朝"让自己好看"漂
  叙事一致性     → 朝"角色自洽的故事"漂
  图式驱动       → 被角色已有认知框架拉着走
```

distortion_bias字段记录当前偏向哪个方向，保持扭曲的连贯性。

### 5.2 情绪化的两种方向

情绪强烈对记忆的影响有两种完全相反的结果，都是真实的：

```
情绪被处理过（emotion_processed高）
→ 闪光灯效应：细节反而清晰烙印
→ detail_clarity衰减慢

情绪未被处理（emotion_processed低）
→ 保护性模糊：大脑压抑痛苦细节，只剩情绪轮廓
→ detail_clarity衰减快，emotion_clarity保留
```

不能简单取一种，两种都是角色心理状态的体现。
emotion_processed设计为0-1连续值而非boolean，因为情绪处理是渐进的过程。

### 5.3 层级转化：生长而非搬家

**核心设计决策**：记忆永远不从一个zone移动到另一个zone。

```
不是搬家：M_001的zone从episodic_short改为episodic_long  ✗

而是生长：
  原条目M_001保持zone=episodic_short，打吸收标记
  新生长出M_010，zone=episodic_long，derived_from包含M_001
```

**原因**：
- 层级转化是产生新认知，不是同一件事变稳定
- 原条目仍有独立价值，不应该消失
- 用户可以审查被吸收的条目，支持回退操作
- 历史记录完整保留，系统可追溯

**唯一例外**：episodic_short的相似条目巩固合并时，也是生长新的episodic_long条目，原short条目打吸收标记。zone字段在任何情况下都不改变。

### 5.4 来源混淆

来源混淆不是memory_text内容变了，是source_hints变模糊：

```
原本：来自某次具体对话
混淆后：来源范围扩大，或来源个数增加
       "我好像在好几个地方都感受到过这个"
```

通过source_confidence字段追踪，随时间衰减，反映来源线索的可靠程度。

## 六、动态系统总览

记忆的动态部分由四个主题构成，本轮讨论的核心：

```
演变        → 单条记忆内部怎么变化
              weight衰减、clarity衰减、扭曲、重构

层级转化    → 记忆在不同zone之间的关系
              生长新条目、吸收标记、derived_from

网络效应    → 记忆之间互相影响
              RIF压制、灾难性干扰、连接形成/断裂

激活        → 记忆怎么被触发浮现
              检索召回、连锁激活、不随意记忆、闯入性记忆
```

四个主题的依赖关系：演变是地基，层级转化和网络效应建在上面，激活贯穿全部。

## 七、驱动核与记忆的关系

驱动核不只是"角色是什么样的人"，也决定"角色的记忆系统怎么运作"。

### 7.1 驱动核包含的内容

```
人格向量          → 角色怎么解读事件、怎么反应（核心性格参数）
检索偏好          → 这个角色先被情绪激活还是先被画面激活
连锁激活方式      → 回避型/焦虑型/安全型
习惯模式          → 内隐记忆积累形成的行为倾向（由implicit区记忆动态更新）
```

### 7.2 驱动核的数据结构（草案）

```json
{
  "character_id": "char_001",

  // 人格向量（写入时定，极慢漂移）
  "personality_vector": [...],       // 768维，描述性格的语义向量

  // 检索偏好（决定各检索模式的权重系数）
  "retrieval_preference": {
    "semantic_weight": 0.4,          // 默认权重
    "emotion_weight": 0.3,           // 情绪优先型角色此值更高
    "sensory_weight": 0.15,          // 画面优先型角色此值更高
    "people_weight": 0.1,
    "time_weight": 0.025,
    "location_weight": 0.025
  },

  // 连锁激活方式
  "chain_activation_style": "avoidant",  // avoidant / anxious / secure
  // avoidant → 链条被切断，weight不升反降，话题漂移
  // anxious  → 过度激活，一触即发，大量涌出
  // secure   → 平稳浮现，越聊越清晰

  // 习惯模式（由implicit区记忆汇总生成，定期更新）
  "habit_patterns_summary": "...",   // 当前所有implicit记忆的摘要，注入角色配置层

  "last_updated": "2026-04-22"
}
```

### 7.3 驱动核与记忆系统的双向关系

```
驱动核   → 检索时提供各模式权重系数（检索偏好）
驱动核   → 连锁激活时控制深度和方向
驱动核   → 扭曲时影响distortion_bias的方向倾向

记忆     → 长期激活积累 → 隐性漂移personality_vector（极慢，月/年级别）
内隐记忆 → 直接更新habit_patterns_summary（定期重新生成）
```

### 7.4 检索偏好的人格差异示例

```
情绪优先型（emotion_weight最高）
→ 对话中情绪词优先触发情绪检索
→ 角色容易被情绪共鸣的记忆打动

画面优先型（sensory_weight最高）
→ 感官描述优先触发感官检索
→ 角色容易被场景画面唤起记忆

叙事优先型（semantic_weight更高）
→ 倾向提取"认知"而非"事件"
→ 语义记忆权重相对更高
```

检索偏好存在驱动核里，不在记忆条目里。每次检索时，汇总层的加权公式从驱动核动态读取系数。

### 7.5 待解决

- personality_vector如何初始化？由用户设定角色性格时生成？
- 漂移的具体触发条件和速度参数未定
- retrieval_preference的初始值如何从角色性格推导？

## 八、记忆的预测功能

海马体的核心功能不只是回忆过去，而是**用记忆模拟未来**。

记忆是素材，大脑拿这些素材不断预测"接下来会发生什么"。海马体受损的人不只失去新记忆——他们也无法想象未来，无法做计划，无法模拟"如果我这样做会怎样"。

**记忆的功能不只是检索，还有：**
```
调节情绪    → 难过时主动回忆快乐的事
自我认同    → "我是一个什么样的人"靠记忆支撑
预测未来    → 上次她这样说之后发生了什么，这次可能也是
模拟他人    → 根据记忆推断别人现在在想什么
```

这是记忆系统和角色行为系统之间最深的那条线——记忆不是存档，是角色做决策的基础。

⚠️ 此部分不在本轮讨论范围，后续角色行为系统设计时需要对接。

## 九、ICS整体接口（记忆系统的上下游）

记忆系统不是孤立运行的，它依赖ICS其他模块提供输入，也向其他模块输出。

### 9.1 上游：原文来源

记忆的原材料来自ICS各专项功能模块：

```
时间线（Timeline）    → 角色生活事件的骨架，是最主要的原文来源
日记系统（Diary）     → 角色视角的主观记录，影响memory_text的初始编码
梦境系统（Dream）     → 潜意识内容，写入时需特殊标记（来源类型=dream）
对话记录（Chat）      → 与用户的聊天内容
游戏/约会记录         → 专项互动内容
```

每类原文通过 `source_hints.type` 字段区分来源类型。

### 9.2 写入流程（完整）

```
① 事件发生
   → 写入时间线（摘要形式，系统视角）

② 一天结束
   → 角色写日记（角色视角，主观，有情绪侧重，不完全真实）
   → 日记作为memory_text的主观滤镜

③ 日记完成后，记忆系统整理条目
   → 基于时间线（客观） + 日记（主观）
   → 时间线内容生成 summary（客观）
   → 日记内容/风格影响 memory_text 的初始版本
   → 日记的情绪偏向影响 distortion_bias 的初始值和 emotion_processed

④ 睡眠巩固
   → 短期区条目被筛选、合并，生长出长期区条目
   → 未被巩固的短期条目 weight 降低，逐渐消退

⑤ 长期演变
   → 大量同类情节积累 → 蒸发生长出语义记忆
   → 行为模式重复积累 → 生长出内隐记忆，更新驱动核
```

**关键问题（待解决）**：summary的客观性如何保证？
- 方案A：系统从时间线直接提取，不经过AI（保证客观，但可能缺失细节）
- 方案B：AI根据时间线生成，加严格的客观性约束（灵活但有主观风险）
- 目前倾向方案A，但未确认

### 9.3 下游：记忆输出到哪里

```
外显记忆（episodic/semantic）
  → 检索召回 → 注入对话上下文 → 影响角色当轮回应

内隐记忆（implicit）
  → 常驻角色prompt配置层 → 影响角色所有行为

Working记忆
  → 每次对话自动扫描注入 → 让角色知道"现在正在发生什么"

驱动核（personality_vector + retrieval_preference）
  → 影响检索的权重分配
  → 影响连锁激活的深度和方向
```

### 9.4 与NPC关系网的接口

NPC系统与记忆系统的交互点：

```
people_tags 字段 → 存的是 NPC_ID，与NPC管理系统共享同一套ID
角色的NPC关系     → 对某个NPC的认知存在语义记忆里
                   （"她不擅长表达委屈"这类语义记忆本质是对NPC的认知）

社会传染效应（见下一节）→ NPC言语可以改写角色记忆
```

## 十、社会传染：NPC言语对记忆的影响

**定义**：角色听到NPC对某件事的描述，可能改变角色自己对该事件的记忆。

**真实案例**：
```
用户事后告诉角色："你那天看起来很难过"
→ 角色原本的记忆：那天只是有点累
→ 社会传染后：角色开始相信自己那天确实很难过
→ memory_text 向"难过"方向漂移
→ emotion_tags_text 可能新增"难过"
```

**实现方式**：
```
NPC/用户的言语描述 → 作为一种特殊的"重构触发"
→ 注入的不是"当前角色情绪"而是"他人对该事件的解读"
→ 按照扭曲机制的方向影响 memory_text
→ distortion_bias 可能被这次社会传染改变
```

**影响强度因子**：
- 说话者的信任度（trust_level）越高，影响越大
- 重复被说越多次，影响越稳固
- 时间越久的记忆越容易被社会传染改写（detail_clarity低时抵抗力弱）

**与灾难性干扰的区别**：
```
灾难性干扰  → 相似的新亲身经历腐蚀旧记忆（自身经历之间的干扰）
社会传染    → 他人言语改写自己的记忆（外部信息的干扰）
```

⚠️ 社会传染的具体触发条件和强度计算待设计，目前归入待设计模块。

## 十一、元记忆

**定义**：角色对自己记忆状态的认知——知道自己记得什么、不记得什么、记得多清楚。

**主要表现**：
```
舌尖现象    → "我知道我知道这个，但就是想不起来"
不确定感    → "我记得，但不太确定是不是真的这样发生的"
识别感      → "这件事我以前见过/经历过"
遗忘感知    → "我好像忘了什么，但说不上来是什么"
```

**在ICS中的位置**：
元记忆不是一条具体的记忆，而是角色对记忆系统状态的感知。它属于驱动核的一部分，是角色自我认知的组成部分。

**实现方式（草案）**：
```
不需要单独存储，通过现有字段推导：
  - weight < 0.3 + detail_clarity < 0.2 → "我有印象但记不清了"
  - activation_count高但detail_clarity低 → "我总觉得记得，但细节对不上"
  - source_hints为空或范围很广 → "我不确定是不是真的发生过"
```

AI根据这些元信息自然地呈现角色的"记得的不确定感"，而不是每次都精确地回忆或完全不知道。

⚠️ 元记忆的具体实现参数待确认。

## 十二、重构机制的具体实现

重构发生在**提取时**，不是存储时。

**为什么重构结果不立刻写回memory_text**

这是一个重要的设计决策。如果每次检索到记忆，重构后立刻写回，会导致：
- 一次偶然的对话（角色当时心情很差）就永久改写了记忆的内容
- 重构是根据当前情绪生成的"此刻版本"，不一定代表记忆真正应该演变的方向
- 频繁写回会导致memory_text快速"磨损"，reconstructed_count暴涨

因此设计为：重构结果只用于当次对话注入，对话结束后根据偏差程度决定是否写回。只有当重构结果与原memory_text偏差足够大（超过阈值），才视为"这次重构反映了真实的演变方向"，写回并更新last_modified_at。

```
检索命中memory_text
        ↓
注入当前状态参数：
  - 角色当前情绪（从变量系统读取）
  - 当前对话场景
  - distortion_bias（已有的扭曲偏向列表）
  - detail_clarity / emotion_clarity（清晰度）
        ↓
小模型重构，生成"此刻角色想起的版本"
        ↓
重构结果注入对话上下文（不立刻写回memory_text）
        ↓
对话结束后，计算重构结果与原memory_text的偏差
若偏差超过阈值 → 写回memory_text，reconstructed_count+1，last_modified_at更新
若偏差未超阈值 → 不写回，本次重构不留痕迹
```

**重构方向由当前状态决定：**
```
角色现在很难过  → 重构时过滤掉当时的快乐细节
时间过去很久    → 细节模糊，情绪轮廓保留
distortion_bias = [{direction:"美化",strength:0.6}] → 往更好的方向重构
反复重构        → 越来越像"故事"，越来越不像"事件"
```

**重构 vs 扭曲的区别：**
```
重构  → 提取时发生，主动，由当前状态驱动，结果不一定写回
扭曲  → 存储期间自然发生，被动，慢慢漂移，结果直接改变memory_text

两者都会改变memory_text，但触发时机、机制和持久性完全不同
```

## 十三、RIF与cluster_id的逻辑

RIF（提取诱发遗忘）的关键：**"同类"不是内容相似，是争夺同一个检索线索的记忆。**

```
提取线索是"咖啡厅"
→ 所有绑定咖啡厅这个线索的记忆都是竞争者
→ 检索到A，其他竞争者被压制
→ 和咖啡厅无关、靠情绪线索存储的记忆D不受影响
```

cluster_id就是"共享同一检索线索"的记忆簇标识，同簇的记忆互相竞争。

**实现逻辑：**
```
每次检索命中A
→ 找到A的cluster_id
→ 同cluster的其他记忆suppression_count+1
→ suppression_count影响weight衰减速度
→ 被频繁压制的记忆越来越难浮现
```

一条记忆可以属于多个cluster（咖啡厅+她+等待），被命中时只压制同一条检索线索对应cluster里的竞争者。

## 十四、通道层（扩展功能）

**存在条件**：原始输入信号不够用，需要AI转化才能被检索层利用。

**为什么通道层不是所有输入的必经入口**

早期曾把通道层理解为"所有输入的前置处理器"，这是错误的。

通道层存在的唯一理由是：**某些输入的字面含义不足以触发有效检索**。感官描述（"草木香"）、隐喻（"像刺猬一样"）、身体感受（"胃突然紧"）——这些如果直接进检索层，语义太贫乏，检索精度很差。通道层的工作就是把这类输入丰富化，然后再送去检索。

而人名、时间词、语义内容，字面含义本身就足够用来检索，完全不需要AI转化，直接进对应检索模式。通道层对这些输入没有任何作用。

所以正确的关系是：

```
感官描述/隐喻/身体感受 → 通道层（AI丰富化）→ 广播给各检索模式
人名/时间词/语义内容   → 直接进对应检索模式（不经过通道层）
```

**开销**：每次需要调用AI推断，费token，可选择小模型处理。

**不开扩展时**：感官类输入降级为sensory_tags精确匹配，其他输入不受影响。

### 目前识别的通道

| 通道 | 输入 | 转化方式 | 输出 |
|------|------|---------|------|
| 感知通道 | "草木香"、"晨光" | AI补全感官维度 | smell/visual/touch/sound信号包 |
| 隐喻通道 | "像刺猬一样" | AI解读情绪含义 | 防御、受伤、不想被碰 |
| 梦境通道 | "梦见笼子" | AI解读象征含义 | 困住、失控、渴望逃离 |
| 身体感受通道 | "胃突然紧" | AI解读内部信号 | 焦虑、预感 |

通道层输出的信号包广播给所有检索模式，各模式按需取用。

## 十六、汇总层：取舍与排布

### 6.1 多模式汇总

各模式独立检索，返回各自Top N，汇总去重后计算综合分数：

```
最终分数 = 语义相似度 × 0.4
         + 被命中的模式数量 × 加权系数
         + 记忆weight字段
         - 时间衰减系数
```

被多个模式同时命中的记忆自动排到前面。权重系数从驱动核的retrieval_preference动态读取，不是硬编码。

### 6.2 取舍规则

**5条上限的设计依据**

5条是综合考虑token消耗和实际效果的初始参数，做成可配置，使用后根据效果调整。

implicit区和working区**不占这5条名额**：
- implicit：常驻角色配置层，不走检索，不通过汇总层
- working：每次对话开始时自动全量注入，也不走汇总层

因此5条全部来自episodic/semantic区的检索结果，是真正的"此刻想起了什么"。

```
注入上限：5条（可配置参数）

保留策略：
→ 分数最高的前3条（强相关）
→ weight最高的1条（重要但当前相关度可能不高）
→ 最近发生的1条（时效性保底）

去掉：
→ 内容高度重叠的（保留分数高的那条）
→ weight极低且分数也低的（已高度模糊的记忆）
```

**5条中各zone的比例**：目前未定。是否需要保证每种zone至少有1条（如至少1条semantic），还是完全按分数排序？待使用后确认。

### 6.3 注入排布

```
[强相关记忆 1-3条]   ← 最前，AI优先参考
[重要记忆 1条]       ← 标注"对角色很重要"
[近期记忆 1条]       ← 标注"最近发生的"
```

注入prompt格式：
```
以下是角色可能在此刻想起的记忆，按相关程度排列：
1. [摘要]（情绪：xxx，时间：xxx）
2. [摘要]（情绪：xxx，重要）
3. [摘要]（情绪：xxx，最近）

角色不需要全部提及，自然地融入对话即可。
```

## 十七、连锁激活机制

**实现方式：实时相似度计算，不维护静态关联列表**

早期设计有 `linked_memories` 字段，存储关联记忆的ID列表。后来发现这个设计有根本性的问题：

1. 新记忆写入时，需要扫描所有现有记忆来建立连接——开销大，而且新记忆和旧记忆之间的关联是动态的，难以准确预设
2. 关联列表需要持续维护，记忆被删除或absorbed时，所有引用它的linked_memories都要更新
3. 静态列表无法反映"关联程度"——两条记忆可能非常相关，也可能稍微相关，列表只能存"有/没有"

**改为实时计算**：激活记忆A后，用A的summary_vector / emotion_vector / people_tags / cluster_id对记忆库做一次相似度查询，最相近的N条自然就是连锁激活的候选。

优点：
- 新记忆写入后自动和相关旧记忆产生关联，无需维护
- 关联程度连续（相似度分数），而不是二元的有/没有
- 记忆被修改或删除，连锁计算自动反映最新状态

**连锁激活的行为**

不是一次性全部涌出，随对话持续逐渐浮现：

```
第一轮对话 → 检索到记忆A，计算A的相关记忆候选
A被提及后 → A的weight临时上升
第二轮对话 → 相关候选的检索权重提高，开始浮现
越聊越清晰（安全型人格）
```

**连锁激活受驱动核影响**：

```
回避型 → 相关记忆weight不升反降，越聊越模糊，话题自动漂移
焦虑型 → 连锁过度激活，相关记忆大量涌出，难以控制
安全型 → 平稳浮现，越聊越清晰，有自然的深度
```

**深度限制**：连锁不能无限展开，需要设置最大层数（待设计具体参数）。

## 十八、原始内容关联（模糊溯源）

记忆不一定对应单一原文，来源关联是**模糊的线索**，不是精确ID：

```
记忆条目
  → source_hints（时间范围 + 内容类型）
  → 大致定位原文区间：时间线 / 对话记录 / 梦境记录
  → 可能找到，也可能找不到对应原文——这是正常的
```

**溯源只用于**：用户想"查看当时发生了什么"时提供参考线索，不是系统运行的必要依赖。记忆本身是独立存在的，不依赖原文。

## 十九、性能估算

**字段数量：40个**（详见3.1记忆条目字段，含id共40个）

| 指标 | 数据 | 说明 |
|------|------|------|
| 三个向量 | ~14KB | 768维×3，每个约4.6KB |
| 文本字段 | ~3-5KB | summary+memory_text+tags等 |
| 数值/标签字段 | ~2KB | 41个字段中的结构化数据 |
| **每条记忆合计** | **~20-22KB** | |
| 每天产生条目 | 5-15条 | 只存有意义的节点 |
| 单角色舒适上限 | ~50000条 | 约13年，之后可清理低weight条目 |
| 查询耗时（本地embed） | 20-55ms | 推荐方案 |
| 查询耗时（API embed） | 120-250ms | 有成本，有延迟 |
| 真正的瓶颈 | embed模型选择 | 推荐本地`bge-m3`，中文友好，免费 |

⚠️ 字段增多主要是标量字段（数字/布尔），对存储影响有限，主要开销仍在三个向量上。

## 十八、设计疑问与待确认问题

以下是文档整理过程中发现的潜在矛盾或未达成明确共识的地方，需要专项讨论：

### 疑问1：summary由谁生成？客观性如何保证？
- 方案A：系统从时间线直接提取关键信息（客观，但可能机械）
- 方案B：AI根据时间线生成，加客观性约束（灵活，但有主观风险）
- 未解决：如果AI生成，怎么确保summary不带角色色彩？

### 疑问2：distortion_bias单字段够用吗？
- 当前设计：单一枚举值（none/美化/丑化/自我服务/叙事一致性）
- 潜在问题：美化和自我服务偏差可能同时存在
- 待讨论：是否需要改为数组，允许多个方向并存？

### 疑问3：日内压缩 vs zone永不变
- 问题：短期区日内压缩时，多条short条目合并生长出新条目
- 新条目的zone是short还是long？
- 如果是short：等待当晚睡眠巩固时再生长出long
- 如果是long：绕过了短期区，是否符合CLS理论？

### 疑问4：5条注入上限的合理性
- implicit区常驻prompt，不占检索注入名额
- working区自动扫描注入，也不占检索名额
- 实际可检索注入的只有5条（均来自episodic/semantic区）
- 暂定5条，做成参数，使用后再调整

### 疑问5：小模型重构的具体触发时机
- 当前描述：每次检索到记忆后触发重构
- 潜在问题：每次对话都触发重构会不会太贵？
- 待讨论：是否只在 detail_clarity 低于某阈值时才调用小模型重构？

### 疑问6：cluster_id的自动生成规则
- 问题：cluster_id已改为数组，但谁来生成？什么时候生成？
- 方案A：写入时AI自动提取关键词作为cluster标签
- 方案B：系统根据people_tags/location_tag/emotion_tags自动生成
- 未解决：cluster粒度怎么控制？"咖啡厅"和"星巴克"是同一个cluster吗？
