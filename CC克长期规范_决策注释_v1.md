## 一、为什么有这份规范

代码库里真正难懂的,从来不是"这段代码做了什么"——那些通过变量名、函数名、类型签名都能看出来。真正难懂的是**"为什么这样做"**:

- 为什么这个值是 30 不是 60
- 为什么这里用同步而不是异步
- 为什么这个看似多余的判断不能删
- 为什么这个 API 用旧接口而不是新接口
- 为什么这里的算法是 O(n²) 不是 O(n log n)

这些问题的答案,**只有写代码的当下才知道**。三个月后回来看,自己也会忘。半年后另一个人接手,完全无从下手。

代码库里好的注释保留四类决策上下文:

1. **业务上下文**:这个业务规则为什么存在
2. **历史上下文**:为什么当初选了这种实现而不是别的
3. **约束上下文**:有哪些现实限制影响了这个决策(API 限制、性能限制、用户习惯、外部依赖等)
4. **未来上下文**:未来计划如何改进或在什么情况下要重新考虑

没有这些信息,所有代码看起来都像随意写的。当系统要扩展时,后来人不知道改动会破坏什么平衡,只好凭感觉。然后 bug 就出现了。

## 二、什么时候必须写"为什么"注释

满足以下任一条件,提交时必须附带"为什么"注释:

### 2.1 魔法数字 / 特殊常量

任何不是显而易见的数值。

```typescript
// ❌ 不好
const RETRY_LIMIT = 3;

// ✅ 好
// 重试 3 次:Anthropic API 的 5xx 错误大多是瞬时的,3 次足够覆盖。
// 超过 3 次基本是配置或网络问题,继续重试只会浪费配额。
const RETRY_LIMIT = 3;
```

### 2.2 在多个合理方案中选了一个

```typescript
// ❌ 不好
async function saveMemoryDraft(content: string) { ... }

// ✅ 好
// 选择同步写入而不是异步队列:
// 记忆 draft 必须在用户消息显示前完成,否则对话有时间差感。
// 异步队列在压力测试下出现过 200ms+ 延迟,用户能感知到。
// 如果未来 draft 写入耗时超过 50ms,需要重新评估这个决策。
async function saveMemoryDraft(content: string) { ... }
```

### 2.3 看起来"多余"但不能删的代码

```typescript
// ❌ 不好
if (mode === 'online' && !value) {
  return null;
}

// ✅ 好
// 在线模式下 value 可能为空(用户首次进入,数值还未初始化),
// 这种情况返回 null 让前端走默认渲染,而不是报错。
// 不要改成抛异常——在线模式下数值为空是正常状态,不是错误。
if (mode === 'online' && !value) {
  return null;
}
```

### 2.4 临时方案 / 已知不完美的实现

明确标注为什么是临时的、什么情况下应该改。

```typescript
// ✅ 好
// 临时方案:目前用全表扫描查 condition_subscriptions。
// 当事件数 < 50 时无性能问题,但超过 200 必须改成 indexed lookup。
// 重构计划:路标 5 之后,事件系统稳定时一并优化。
const subs = await db.select().from(conditionSubscriptions);
```

### 2.5 因为外部约束而做的妥协

API 限制、第三方库行为、平台差异、用户习惯——任何"按理说不该这样写但没办法"的代码。

```typescript
// ✅ 好
// Anthropic messages API 的 system 是独立字段不在 messages 数组里,
// 所以这里要单独处理,不能用 OpenAI 兼容客户端的统一逻辑。
// 这是 API 结构差异,不是我们的设计选择。
const systemPrompt = extractSystem(messages);
```

### 2.6 与设计文档不一致的偏离

如果实现与设计文档(共识文档 v1、动态系统设计等)有偏离,**必须**在偏离处写注释,说明:

- 偏离的具体内容
- 为什么偏离(发现了文档没考虑的情况、文档过时、技术限制等)
- 是否需要回写文档

```typescript
// ✅ 好
// 与共识文档 4.4 节偏离:
// 文档里 Trigger 类型库列了 ai_output_match,
// 但实际实现时发现需要拆成 ai_output_regex / ai_output_keyword 两类,
// 因为它们的订阅匹配性能特性完全不同(正则要全文扫描,关键词可以建倒排索引)。
// 待 Agnes 确认后回写文档。
```

## 三、什么时候不需要写

避免无意义注释。以下情况不需要:

### 3.1 代码本身已经清晰

```typescript
// ❌ 多余
// 把 user 的 name 字段取出来
const userName = user.name;

// ❌ 多余
// 遍历每个事件并触发
events.forEach(event => trigger(event));
```

如果通过变量名、函数名、类型就能看出"是什么",不需要注释重复说明。

### 3.2 标准模式

业内通用模式(单例、观察者、useState 等)不需要解释为什么用——除非你**没**用它而用了别的,那时要解释为什么没用标准模式。

### 3.3 一目了然的工具函数

```typescript
// ❌ 多余
// 把数组求和
function sum(arr: number[]) { return arr.reduce((a, b) => a + b, 0); }
```

## 四、注释格式约定

### 4.1 位置

- **文件级决策**:写在文件顶部
- **函数级决策**:写在函数定义上方
- **行级决策**:写在该行上方,不写在行尾

```typescript
// ✅ 好
// 这里强制 await 而不是 fire-and-forget:
// 写入失败时上层需要回滚 UI 状态,不能静默失败。
await persistMessage(msg);

// ❌ 不好
await persistMessage(msg);  // 必须 await 否则 UI 会出错
```

### 4.2 长度

- 短决策:1-3 行,直接说清楚
- 复杂决策:用块注释,分段写,每段一类上下文(业务/历史/约束/未来)
- 超长决策(超过 20 行):说明这个决策值得有专项设计文档,在注释里**链接到文档**而不是把文档抄进注释

### 4.3 语气

- 直接陈述事实和理由,不需要"我认为""可能""也许"
- 不要给"未来的自己"或"未来的同事"留鸡汤式留言("这里要小心哦~""请仔细思考~"),直接讲技术内容
- 用现在时,不要用"当时""那时"——读注释的人不关心你写它的时间,关心它现在还成不成立

## 五、提交时的强制要求

### 5.1 PR/Commit 描述

每次提交代码到主分支前,commit message 或 PR 描述里**必须有一段"为什么这样做"**:

```
[功能/修复简述]

为什么这样做:
- [核心决策点 1 及理由]
- [核心决策点 2 及理由]

偏离设计文档:
- [如有偏离,列出;如无,写"无偏离"]

测试:
- [如何验证这次改动是对的]
```

### 5.2 自检

每次提交前,自己回答两个问题:

1. **三个月后我自己回来看这段代码,能否在不重新调试的情况下理解为什么这样写?** 如果不能,加注释。
2. **如果 Agnes 现在问我"为什么这里用 X 不用 Y",我能否立刻回答?** 如果需要现想,就需要注释。

## 六、回顾机制

每月做一次"决策回顾":

- 选 1-3 个本月做过的重要决策
- 写一段说明:做了什么、为什么这样选、当时排除了哪些其他方案
- 如果这个决策已经在代码注释里,直接引用注释位置即可
- 如果没有,补一段——这就是发现遗漏注释的机制

回顾结果汇总到一个长期累积的 `DECISIONS.md` 文件,按时间倒序排,新决策加在最上面。

## 七、与共识文档的分工

为避免重复:

- **架构层决策**(子系统边界、数据流、设计原则、各方向的归属)——记在共识文档和专项设计文档里
- **代码层决策**(具体实现选择、参数取值、临时方案、性能取舍)——记在代码注释和 DECISIONS.md 里

如果一个决策同时跨架构层和代码层,两边都要有,但代码注释里要**链接到架构文档**,不要复制粘贴。

```typescript
// 这里走 Trigger/Effect 基础设施而不是事件系统内部实现,
// 详见共识文档 v1 第 4.4 节。
// 本地实现细节:订阅时根据 trigger 的 source 字段分桶存储,
// 提升匹配性能。
const trigger = registerTrigger({ ... });
```

## 八、不需要做的事

明确说几件不要做的:

### 8.1 不要回头给所有已有代码补注释

存量代码的"为什么"应该靠**架构层文档**承载(共识文档、专项设计文档)。事后给散落的代码补注释,要么写出空话要么写出虚构理由,价值低于零。

新代码必须按规范写注释,旧代码遇到要修改的时候顺便加。**不专门组织一次"补注释 sprint"。**

### 8.2 不要把"为什么"注释写成自我辩护

```typescript
// ❌ 不好(辩护语气)
// 我知道这看起来不优雅,但是因为时间紧没办法只能这样写,
// 后面有时间一定会重构...

// ✅ 好(技术陈述)
// 临时方案:目前同步处理足够覆盖单用户场景。
// 重构触发条件:并发用户数 > 10,届时改为消息队列。
```

### 8.3 不要用注释代替命名

如果一个变量叫 `x` 然后注释解释它是"用户最近一次活动时间",**改名字**(`lastActiveAt`),不要靠注释救。注释解释"为什么",名字解释"是什么"。

### 8.4 不要在注释里抱怨代码

```typescript
// ❌ 不好
// 这个第三方库的 API 设计真垃圾,只能这样用

// ✅ 好
// xxx 库的 query() 不支持参数化,所以这里必须手动转义。
// 升级到 v3 后可以用 queryWithParams 替代。
```

## 九、生效与执行

### 9.1 生效条件

本规范在以下条件全部满足后启动:

- 路标 3(双模式 bug 修复 + Trigger/Effect 架构决策)完成
- 共识文档稳定到 v2 及以后
- Agnes 确认开始执行

启动前的提交,按现有方式即可。

### 9.2 执行后的协作

- Agnes 验收新代码时,会随机抽查 commit message 和代码注释,看"为什么"是否到位
- 如果发现新代码没有"为什么"注释或注释只重复了"是什么",会要求补
- 三次以上违反会触发协作模式重新讨论

### 9.3 弹性

如果某次提交的代码量很大,逐行写"为什么"不现实——挑**最重要的 5-10 个决策点**写到位即可,其他在 commit message 里集中说明。质量优先于覆盖率。

## 十、最后

写注释的时间永远比"未来某个人重新理解这段代码"的时间短。这不是负担,是杠杆。

而且这个"未来某个人",最常见的是**三个月后的你自己**。
