# ICS 设计讨论记录（完整版）

## 一、基础架构决策

### 1.1 关于 Tool/Skill/MCP/Agent

**讨论过程：**
用户问这些概念有什么区别，以及ICS需不需要学习和集成这些。

**核心区别：**
| 概念 | 本质 | 类比 |
|------|------|------|
| Tool | 单一原子能力 | 一把锤子 |
| Skill | 一组工具+使用说明 | 木工手册+工具箱 |
| MCP | 工具的标准化接口协议 | USB-C接口标准 |
| Agent | 能自主决策调用工具的LLM | 会用工具的工人 |

**最终决策：不需要学这些标准化框架**

ICS是垂直产品，需要的是背后的思想，不是标准实现：

| 概念 | 背后的思想 | ICS需要吗 | 怎么做 |
|------|-----------|----------|--------|
| Tool | 模型能触发外部操作 | ✅ 需要 | 自己写函数，模型输出特定格式触发 |
| Skill | 一组能力+使用说明 | ❌ 不需要 | ICS只有一个领域 |
| MCP | 标准化连接外部服务 | ❌ 不需要 | 数据都是app内的 |
| Agent | 循环决策-执行-观察 | ✅ 需要 | 自己写状态机，用于生活模拟 |

### 1.2 后端技术选型

**讨论过程：**
1. 用户说纯本地app，没有云服务器
2. 前后端分离，client和server文件夹
3. 每次npm run dev，去网页看效果 → 确认是Node.js
4. 目前是JS/JSX

**关于JS→TS迁移：**
- 用户说项目"千疮百孔"
- 讨论后决定：趁乱一起重构，迁移到TypeScript
- 用户已经让CC克开始迁移了

**最终决策（2026-04-11 实施后更新）：**
- 数据库：SQLite
- 驱动：better-sqlite3（同步接口，Windows 有预编译二进制，Node.js 22 + ESM 直接支持）
- ~~ORM：Drizzle~~（暂不引入，理由如下）

**关于 Drizzle 的决策说明：**

原计划选 Drizzle 是假设会写列级查询（如 `SELECT * FROM characters WHERE id = ?`）。实际实现采用了"通用 blob 存储"模式，每张表只有 4 列（`id, char_id, data TEXT, created_at`），数据全量存为 JSON blob。Drizzle 的类型安全针对列级操作——blob 模式下没有列可约束，引入 ORM 意义不大。

**何时重新考虑 Drizzle：** 如果未来 `character_values` 等需要范围查询（`WHERE current_value BETWEEN min AND max`）的表开始开发，那时候针对那几张表引入 Drizzle。

### 3.2 总结表 summaries

**讨论过程：**

1. **要不要分四张表？**
   - 最初用户想分：日总结、模式总结、固定条数总结、自定义总结
   - 结论：不用分，用 `type` 字段区分，一张表就够
   - 原因：存储上一样，查询时用筛选即可

2. **外键放哪边？**
   - 结论：总结指向消息（总结记录"我覆盖了哪些消息"）
   - 这样删消息时总结还在

3. **用消息ID还是时间范围？**
   - 讨论了两种方案的优缺点
   - 用时间：消息删了时间范围还有意义，ID可以重排
   - 用ID：更简单，消息有三个时间字段会造成歧义
   - 最终决策：**用ID，不设外键约束**
   - 补充：ID不需要重排，有洞就有洞，数据库不在乎

4. **要不要设外键约束？**
   - 外键是可选的约束，不是自动的
   - 设外键约束 → 删消息时数据库报错或级联删除
   - 不设外键约束 → ID就是普通整数，删消息不影响总结
   - 最终决策：**不设外键约束**，保持数据独立

**最终设计：**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | 主键 | |
| type | 枚举 | daily / mode_switch / fixed_count / custom |
| content | 文本 | 总结内容 |
| start_message_id | 整数 | 覆盖的起始消息（不设外键约束） |
| end_message_id | 整数 | 覆盖的结束消息（不设外键约束） |
| created_at | 时间戳 | |

**删除逻辑：**
- 删单条消息 → 总结不动
- 删一段消息 → 总结不动，查询时按范围查，中间有删的就跳过
- 重新总结 → 检查消息是否存在，存在就重新生成
- 全删 → 总结变成"孤儿"但内容保留（作为残留记忆）

**前端需要加：重新总结按钮**

**关于递归扫描的理解：**

用户问：把A设全局开启放目录，BCD关键词分别是目录之一，递归可以只触发B吗？

答案：**不能，会全部触发BCD。**

递归的逻辑：
```
第一轮：扫描聊天记录 → 找到匹配关键词 → 激活条目
第二轮：扫描第一轮激活的条目内容 → 找到新关键词 → 激活更多条目
```

例子：
```
目录条目（常驻），内容："本书包含：第一章、第二章、第三章"
条目B，关键词："第一章"
条目C，关键词："第二章"
条目D，关键词："第三章"

结果：目录常驻 → 递归扫描 → 发现三个关键词 → BCD全部激活
```

**递归的正确用途：关联展开**
```
条目A，关键词："小明"，内容："小明的老师是张教授"
条目B，关键词："张教授"，内容："张教授是火系魔法大师"

提到"小明" → 激活A → 递归发现"张教授" → 激活B
一句话带出整个知识网络
```

**想做章节选择怎么办？**
| 想要的效果 | 用什么 |
|-------------|--------|
| 提到A就展开A相关的一堆东西 | 递归 |
| 提到哪章就只激活哪章 | 普通关键词（不用递归） |
| 只能同时激活一个章节 | 互斥组 |
| 章节按顺序解锁 | delay |

### 4.1 与事件系统的关系

```
┌─────────────────┐      ┌─────────────────┐
│    数值系统      │      │    事件系统      │
├─────────────────┤      ├─────────────────┤
```

### 4.4 提示词动态注入

**讨论过程：**
1. 用户提到SillyTavern有个EJS语法
2. 我搜索后发现是第三方扩展ST-Prompt-Template
3. 它用EJS（Embedded JavaScript）在提示词里写逻辑
4. ICS可以用更简单的占位符替换实现

**用户的问题：**
- 表从哪里来？→ 需要做编辑器UI，作者在前端填写，存入数据库
- 提示词如何动态注入？→ 占位符替换
- 变量参数放哪里？→ 数据库（定义和当前值都在数据库）
- 怎么识别变量位置？→ 正则匹配 `{{...}}`

**实现方案：占位符替换**

提示词里写：
```
{{char}}对{{user}}的态度：{{v:affection:desc}}
```

后端发送前：
1. 正则匹配找到所有 `{{...}}`
2. 查数据库拿到值
3. 替换回去

```javascript
// 伪代码
const VAR_PATTERN = /\{\{(\w+)\}\}/g;
// 匹配到 "v:affection:desc"
// 解析：v = 数值变量，affection = 变量名，desc = 取描述
// 查询 character_values 找到好感度当前值
// 查询 value_stages 找到对应阶段的description
// 替换
```

**完整流程：**
```
1. 作者在编辑器里创建数值、设置阶段、写提示词片段 → 存入数据库
2. 作者在提示词里写占位符 {{v:affection:desc}}
3. 用户聊天时，后端：
   a. 读取提示词
   b. 读取当前数值
   c. 根据数值查对应阶段
   d. 替换占位符
   e. 发送给API
4. AI回复后，后端根据规则更新数值
```

## 五、事件系统

### 5.1 三种"事件"来源

| 类型 | 谁设计 | 什么时候产生 | 例子 |
|------|--------|-------------|------|
| 预设事件 | 作者 | 提前写好 | 主线剧情、表白节点 |
| 随机事件 | 系统 | 角色生活时随机触发 | 偶遇邻居、下雨了 |
| 生成事件 | AI/角色 | 对话中产生意图 | "想送你礼物" → 生成计划 |

### 5.2 角色计划（第三种事件）

**用户的设想：**
角色在对话中产生意图 → 系统记录为"计划" → 后台执行 → 适时提醒 → 完成闭环

**例子：**
```
对话中角色说"想送你礼物"
      ↓
系统捕捉意图，生成一条计划
      ↓
┌──────────────────────────────┐
│ 计划：送礼物                  │
│ 状态：准备中                  │
│ 步骤：                        │
│   1. 去商场买 [待执行]        │
│   2. 送给user [待执行]        │
│ 预计完成：3月20日             │
└──────────────────────────────┘
      ↓
角色生活模拟时，执行步骤1
生成生活片段："3月18日下午去了商场"
      ↓
下次聊天，系统注入提醒："角色准备了东西想给你..."
      ↓
角色送出礼物，计划标记完成
```

**这不是备忘录功能**：
| | 用户备忘录 | 角色计划 |
|---|----------|---------|
| 谁的 | 用户 | 角色 |
| 谁执行 | 用户自己 | AI模拟执行 |
| 用户能看吗 | 能 | 不一定（惊喜） |
| 存在哪 | 用户数据 | 角色状态 |

**好处：** 因为存在系统里，角色不会每轮都提及，不会"记不住事"，对用户有惊喜感。

### 5.3 拆分后相比酒馆世界书的优势

酒馆世界书的本质：**关键词 → 插入文本**

酒馆做不到的事：
| 场景 | 酒馆世界书 | ICS |
|------|-----------|-----|
| 好感度累积 | ❌ 没有变量概念 | ✅ 数值系统存着，规则加减 |
| "连续三天没聊天，角色生气" | ❌ 只能匹配当前对话 | ✅ 事件系统记录时间 |
| "第一次送花热烈，第二次淡定" | ❌ 没有历史记录 | ✅ 事件系统记录次数 |
| "理智低于30说话变阴暗" | ❌ 没法判断数值 | ✅ 规则检查数值 |
| "角色自己决定今天去哪" | ❌ 完全做不到 | ✅ 规则+数值+事件驱动 |

## 八、数据库实施 Q&A（2026-04-11）

### Q1：messages 表字段如何处理？

**背景：** 设计文档的 `role(user/char)` 和代码里存在两个 role 概念的混淆。

**代码里实际的字段含义：**

| 字段 | 说明 | 迁移决策 |
|------|------|---------|
| `sender: 'user'\|'character'` | 角色扮演层发送者（谁说的话）。这就是文档里的 `role(user/char)`，只是字段名不同 | ✅ 保留，叫 sender |
| `role?: 'user'\|'assistant'\|'system'` | AI API层的 role，用于组装发给 OpenAI 的请求。和 sender 是两回事 | ❌ 丢弃，context.ts 根据 sender 重新推断，这个字段冗余 |
| `charId` | 属于哪个角色的对话（多角色支持的关键字段）| ✅ 保留 |
| `personaId` | 发消息时激活的用户马甲ID | ✅ 保留 |
| `mode: 'online'\|'offline'` | 是真实AI回复还是离线模拟 | ✅ 保留 |
| `timestamp` | 沉浸式时间（故事内时间，可与现实不同）| ✅ 保留，对应文档 user_time |
| `userTimestamp` | 与 timestamp 同义，字段名不统一的历史遗留 | ❌ 丢弃，合并到 timestamp |
| `charTimestamp` | 角色时间线时间 | ✅ 保留，对应文档 char_time |
| `createdAt` | 消息实际存储的真实时间 | ✅ 保留，对应文档 created_at |

**设计文档的 `status（生成中/已完成）`：** 暂不加。现在消息是流式完成后整条存入，等做断点恢复时再加。

### Q2：世界书条目表，哪些字段暂时不做？

**保留（现有代码已有）：** id, book_id, name, content, enabled, activation_mode, insertion_position, keywords, weight, priority, no_recurse, no_further_recurse, condition, event_config, created_at, updated_at

**暂时不做（跳过）：**

| 字段组 | 原因 |
|-------|------|
| inclusion_group / group_weight / prioritize_by_order | 互斥分组逻辑复杂，用户暂时不需要 |
| character_filter / filter_mode / trigger_types / additional_sources | 过滤器，用户看不懂，跳过 |
| sticky / cooldown / delay | 定时效果，跳过 |
| automation_id | 脚本联动，跳过 |
| filter_keywords / filter_logic | 二级关键词，跳过 |
| use_group_scoring | CC克也没搞明白，跳过 |
| probability | 触发概率，现有激活逻辑没有，跳过 |

### Q3：迁移策略

**采用启动时自动迁移：**
1. 服务器启动时检测 JSON 文件是否存在
2. 若存在：读取所有数据 → 写入 SQLite → 把 JSON 文件改名为 `.json.bak`
3. 此后所有请求走 SQLite

**技术选型：**
- 驱动：better-sqlite3（同步接口，Windows 有预编译二进制，无需手动 build）
- 不用 Drizzle ORM：通用 blob 存储模式下 Drizzle 的类型优势无法发挥，直接用 better-sqlite3
- 每个实体一张表：`id TEXT PRIMARY KEY, char_id TEXT, data TEXT NOT NULL, created_at TEXT`
- 新功能表（character_values 等）单独建表，等有 UI 时再加路由

**不用 Drizzle 的理由：** 现有路由都通过 FileStore 的通用 IFileStore<T> 接口操作数据，不直接写 SQL/ORM 查询语句。创建通用的 SqliteStore<T> 实现同样接口，所有路由无需改动。
